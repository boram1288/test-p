# Quality Attribute Specification

> 본 문서는 `02_requirements.md`에서 도출한 품질 속성 요구사항을 Quality Attribute Specification으로 정리한다.
>
> 진행 순서: 요구사항 수집 → 요구사항 도출 → **품질 속성 명세(본 문서)** → Utility Tree 정리 → Architectural Driver 선정

---

## 1. 품질 속성 명세 요약

| QS | 품질 속성 | 목표 응답 측정치 | 검증 단계 |
|----|----------|----------------|----------|
| QS-01 | 보안 (Host 침해 기밀성) | 격리 메모리 노출 0건 | 보안 침투 시험 / 공격 벡터 자동화 |
| QS-02 | 보안 (HW 접근 격리) | S2MPU 권한 중첩 0건 | 권한 전환 로그 / fault 주입 |
| QS-03 | 성능 (실시간 처리) | 비격리 대비 성능 저하 10% 이내 | 파이프라인 성능 계측 |
| QS-04 | 확장성 (Workload 수용) | Framework 코어 수정 0 LoC | Workload 추가 전후 diff |
| QS-05 | 성능 (통신 오버헤드) | 도메인 간 dma-buf 버퍼 전달 시 memcpy 호출 0회 | 복사 경로 검사 / zero-copy 확인 |
| QS-06 | 가용성 (pVM 장애 격리) | Host/타 pVM 다운타임 0, Workload 서비스 재개 3초 이내 | 장애 주입 시험 |
| QS-07 | 자원 효율 | 추가 메모리 256MB 이하, 전력 증가 5% 이내 | 메모리/전력 프로파일링 |
| QS-08 | 시험 용이성 | 격리 요구사항별 공격 벡터 자동화 시험 커버 100% | 공격 벡터 매핑 / CI 회귀 시험 |
| QS-09 | 변경 용이성 (Secure OS 교체) | Secure OS 패키지 외부 변경 파일 0개 | 교체 전후 diff |

## 2. 상세 명세

### QS-01: 보안 — Host 침해 시 기밀성

| 요소 | 명세 |
|------|------|
| 자극원 | Host OS 루트 권한을 탈취한 공격자 |
| 자극 | Host 커널 권한으로 pVM 메모리 읽기 시도 |
| 환경 | Secure Vision AI 파이프라인 정상 동작 중 |
| 대상 | pVM 내 영상 원본, AI 모델 가중치, 추론 중간 데이터 |
| 응답 | Stage-2 격리에 의해 모든 접근이 차단되고, 비정상 접근 시도가 기록된다 |
| 응답 측정치 | 침투 시험 전 케이스에서 격리 메모리 노출 0건 |
| 검증 방법 | `/dev/mem`, KVM 인터페이스 오용, 페이지 테이블 조작 등 공격 벡터를 자동 실행하고 pVM 소유 메모리 노출 여부를 확인 |
| 수용 기준 | 읽힌 바이트 수가 0이어야 하며, 비정상 접근 로그가 남아야 한다 |

### QS-02: 보안 — HW 접근 격리

| 요소 | 명세 |
|------|------|
| 자극원 | Host 일반 기능과 보안 파이프라인 |
| 자극 | Camera/AI HW 사용 주체 전환 또는 동시 사용 요구 |
| 환경 | SW 중재 기반 HW IP 공유 운용 중 |
| 대상 | HW IP 중재 계층, S2MPU 권한 상태 |
| 응답 | 이전 주체 권한 회수 후 다음 주체 권한을 부여하며, 권한 중첩 구간이 발생하지 않는다 |
| 응답 측정치 | S2MPU 권한 상태 로그에서 권한 중첩 구간 없음 (위반 횟수 = 0) |
| 검증 방법 | 권한 전환 코드 경로에서 회수→부여 순서를 검사하고, 비활성 주체의 HW 레지스터 접근을 주입 |
| 수용 기준 | 권한 중첩 로그가 없어야 하며, 비활성 주체 접근은 S2MPU fault로 차단되어야 한다 |

### QS-03: 성능 — 실시간 처리

| 요소 | 명세 |
|------|------|
| 자극원 | 카메라 센서 |
| 자극 | 1080p 30fps 영상 스트림 지속 유입 |
| 환경 | 격리 파이프라인(Secure Camera→Secure AI) 정상 동작, Host 통상 부하 |
| 대상 | End-to-End 파이프라인(캡처→Camera HW→AI HW 추론→판단) |
| 응답 | Camera/AI HW 가속으로 프레임 드롭 없이 처리 |
| 응답 측정치 | 비격리 구성 대비 처리 성능 저하 10% 이내 |
| 검증 방법 | 1080p 30fps 입력 조건에서 동일 HW의 비격리 baseline과 격리 구성의 fps 및 E2E 지연을 비교 |
| 수용 기준 | `(비격리 fps - 격리 fps) / 비격리 fps <= 10%`를 만족해야 한다 |

### QS-04: 확장성 — 신규 Workload 수용

| 요소 | 명세 |
|------|------|
| 자극원 | 로봇 제조사 / Workload 개발자 |
| 자극 | 신규 보안 Workload 추가 요구 |
| 환경 | Framework가 제품에 배포/운용 중인 상태 |
| 대상 | Framework 본체(Middleware/커널 드라이버) |
| 응답 | 펌웨어 재배포/Framework 수정 없이 Workload 패키징/탑재만으로 수용된다 |
| 응답 측정치 | Framework 코어 수정 0 LoC |
| 검증 방법 | Workload 추가 전후 diff에서 Framework 코어 디렉터리 변경 라인 수를 카운트 |
| 수용 기준 | 코어 변경 라인이 0이어야 하며, 서명 검증된 패키지만 탑재되어야 한다 |

### QS-05: 성능 — 도메인 간 통신 오버헤드

| 요소 | 명세 |
|------|------|
| 자극원 | Secure Camera pVM |
| 자극 | 프레임 단위 대용량 영상 데이터를 Secure AI pVM으로 전달 |
| 환경 | 파이프라인 정상 동작 중 |
| 대상 | 도메인 간 보안 채널(공유 메모리/RPC) |
| 응답 | 데이터 노출 없이 전달되며 파이프라인 실시간성이 유지된다 |
| 응답 측정치 | 도메인 간 dma-buf 버퍼 전달 시 memcpy 호출 횟수 0회 |
| 검증 방법 | dma-buf 등 버퍼 전달 경로에서 복사 함수 호출 여부를 코드 경로와 런타임 trace로 확인 |
| 수용 기준 | 프레임 전달 경로에서 memcpy 호출이 0회여야 하며, zero-copy 경로가 유지되어야 한다 |

### QS-06: 가용성 — pVM 장애 격리

| 요소 | 명세 |
|------|------|
| 자극원 | 오동작하는 보안 Workload |
| 자극 | pVM 비정상 종료(크래시, 무응답) |
| 환경 | 다중 pVM 운용 중인 로봇 통상 동작 상태 |
| 대상 | Host OS, 타 pVM, 로봇 기본 동작 |
| 응답 | 장애가 해당 pVM에 한정되고, Framework가 자원을 안전하게 회수한 뒤 재시작한다 |
| 응답 측정치 | Host/타 pVM 다운타임 0, Workload 서비스 재개 3초 이내 |
| 검증 방법 | 장애 주입 시각부터 Workload 서비스 재개 시각까지 계측하고 Host/타 pVM 상태를 확인 |
| 수용 기준 | 장애가 다른 도메인으로 전파되지 않고, `(Workload 서비스 재개 시각) - (장애 주입 시각) <= 3초`를 만족해야 한다 |

### QS-07: 자원 효율

| 요소 | 명세 |
|------|------|
| 자극원 | 시스템 통합자(로봇 제조사) |
| 자극 | 보안 Framework + pVM 2개(Camera, AI) 상시 운용 |
| 환경 | 로봇 제품의 통상 동작 상태 |
| 대상 | SoC 메모리/전력 예산 |
| 응답 | 비격리 구성 대비 추가 자원 소모가 제품 탑재 가능 한도 이내로 유지된다 |
| 응답 측정치 | 추가 메모리 256MB 이하, 전력 증가 5% 이내 |
| 검증 방법 | `/proc/meminfo`, cgroup 집계, PMIC 텔레메트리 또는 전력 계측 장비로 비격리 구성과 비교 |
| 수용 기준 | 메모리와 전력 증가분이 각각 목표 이내여야 한다 |

### QS-08: 시험 용이성 — 격리 보장 검증

| 요소 | 명세 |
|------|------|
| 자극원 | 품질/검증 조직 |
| 자극 | 격리 요구사항별 공격 벡터에 대한 객관적 검증 요구 |
| 환경 | 통합 시험 단계 및 회귀 시험 |
| 대상 | 격리 메커니즘(Stage-2, S2MPU, 보안 채널) |
| 응답 | Host 침해 모사 도구 등 재현 가능한 자동화 시험으로 격리 유지가 검증된다 |
| 응답 측정치 | 격리 요구사항별 공격 벡터 자동화 시험 커버 100% |
| 검증 방법 | 공격 벡터 카탈로그와 자동화 테스트 매핑표에서 미커버 항목 수를 카운트하고 CI 반복 실행 여부를 확인 |
| 수용 기준 | `(전체 공격 벡터 항목 - 미커버 항목) / 전체 항목 = 100%`를 만족하고, 회귀 시험이 반복 실행 가능해야 한다 |

### QS-09: 변경 용이성 — Secure OS 교체

| 요소 | 명세 |
|------|------|
| 자극원 | 서드파티 Secure OS 벤더 / 로봇 제조사 |
| 자극 | 탑재된 Secure OS를 다른 Secure OS로 교체 |
| 환경 | 설계/통합 단계 또는 제품 유지보수 단계 |
| 대상 | Secure OS와 무관한 SW(Framework, Host Middleware, 타 Workload) |
| 응답 | Secure OS 패키지 교체만으로 완료되며 무관 SW는 수정되지 않는다 |
| 응답 측정치 | Secure OS 패키지 외부 변경 파일 0개 |
| 검증 방법 | Secure OS 교체 전후 diff에서 Secure OS 패키지 외부 파일 변경 수를 카운트 |
| 수용 기준 | 패키지 외부 변경 파일이 0개여야 하며, 기존 ENC/DEC 기능이 회귀하지 않아야 한다 |

## 3. 품질 속성 간 상충 관계

| 상충 관계 | 관련 시나리오 | 설계 시 확인할 사항 |
|----------|-------------|-------------------|
| 보안 vs 성능 | QS-01, QS-02, QS-03, QS-05 | Stage-2/S2MPU/보안 채널 강화가 실시간 경로에 주는 비용을 계측한다. |
| 보안 vs 확장성 | QS-01, QS-02, QS-04 | Workload 동적 탑재가 신뢰 경계와 공격 표면을 넓히지 않도록 서명/권한 정책을 강제한다. |
| 성능 vs 자원 효율 | QS-03, QS-05, QS-07 | 실시간 처리를 위한 버퍼, pVM, HW 예약량이 메모리/전력 예산을 넘지 않는지 확인한다. |
| 변경 용이성 vs 안정성 | QS-06, QS-09 | Secure OS 교체와 pVM 재시작 경로가 기존 TrustZone 기능과 생명주기 관리에 회귀를 만들지 않는지 확인한다. |
