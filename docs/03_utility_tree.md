# 품질 속성 선정 (Utility Tree)

> 본 문서는 `02_requirements.md`의 QA-01~QA-09을 ATAM Utility Tree로 구조화하고, 각 리프 시나리오를 (중요도, 난이도)로 평가하여 **핵심 품질 속성**을 선정한다.
> 진행 순서: 요구사항 수집 → 요구사항 도출 → **품질 속성 선정(본 문서)** → Architectural Driver 선정
>
> 관련 유즈케이스 명세는 [`01_use_case_spec.md`](01_use_case_spec.md) 참조.

---

## 1. 평가 기준

| 축 | 기준 |
|----|------|
| 중요도 | 미충족 시 비즈니스 영향 (사업 진입/제품 출시/사고 피해). H/M/L |
| 난이도 | 아키텍처 구조에 미치는 영향과 달성의 기술적 어려움. H/M/L |

> 수치 목표는 가정치이며, 로봇 제조사 협의 및 PoC 결과에 따라 보정한다.

## 2. Utility Tree

```
Utility
├─ 보안 (Security)
│  └─ 기밀성 (Host 침해)
│      └─ (H,H) Host 루트 권한 공격자가 pVM 메모리 읽기를 시도해도 전 케이스 차단, 격리 메모리 노출 0건 — QA-01
├─ 성능 (Performance)
│  ├─ 실시간 처리
│  │   └─ (H,H) 1080p 30fps 유입 시 비격리 대비 저하 10% 이내 — QA-02
│  ├─ 통신 오버헤드
│  │   └─ (H,M) 도메인 간 프레임 전달 시 zero-copy — QA-03
│  └─ HW IP 공유 오버헤드
│      └─ (H,H) Host/pVM 사용 시 기존 프레임율 유지(프레임율 Drop 0%) — QA-09
├─ 자원 효율 (Resource Efficiency)
│  └─ 메모리/전력 footprint
│      └─ (M,M) Framework+pVM 2개 상시 운용 시 추가 메모리 256MB 이하, 전력 증가 5% 이내 — QA-04
├─ 확장성 (Extensibility)
│  └─ 신규 Workload 수용
│      └─ (H,H) 신규 보안 Workload 추가 시 코어 수정 0 LoC, 패키징/탑재만으로 수용 — QA-05
├─ 변경 용이성 (Modifiability)
│  └─ Secure OS 교체
│      └─ (M,M) Secure OS 교체 시 Secure OS 외 모듈 수정 파일 0개 — QA-06
├─ 가용성 (Availability)
│  └─ pVM 장애 격리
│      └─ (M,M) pVM 비정상 종료 시 장애 pVM 재시작 3초 이내 — QA-07
└─ 시험 용이성 (Testability)
   └─ 격리 보장 검증
       └─ (M,M) 격리 검증 요구 시 QA-01~03 자동화 시험 커버 100%, 회귀 반복 실행 가능 — QA-08
```

### 2.1 응답 측정치 상세와 수치 근거

각 리프 시나리오의 응답 측정치에 대해 측정 방법, 수치 설정 근거, 근거의 성격을 정리한다. 근거 유형은 세 가지로 구분한다.

- **불변 조건**: 임계값이 아니라 해당 품질 주장이 성립하기 위한 정의상의 조건 (0건/0 LoC/100% 등). 다른 값을 목표로 삼는 것 자체가 무의미하므로 객관적이다.
- **표준/문헌**: 산업 표준, 공개 문헌, 공개된 유사 시스템 실측치에서 도출.
- **가정치**: 현재 객관적 근거가 없어 실측/협의로 보정해야 하는 값.

| QA | 응답 측정치 | 측정 방법 | 수치 설정 근거 | 근거 유형 / 확보 방법 |
|----|------------|----------|---------------|---------------------|
| QA-01 | 격리 메모리 노출 0건 (전 케이스 차단) | Host 루트 권한 침해 모사 도구로 공격 벡터 카탈로그(물리 메모리 직접 매핑, /dev/mem 접근, KVM ioctl 오용, 페이지 테이블 조작 등) 전 항목을 자동 실행하고, pVM 소유 메모리에서 실제로 읽힌 바이트 수를 계측 | 보안 속성은 부분 달성이 무의미 — 단 1건의 노출로 기밀성 침해가 성립한다. CC 침투 시험(AVA_VAN)과 GP TEE Protection Profile도 "식별된 공격 전부 차단"을 합격 기준으로 사용 | **불변 조건**. 단, 공격 카탈로그(분모)의 완전성이 관건 — GP TEE PP 공격 목록과 Android VRP의 pKVM Boundary Defeat 사례로 카탈로그를 구성 (`99_security_qa_metrics.md` 참조) |
| QA-02 | 비격리 대비 저하 10% 이내 | 동일 HW에서 비격리 구성(Host 직접 실행) 파이프라인을 baseline으로 실측하고, 격리 구성의 fps/지연과 비교 | KVM 계열 가상화의 I/O 경로 오버헤드가 문헌상 통상 5~15%로 보고되는 범위에서, 달성 가능하면서 제품 기능이 유지되는 상한으로 설정 | **가정치**. 비격리 baseline을 먼저 실측한 뒤 PoC 결과와 제조사 협의로 보정 |
| QA-03 | zero-copy | dma-buf 등 버퍼 전달 시 복사 횟수를 코드 경로로 확인(0회=zero-copy) | 복사가 1회라도 발생하면 zero-copy 채널의 설계 목표가 달성되지 않음 — 임계값이 아니라 설계 목표의 정의상 조건 | **불변 조건** |
| QA-04 | 추가 메모리 256MB 이하 | Framework+pVM 2개 상시 운용 상태와 비격리 구성의 시스템 메모리 사용량 차이를 계측(/proc/meminfo, cgroup 단위 집계) | AVF Microdroid pVM이 통상 128~256MB급으로 운용되는 공개 구성 사례를 기준으로 pVM 2개+Framework 상주분을 추정한 값 | **유사 시스템 참조 + 가정치**. 타겟 SoC의 총 메모리와 제품 메모리 예산표(BOM)를 제조사에서 입수하여 허용 비율로 확정 |
| QA-04 | 전력 증가 5% 이내 | 전력 계측 장비(또는 PMIC 텔레메트리)로 통상 운용 시나리오의 평균 소비 전력을 비격리 구성과 비교 | 현재 객관적 근거 없음 | **가정치**. 로봇 제품의 배터리 가동시간 요구(예: 1회 충전 N시간)에서 허용 전력 증가분을 역산하고, PoC 전력 프로파일링으로 보정 |
| QA-05 | Framework 코어 수정 0 LoC | 신규 Workload 추가 전후의 git diff에서 Framework 코어 모듈(Middleware/커널 드라이버) 디렉터리의 변경 라인 수를 카운트 | 플러그인 아키텍처의 정의(개방-폐쇄 원칙: 확장에 열려 있고 수정에 닫힘) 자체가 코어 무수정을 요구. 0이 아닌 임계값은 "수정 없는 확장" 주장을 무효화 | **불변 조건**. 단, "코어"의 경계(어느 디렉터리까지인지)를 설계 단계에서 명시적으로 정의해야 측정이 객관화됨 |
| QA-06 | Secure OS 외 모듈 수정 파일 0개 | Secure OS 교체 작업의 diff에서 Secure OS 패키지 외부의 변경 파일 수를 카운트 | 인터페이스 추상화 목표의 정의상 조건. 표준 GP TEE API(Client/Internal Core API) 계약 준수가 전제 | **불변 조건**. 교체 대상 Secure OS가 GP 표준을 따르는지가 전제 조건 — 후보 Secure OS 목록의 API 준수 여부를 사전 조사 |
| QA-07 | 장애 pVM 재시작 3초 이내 | 장애 검출 시각부터 pVM 재시작 완료(Workload 서비스 재개) 시각까지 계측 | AVF pVM(Microdroid) 부팅이 초 단위로 보고되는 점에 기반한 추정치 | **가정치**. PoC에서 pVM 부팅 시간을 실측하고, 로봇 서비스의 허용 중단 시간(보안 기능 공백 허용치)을 제조사와 협의하여 확정 |
| QA-08 | 자동화 시험 커버 100%, 회귀 반복 실행 가능 | 격리 요구사항별 공격 벡터 카탈로그와 자동화 테스트의 매핑표를 작성하고 미커버 항목 수를 카운트(0개=100%). CI에서 반복 실행 가능 여부 확인 | 시험되지 않은 격리 주장 항목이 하나라도 있으면 "객관적으로 검증되었다"는 주장이 성립하지 않음 | **불변 조건**. 커버리지의 분모가 되는 공격 카탈로그의 완전성은 GP TEE PP 공격 목록을 기준으로 확보 (`99_security_qa_metrics.md` 참조) |
| QA-09 | 기존 프레임율 유지 (Drop 0%) | Host 일반 촬영과 보안 파이프라인을 동시 실행하며 양측의 프레임 drop 수를 각각 카운트 | 프레임 drop은 Host 측에서는 사용자 체감 기능 실패(촬영 끊김), 보안 파이프라인 측에서는 실시간성 위반으로, 임계값이 아니라 기능 성립 조건 | **불변 조건**. 단, 순간 부하에서의 허용 여부(예: 전환 시점 1프레임 지연 허용)는 제조사와 협의 여지 있음 |

## 4. 핵심 품질 속성 선정

- **1순위 (H,H)**: 아키텍처 구조를 직접 결정하며 실패 시 과제 목표가 무산되는 항목
- **2순위 (H,M)**: 1순위 해결 구조 위에서 함께 달성되도록 설계
- **3순위 (M,M)**: 구조 결정 후 설계/검증 단계에서 충족 확인

| 순위 | 품질 속성 그룹 | 포함 QA | 비고 |
|:----:|---------------|---------|------|
| 1 | **보안 (Security)** | QA-01 | 핵심 driver |
| 1 | **성능 (Performance)** | QA-02, QA-09 | 핵심 driver |
| 1 | **확장성 (Extensibility)** | QA-05 | 과제 핵심 차별화 지점 |
| 2 | **성능 (Performance)** | QA-03 | 과제 핵심 차별화 지점 |
| 3 | 자원 효율, 변경 용이성, 가용성, 시험 용이성 | QA-04, QA-06, QA-07, QA-08 | 구조 결정 후 설계/검증 단계에서 충족 확인. 자원 효율은 가격 경쟁력과 연계된 독립 교차 관심사 |

## 다음 단계

선정된 핵심 품질 속성(보안, 성능, 확장성)을 FR/CONST와 결합하여, **Architectural Driver 선정** 단계에서 아키텍처 설계를 주도할 동인을 확정한다.
