# Design Point 목록

> 본 문서는 `04_architectural_drivers.md` 5.2절의 설계 착수 순서(T-1 ~ T-5 연계 6단계), `99_additional_design_points.md`의 교차(cross-cutting) Design Point, 추가로 식별한 보안 정책 DP를 통합하여, 아키텍처 설계 착수 전 확정해야 할 **Design Point**를 정리한다.

> 진행 순서: 요구사항 수집 → 요구사항 도출 → QAW → Architectural Driver 선정 → **Design Point 목록(본 문서)**

## 1. 개요

Design Point(DP)는 Architectural Driver로부터 도출되는 구체적인 설계 결정 단위다. DP를 해결하기 전까지는 후보 구조를 평가할 기준이 성립하지 않는다.

격리 증빙·감사 체계, pVM 아이덴티티·채널 상대 인증, 침해 시도 탐지·변조 불가 로깅, 보안 워크로드·Secure OS 탑재 방식 결정은 범위에서 제거했다. 남은 DP는 선행 의존성 기준으로 `DP-01`부터 `DP-08`까지 재정렬했다.

각 DP의 제목은 해당 DP가 해결을 통해 확보하려는 **핵심 품질 속성**이 수식하는 형태로 표기한다. 따라서 각 DP의 문제 상황과 설계질문은 그 품질 속성을 중심으로 정리한다.

| 유형 | 설명 | 건수 |
|------|------|:----:|
| **T-연계 DP** | 핵심 기술 T-1 ~ T-5(또는 공통 전제)와 직결된 설계 결정 | 5건 |
| **교차 DP** | 단일 T에 속하지 않고 Driver에서 직접 요구되는 설계 결정 | 3건 |

## 3. Design Point 상세

---

#### DP-01: 시스템의 보안 정책 신뢰를 위한 TCB 바운더리 결정

**산출근거 Driver**
- QA-01: Host 침해 시 pVM 내 보호 자산의 기밀성 보장
- QA-02: pVM 간 독립 격리 보장
- CONST-02: 기 포팅된 pKVM 커널을 전제로 하며 EL2 수정 불가

**선행 DP**
- 없음 — 본 DP는 전체 DP의 출발점이다. 여기서 확정되는 TCB 범위·공격자 모델·보호 속성이 후속 DP 전체의 후보 구조 평가 기준이 된다.

**문제 상황**
- 시스템이 주장하는 모든 보안 정책의 신뢰는 무엇을 신뢰 기반(TCB)에 두느냐에 달려 있다. pKVM이 Host 비신뢰라는 큰 방향은 제공하지만, 본 과제의 Host 측 프레임워크·커널 드라이버를 TCB 안에 둘지 밖에 둘지에 따라 보안 정책의 신뢰 수준이 달라진다.
- TCB 바운더리가 확정되지 않으면 HW IP 중재자, 보안 채널 중개자, 부팅 검증 주체를 어느 신뢰 영역에 배치할지 판단할 수 없다.
- 병렬 설계 중 팀별 신뢰 가정이 달라지면 전체 보안 정책의 신뢰는 가장 약한(TCB가 가장 넓은) 가정으로 무너진다.

**해결해야 할 설계질문**
- Host 측 프레임워크(미들웨어·커널 드라이버)는 TCB에 포함하는가, 아니면 비신뢰 영역으로 보는가? 그 결정이 보안 정책의 신뢰 수준에 어떤 영향을 주는가?
- 공격자는 Host 커널 권한, pVM 내부 권한, 물리 접근 권한 중 어디까지 가진다고 가정하는가?
- 신뢰로 보장할 보호 속성은 기밀성·무결성까지인가, 가용성(DoS)도 포함하는가?
- EL2(pKVM), TrustZone EL3·S-EL1, 별도 서비스 pVM 중 어떤 컴포넌트를 신뢰 주체로 둘 것인가?
- TCB 목록·공격자 모델·보호 속성별 의존 메커니즘을 신뢰 모델 문서에 어떤 수준으로 명시하여, 후속 DP가 동일한 신뢰 기준을 공유하게 할 것인가?

> 상세 논거: `99_trust_boundary_qna.md` 참조

#### DP-02: 접근 통제 일관성을 위한 보안 정책 및 권한 집행 모델 설계

**산출근거 Driver**
- QA-01, QA-02: Host 및 pVM 간 비신뢰 전제에서 보호 자산과 도메인 격리 보장
- FR-03: HW IP 사용 권한과 DMA 경로 할당 필요
- FR-04: pVM 간 보안 채널 수립 권한 필요
- FR-06: 보안 워크로드 동적 탑재 권한 필요
- CONST-02: EL2 수정 불가 제약 아래 권한 집행 위치 결정 필요

**선행 DP**
- DP-01 (TCB 바운더리 결정): Host 프레임워크가 TCB 밖이라면 권한 집행을 Host에 단독으로 둘 수 없으므로, TCB 범위와 신뢰 주체가 먼저 확정되어야 권한 집행 위치를 결정할 수 있다.

**문제 상황**
- pVM 생성, HW IP 사용, 채널 수립, 워크로드 로딩은 모두 권한 판단이 필요한 동작인데, 동작마다 권한 판단 주체·규칙이 제각각이면 접근 통제의 일관성이 깨진다.
- Host가 비신뢰이면 Host 프레임워크가 단독으로 권한을 허가하는 구조는 신뢰 모델(QA-01, QA-02)과 충돌하므로, 일관된 권한 집행 위치가 정의되어야 한다.
- 통일된 정책 모델이 없으면 DP-04와 DP-05가 서로 다른 권한 가정으로 설계되어 시스템 전체의 접근 통제 일관성이 무너진다.

**해결해야 할 설계질문**
- pVM 생성, HW IP 사용, 채널 수립, 워크로드 로딩 권한을 누가 정의·검증·집행하며, 동작 간 일관된 판단 기준을 어떻게 유지하는가?
- 권한 정책은 워크로드 패키지 manifest, 도메인별 권한 테이블, 런타임 정책 중 무엇으로 표현하여 단일 정책원(single source)으로 일관성을 확보하는가?
- EL2, TrustZone TEE, 별도 서비스 pVM, Host 프레임워크 중 어떤 컴포넌트가 어떤 권한을 집행하며, 집행이 분산될 때 일관성을 어떻게 보장하는가?
- 권한 불일치, 정책 위반, 권한 철회 시 pVM·채널·HW IP 할당을 어떻게 일관되게 차단·회수하는가?
- DP-08의 부팅 측정값을 권한 정책과 어떻게 연결하여 측정값과 권한 판단의 일관성을 유지할 것인가?

#### DP-03: GP API Backward Compatibility를 위한 TrustZone 연동 구조 설계

**산출근거 Driver**
- FR-08: pVM 워크로드가 기존 TrustZone Secure OS 기반 TEE 기능과 연동해야 함
- CONST-03: 기존 TrustZone Secure OS의 SMC 경로와 TEE 기능은 무회귀로 유지해야 함

**선행 DP**
- DP-01 (TCB 바운더리 결정): TrustZone EL3·S-EL1을 신뢰 주체로 둘 것인지가 신뢰 모델에서 결정되어야, pVM→TEE 경로의 배치와 TEE를 신뢰 앵커로 사용하는 범위를 정할 수 있다.

**문제 상황**
- 기존 키 관리·인증 등 TEE 기능은 GlobalPlatform(GP) TEE Client API와 SMC 경로로 호출되며, 기존 Client App과 Secure OS는 이 API 계약에 의존한다. 신규 pVM 기반 구조가 이 계약을 바꾸면 기존 제품 기능이 회귀한다.
- pVM→TEE 경로를 새로 추가하더라도, 기존 Host→TEE의 GP API 호출 의미와 SMC 경로는 변경 없이(backward compatible) 유지되어야 한다.
- DP-07(Secure Storage)과 DP-08(pVM 부팅)이 TrustZone TEE를 신뢰 앵커로 사용하더라도, GP API 호환을 깨지 않는 범위에서 연동해야 한다.

**해결해야 할 설계질문**
- 기존 GP TEE Client API 계약(함수 시그니처·세션·공유 메모리 의미)을 변경 없이 유지하면서 pVM→TEE 경로를 어떻게 추가할 것인가?
- pVM이 기존 TEE를 호출할 때 SMC 포워딩·라우팅 경로를 어디에 두어, 기존 Host→TEE SMC 경로에 영향을 주지 않을 것인가?
- pVM이 GP API로 TEE를 호출할 때, 호출 주체(pVM)의 신뢰성을 TEE가 기존 API 의미를 깨지 않고 어떻게 식별·검증할 것인가?
- 기존 Client App·Secure OS의 무수정 동작(backward compatibility)을 어떤 회귀 테스트 기준으로 검증할 것인가?
- GP API 호환을 유지하면서 DP-07의 키 관리와 DP-08의 이미지 검증에 TrustZone TEE를 어디까지 사용할 것인가?
- TrustZone 연동 실패가 기존 GP API 경로와 신규 pVM 경로에 어떤 영향을 주도록 격리할 것인가?

#### DP-04: 확장성 및 가용성을 위한 pVM 관리 골격 설계

**산출근거 Driver**
- FR-01: pVM 생성·시작·정지·종료와 자원 할당·회수 관리 필요
- FR-02: Secure Camera, Secure AI 등 다중 pVM 동시 운용 필요
- CONST-01: Android 스택 의존 없이 Linux 네이티브로 동작해야 함

**선행 DP**
- DP-02 (보안 정책 및 권한 집행 모델 설계): pVM 생성·자원 할당 권한을 누가 정의·검증·집행하는지가 정해져야 pVM 제어 인터페이스에 권한 검증 지점을 배치할 수 있다.

**문제 상황**
- Secure Vision AI는 최소 두 개 이상의 pVM을 동시 운용하며, 이후 보안 도메인이 추가될 때 골격이 이를 수용하지 못하면(확장성 부족) 매번 관리 구조를 재설계해야 한다.
- 생명주기 관리 구조가 없으면 한 pVM의 장애나 자원 고갈이 다른 pVM·Host로 전파되어 시스템 가용성을 해칠 수 있다.
- Linux 네이티브 제약 때문에 Android AVF의 VirtualizationService 같은 구조를 그대로 사용할 수 없어, 확장성과 가용성을 만족하는 골격을 직접 설계해야 한다.
- DP-05의 HW IP 중재 PoC는 기술 리스크가 크므로, DP-04 전체 완료를 기다리기보다 최소 계약이 정해지는 즉시 병행 착수할 필요가 있다.

**해결해야 할 설계질문**
- 새로운 pVM·보안 도메인을 골격 수정 없이 추가할 수 있도록(확장성), 생명주기 관리자·pVM 제어 인터페이스·자원 할당자의 책임 경계를 어떻게 나눌 것인가?
- 도메인별 메모리·CPU·디바이스 자원을 어떤 단위로 분리·회수하여, 자원 고갈·누수 없이 다수 pVM을 운용할 것인가?
- 한 pVM의 비정상 종료·장애가 다른 pVM과 Host로 전파되지 않도록(가용성), 어떤 격리·자원 회수·재시작 정책을 둘 것인가?
- KVM ioctl, VFIO, dma-buf 등 어떤 Linux 네이티브 인터페이스를 기본 제어면으로 사용할 것인가?
- HV 파트와 확정해야 할 최소 pKVM hypercall 계약은 무엇이며, 어떤 항목이 정해지면 DP-05 PoC를 병행 착수할 수 있는가?

#### DP-05: 성능(실시간 처리) 및 자원 효율을 위한 HW IP 중재 구조 설계

**산출근거 Driver**
- FR-03: ISP·NPU 등 HW IP를 Host와 pVM이 공유·할당할 수 있어야 함
- QA-03: HW IP DMA 경로 격리와 잔류 데이터 소거 필요
- QA-11: HW IP 공유·전환 오버헤드가 실시간성을 해치지 않아야 함
- CONST-05: 단일 컨텍스트 HW IP이며 HW 변경 불가

**선행 DP**
- DP-02 (보안 정책 및 권한 집행 모델 설계): HW IP 사용 권한과 DMA 경로 할당 권한의 집행 주체가 정해져야 중재자 배치 후보(Host 커널 드라이버 vs 별도 서비스 pVM)를 일관된 권한 가정 위에서 평가할 수 있다.
- DP-04 (pVM 관리 골격 설계, 병행): HW IP 중재 PoC는 pVM 생성·메모리 할당·hypercall 최소 계약만 확보되면 착수할 수 있으므로, DP-04 전체 완료를 기다리지 않고 최소 계약 확정 즉시 병행 착수한다.

**문제 상황**
- ISP·NPU는 다중 컨텍스트를 지원하지 않는 공유 자원이므로 Host와 pVM이 번갈아 사용해야 하며, SW 중재 과정의 전환 오버헤드가 영상 처리·AI 추론의 실시간성을 해칠 수 있다.
- 중재·전환이 비효율적이면 HW IP 유휴 시간이 늘어 자원 효율이 떨어진다.
- 격리(QA-03)를 위한 SMMU/IOMMU 재설정과 잔류 데이터 소거를 매 전환마다 수행해야 하는데, 이 보안 처리가 실시간 성능·자원 효율과 상충한다.
- HW IP 공유는 본 과제의 핵심 기술 리스크이므로, 조기 PoC로 성능과 효율을 검증해야 한다.

**해결해야 할 설계질문**
- HW IP 중재자를 Host 커널 드라이버와 별도 서비스 pVM 중 어디에 두어야 전환 지연을 최소화하면서 격리(QA-03)를 유지하는가?
- 사용 주체 전환 시 SMMU/IOMMU 재설정·잔류 데이터 소거 비용을 어떻게 줄여 실시간성을 보장할 것인가?
- HW IP 유휴를 줄이고 Host·pVM 간 사용 시간을 효율적으로 배분하는 중재·스케줄링 정책은 무엇인가?
- 중재·전환 오버헤드는 어떤 시나리오와 수치 기준(프레임 지연·처리량)으로 측정할 것인가?
- DP-04의 어떤 최소 기능이 확보되면 HW IP 중재 PoC를 병행 착수할 수 있는가?

#### DP-06: 성능(통신 오버헤드)을 위한 zero-copy 보안 채널 구조 설계

**산출근거 Driver**
- FR-04: pVM↔pVM, pVM↔Host 간 보안 채널 제공 필요
- QA-04: Secure Vision AI 파이프라인의 실시간 처리 필요
- QA-05: 도메인 간 통신 오버헤드가 실시간성을 해치지 않아야 함

**선행 DP**
- DP-05 (HW IP 중재 구조 설계): Secure Camera→Secure AI 영상 데이터는 ISP·NPU의 DMA 버퍼와 직결되므로, HW IP 중재 구조의 버퍼 소유권과 DMA 경로가 확정되어야 zero-copy 공유 메모리 구조를 결정할 수 있다.

**문제 상황**
- Secure Camera에서 Secure AI로 전달되는 영상 데이터는 대용량이므로, 복사 기반 전달은 통신 오버헤드로 실시간 파이프라인의 성능 병목이 된다.
- zero-copy 공유 메모리로 오버헤드를 줄이되, Host 비신뢰 전제에서 채널 내용을 Host가 읽거나 변조하지 못하게 격리해야 하며, 이 보안 처리가 다시 오버헤드를 늘릴 수 있다.
- 제어 경로와 데이터 경로를 섞으면 오버헤드 측정·최적화와 실패 처리가 복잡해진다.

**해결해야 할 설계질문**
- Secure Camera→Secure AI 영상 데이터를 복사 없이(zero-copy) 전달하는 공유 메모리 구조는 무엇이며, 통신 오버헤드를 얼마나 줄이는가?
- RPC 제어 경로와 공유 메모리 데이터 경로를 어떻게 분리하여 데이터 경로의 오버헤드를 최소화할 것인가?
- Host가 채널 버퍼를 열람·재매핑하지 못하게 하는 Stage-2/SMMU 정책을 적용하되, 그 비용이 통신 오버헤드를 과도하게 늘리지 않게 하려면 어떻게 할 것인가?
- 채널 수립·해제·재협상 절차와 실패 처리를 오버헤드를 키우지 않도록 어떻게 정의할 것인가?
- 통신 오버헤드는 어떤 지표(지연·대역폭·CPU 점유)로 측정·검증할 것인가?

#### DP-07: 보안(기밀성)을 위한 Secure Storage 구조 설계

**산출근거 Driver**
- QA-01: Host 침해 시 AI 모델 가중치·영상 원본·추론 중간 데이터가 노출되지 않아야 함
- FR-08: 기존 TrustZone TEE의 키 관리·인증 기능과 연동 가능

**선행 DP**
- DP-03 (TrustZone 연동 구조 설계): 암호화 키의 생성·보관 주체로 TrustZone TEE를 사용할 수 있는지가 먼저 결정되어야 키 관리 주체(TEE·EL2·별도 키 관리 서비스)의 선택지를 평가할 수 있다.
- DP-06 (zero-copy 보안 채널 구조 설계): 복호화 키를 pVM에 전달하는 경로는 보안 채널 구조에 의존하므로, 채널 구조가 확정되어야 평문 노출 없는 키 전달 경로를 설계할 수 있다.

**문제 상황**
- 실행 중 메모리 격리가 유지되어도 AI 모델 가중치나 워크로드 패키지가 Host 파일시스템에 평문으로 저장되면, Host 침해 시 저장 자산의 기밀성이 깨진다.
- 저장 자산의 기밀성을 보장하는 암호화 키의 관리 주체는 TrustZone TEE, pKVM EL2, 별도 키 관리 서비스 중 무엇을 신뢰하느냐에 따라 달라진다.
- 복호화 시점과 키 전달 경로가 잘못 설계되면 로딩·실행 직전에 평문이 Host에 노출되어 기밀성이 무너진다.

**해결해야 할 설계질문**
- AI 모델 가중치, 워크로드 패키지, 설정 파일 중 어떤 자산을 저장 시 암호화하여 기밀성을 보장할 것인가?
- 암호화 키는 TrustZone TEE, pKVM EL2, 별도 키 관리 서비스 중 누가 생성·보관·사용할 것인가?
- 암호화 범위는 패키지 단위, 파일 단위, 파일시스템 전체 중 무엇으로 할 것인가?
- 복호화는 어느 시점에 수행하고, 복호화 키는 평문 노출 없이 어떤 경로로 pVM에 전달할 것인가?
- 키 회전, 폐기, 워크로드 삭제 시 저장 자산을 어떻게 무효화하여 기밀성을 유지할 것인가?

#### DP-08: 무결성 보장을 위한 pVM 부팅 방안 설계

**산출근거 Driver**
- QA-01: Host 침해 시 보호 자산이 악성 pVM 또는 변조된 워크로드로 노출되지 않아야 함
- FR-06: 비신뢰 Host 환경에서 보안 워크로드를 동적으로 로딩해야 함

**선행 DP**
- DP-03 (TrustZone 연동 구조 설계): 이미지 서명 검증과 측정값 보관에 TrustZone TEE를 신뢰 앵커로 사용할 수 있는지에 따라 검증 주체의 선택지가 달라진다.
- DP-07 (Secure Storage 구조 설계): 부팅 이미지·워크로드 패키지의 저장 암호화 범위와 복호화 키 방출 조건이 부팅 측정값과 맞물리므로, Secure Storage 구조가 확정되어야 측정값에 따른 키 방출·검증 절차를 설계할 수 있다.

**문제 상황**
- Host가 비신뢰이면 Host가 로딩하는 pVM 이미지와 워크로드 패키지가 승인된(변조되지 않은) 코드인지 무결성을 별도로 검증해야 한다.
- 검증 주체를 Host에만 두면 Host 침해 시 우회되어 무결성을 보장할 수 없고, EL2 검증은 CONST-02(EL2 수정 불가) 제약을 받는다.
- DP-07의 저장 보호는 부팅된 이미지가 승인된 코드라는 무결성 측정·검증 결과에 따라 복호화 키를 방출해야 하므로, 부팅 무결성과 키 방출이 연동되어야 한다.

**해결해야 할 설계질문**
- pVM 이미지와 워크로드 패키지의 서명 검증은 EL2, Host 드라이버, 별도 서비스 pVM, TrustZone TEE 중 누가 수행하여 무결성을 보장할 것인가?
- measured boot 체인은 어디서 시작하고 어디서 종료하여 부팅 전 구간의 무결성을 보장할 것인가?
- EL2 수정 없이 이미지 진위와 측정값을 신뢰할 수 있게 만드는 방법은 무엇인가?
- 무결성 검증 실패 시 pVM 생성, 워크로드 로딩, 채널 수립을 어떻게 차단할 것인가?
- 부팅 측정값을 DP-07의 키 방출 조건 및 DP-02의 정책 manifest와 어떻게 연결할 것인가?
