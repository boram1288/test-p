# DP-C1 / DP-C2 후보 구조 제안 배경 — 문제점 정리

> 본 문서는 `08_candidate_architectures.md`의 후보 구조 중 DP-C1(C1-1, C1-3)과 DP-C2(C2-1, C2-3)를 제안하기에 앞서, 왜 이런 후보 구조가 필요한지를 품질속성 축별 문제점으로 정리한 것이다.
>
> 근거 문서: `08_candidate_architectures.md`(P-C1-2/3, P-C2-1/2/3, 2.5절), `06_qa_utility_tree_metrics.md`(SEC/PERF/EXT/AVL 게이트·KPI)
>
> 수치의 `OO` 표기는 PoC/실측으로 채워야 할 자리 표시자다.

---

## 1. DP-C1. 도메인 간 프레임 전달 채널 — C1-1, C1-3 제안 배경

C1-1(정적 공유 풀)과 C1-3(소유권 이전)은 둘 다 "Host 매핑 없는 구조적 기밀성"을 노리는 후보로, 기밀성 문제가 두 후보의 공통 동기이고 확장성 문제는 채널 배선 방식의 한계에서 나온다.

### 1.1 (기밀성) 전달 구간에서 프레임 원본이 비신뢰 영역에 노출되지 않아야 함.

- 표준 채널(virtio-vsock)은 전달 버퍼가 Host 커널에 매핑된 채 경유함 — Host 침해 시 프레임 원본이 그대로 노출됨 (VOS-09, QA-01 위반)
- 노출 창 = 비소유 주체의 매핑 폭 × 유지 시간. 상시 감시 파이프라인은 OO시간 연속 운용되므로, 매핑이 열려 있는 구조에서는 노출 창이 파이프라인 수명 전체로 커짐
- 보안 인증(SEC-01 게이트)은 root 권한 Host가 전체 메모리를 덤프해도 canary 노출 0건을 요구함 — 암호화로 가리는 방식은 기밀성이 "구조적 비노출"이 아닌 "키 관리 강도"에 의존하게 되어 격리 논증·인증 대응이 어려움
- 도메인 간에도 동시 매핑이 있으면 한쪽 pVM 침해 시 상대 도메인 프레임까지 노출됨 (권한 중첩 0 원칙, SEC-02 위반)

**→ 그래서**: 구성 시 Host 매핑을 제거하고 pVM끼리만 공유하는 **C1-1**, 나아가 어떤 시점에도 소유자가 정확히 1개 도메인인 **C1-3**이 필요함.

### 1.2 (확장성) 도메인 추가가 코어 수정 없이 구성 변경만으로 흡수되어야 함.

- 도메인 쌍마다 전용 채널을 배선하면 N개 도메인에서 O(N^2)개 채널 — 도메인 OO개 기준 채널 OO개, 채널당 버퍼 풀 OO MB로 자원이 곱으로 증가함
- 신규 도메인(센서, 다단 추론 등) 추가 시 채널 설정이 코어 수정을 유발하면 QA-03 "코어 수정 0 LoC" 게이트 위반 (R-4의 다단 파이프라인 로드맵과 충돌)
- upstream pKVM은 guest-guest 공유/이전 프리미티브가 없고 FF-A guest-to-guest relayer 표준화·개발이 느림 — 채널 메커니즘을 늦게 바꿀수록 재배선 비용이 커지므로, 버퍼 풀/소유권 프로토콜을 토폴로지 기술(구성)로 선언하는 구조를 지금 확정해야 함

**→ 그래서**: 채널을 코드가 아닌 구성(풀 배선 1회 / 소유권 프로토콜)으로 수용하는 C1-1·C1-3 계열이 필요함.

### 1.3 유의 사항

1. "FF-A 기술 개발이 느림" 같은 외부 제약 포인트는 확장성 절에 포함했으나, `08_candidate_architectures.md` 2.5절 취지대로라면 C1-1/C1-3 모두 EL2(guest-guest 프리미티브) 확보가 선행 관문이라는 리스크 언급이므로, 발표 흐름상 "문제점"이 아니라 "채택 조건"으로 분리하는 것도 방법이다.
2. 엄밀히는 확장성(P-C1-3) 문제의 정면 해법은 C1-5(브로커 스타)이고 C1-1/C1-3은 부분 대응(비교표 "중")이다. C1-1/C1-3 제안 맥락이라면 기밀성 문제를 주 동기로, 확장성은 보조 동기로 배치하는 편이 논리가 매끄럽다.

## 2. DP-C2. HW IP 중재 위치와 공유 방식 — C2-1, C2-3 제안 배경

C2-1(Host 드라이버 시분할 + 보호 DMA 검증)과 C2-3(배타적 소유권 스위칭)은 둘 다 "격리 집행은 hypercall/S2MPU 뒤로, 기존 드라이버 자산은 유지"를 노리는 후보로, 기밀성·성능이 주 동기가 된다.

### 2.1 (기밀성) 격리 집행이 비신뢰 주체가 위조할 수 없는 위치에 있어야 함.

- Camera/AI HW는 다중 Context를 지원하지 않는 단일 Context IP(CS-05)라, Host 일반 기능과 pVM 보안 파이프라인이 같은 IP를 공유해야 함 (R-2, FR-03)
- 중재자가 Host 커널 드라이버 그 자체이면, 비신뢰 주체가 HW 레지스터·DMA 디스크립터를 직접 작성함 — DMA 대상을 비보호 메모리로 돌리는 순간 보안 프레임 원본이 노출됨 (QA-01 붕괴)
- 보안 인증 게이트는 사용 주체 전환 시 S2MPU 권한 중첩 0(SEC-02, 무위반 누적 전환 10^6회)과 비할당 도메인 DMA 접근 0(SEC-03, Thunderclap형 공격 벡터 전수 차단)을 요구함 — Host 드라이버 전 경로를 검증해야 하는 구조로는 격리 논증 비용이 OO(인월)로 커져 인증 대응이 사실상 불가능함
- 전환 시 잔류 컨텍스트·버퍼가 소거되지 않으면 다음 소유자(root Host 포함)가 이전 세션의 프레임을 덤프할 수 있음 (VOS-08)

**→ 그래서**: 스케줄링은 Host에 두되 S2MPU 프로그래밍·DMA 매핑 검증을 hypercall 뒤로 숨기는 **C2-1**, 소유 기간에는 상대 도메인의 매핑 자체가 존재하지 않고 hand-off에 소거가 강제 내장되는 **C2-3**이 필요함.

### 2.2 (성능) 사용 주체 전환 비용이 프레임 주기와 결합하지 않아야 함.

- Host 세션과 pVM 세션이 교차할 때마다 HW 컨텍스트 저장/복원 + S2MPU 재프로그래밍 + 잔류 소거가 발생함 — 전환 1회에 OOms 소요
- 이 전환이 프레임 주기 33ms(30fps)마다 발생하면 실시간 예산을 직접 잠식함 — 합격 기준은 전환 지연 p99 10ms 이하(주기의 1/3, PERF-04)
- 중재자를 별도 도메인(Driver VM)에 두면 매 요청이 가상 디바이스 경유가 되어 보안 경로뿐 아니라 Host 일반 경로까지 OO% 저하됨 — 산업 요구는 비격리 대비 저하 10% 이내, 30fps 드롭율 0.1% 이하 (PERF-01)
- 로봇(상시 감시)·드론(고속 기동)에서 요구 프레임률은 OO fps, OO fps — 전환 비용이 주기와 결합하는 구조로는 만족 불가

**→ 그래서**: 할당 기간 동안 사용 주체가 HW를 네이티브로 직접 제어해 중재 오버헤드가 0인 **C2-3**, 성숙한 기존 드라이버의 스케줄링·전원 관리를 재사용해 전환 로직만 얹는 **C2-1**이 필요함.

### 2.3 (확장성) 중재 구조가 도메인·IP 수 증가에 닫혀 있어야 함.

- 보안 도메인(pVM)이 OO개로 늘면 세션 교차 조합이 늘어 전환 빈도가 OO배 증가함 — 중재자가 도메인별 특수 처리를 코드로 갖는 구조면 도메인 추가마다 코어 수정 발생 (QA-03 "코어 수정 0 LoC" 게이트 위반)
- 신규 HW IP(ISP, NPU 등) 온보딩 시 중재 로직 전체를 재작성해야 하면 IP당 이식 공수 OO인월이 반복됨 — 중재를 "소유권 원장 + hand-off 프로토콜"이라는 IP 중립 계약으로 정의하면 IP 추가가 드라이버 등록만으로 흡수됨
- 중재자를 전용 Driver VM으로 세우는 방식은 IP마다 Driver VM 자원(OO MB)이 상시 점유되어 QA-06 한도와 곱으로 경합함

**→ 그래서**: 중재 계약을 소유권 원장/전환 프로토콜로 표준화하는 **C2-3**, 기존 드라이버 확장만으로 수용하는 **C2-1**이 필요함.

### 2.4 (가용성, 대체 축) 중재자 장애가 최소 반경으로 갇혀야 함.

> 확장성 축은 `08_candidate_architectures.md`의 DP-C2 문제 정의(P-C2-1/2/3)에 없다. 문서상 DP-C2의 세 번째 문제는 가용성(P-C2-3)이므로, 발표 근거를 문서와 일치시키려면 2.3절 대신 아래를 사용하는 것을 권장한다.

- 중재자가 죽으면 Host 일반 기능과 pVM 보안 파이프라인이 동시에 HW를 잃음 — 장애 반경이 중재자 위치에 따라 달라짐 (P-C2-3, QA-05)
- 전환 프로토콜 중간 실패 시 HW가 어느 쪽에도 속하지 않는 고아 상태가 되면 복구까지 OO초 양쪽 정지 — 게이트는 장애 pVM 외 다운타임 0 (AVL-01)
- 중재 로직이 크고 상태가 집중될수록(전용 Driver VM) 단일 장애점이 커짐 — 중재를 얇게(소유권 원장 수준) 유지해야 장애 반경이 최소화됨

### 2.5 유의 사항

- 2.3절(확장성)의 "IP 중립 계약", "IP당 이식 공수" 논거는 `08_candidate_architectures.md`에 명시된 내용이 아니라 본 문서에서 구성한 주장이므로, 발표에 사용하려면 방어 가능한지 확인이 필요하다. 기밀성·성능 불릿은 전부 문서(P-C2-1/2, SEC-02/03, PERF-01/04, VOS-08) 근거가 있다.
