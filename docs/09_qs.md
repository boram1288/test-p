# Quality Scenario/Specification (QS)

본 문서는 pKVM 기반 가상화 보안 Framework의 핵심 품질 속성 7개에 대한 QS를 정의한다. 작성 형식과 규칙은 `qs-writer` 스킬을 따르며, 품질 특성 분류는 ISO/IEC 25010, 측정 함수는 ISO/IEC 25023에 근거한다.

- 설계 대상(Artifact 범위)은 pKVM 기반 가상화 보안 Framework로 한정한다. Camera Framework, AI Framework, Host Application, Workload, pKVM 커널(EL2), Secure OS는 시스템 외부이므로 Artifact로 지정하지 않는다.
- 별점(3/2/1개) 평가 기준은 별도 문서 `09_qs_evaluation.md`에 정리한다.
- KPI 수치의 출처는 각 QS 하단이 아닌 본 문서 말미의 "KPI 근거" 절에 번호로 정리하고, 본문에서는 `[근거-n]`으로 참조한다.

---

## QS-SEC-01: 보안 - Host 침해 시 격리 기밀성

| Field | Specification |
|-------|---------------|
| Description | Host가 완전히 침해되어도 Framework가 구성한 어떤 경로를 통해서도 pVM 내 데이터(영상 원본, 모델, 중간 데이터)가 노출되지 않아야 한다. (ISO 25010: Security / Confidentiality) |
| Source | Host OS 루트(커널) 권한을 탈취한 공격자 |
| Stimulus | Framework가 구성한 경로(pVM 간 공유 채널, HW IP 접근 경로, 결과 전달 채널, 제어 인터페이스)를 이용한 pVM 메모리 및 전달 중 데이터 접근 시도 |
| Environment | Secure Vision AI 파이프라인 정상 운용 중 |
| Artifact | Framework의 pVM 간 데이터 공유 채널 구성 모듈, HW IP 접근 중개 경로, 결과 전달 채널 |
| Response | 모든 접근 시도가 차단되고, 비정상 접근 시도가 감사 기록으로 남는다 |
| Response Measure | KPI-1: Host 루트 권한 침투 시험 전 케이스에서 pVM 데이터 노출 0건. 측정 함수(ISO 25023 접근 통제성 변형): X = 차단된 비인가 접근 수 / 시도된 비인가 접근 수 = 1.0 [근거-1][근거-2]. 측정 방법: Host 침해 모사 도구로 정의된 공격 벡터 목록 전체를 자동화 주입하고 pVM 메모리/채널 버퍼 덤프 비교 |

## QS-SEC-02: 보안 - Workload 진위성 및 요청 인가

| Field | Specification |
|-------|---------------|
| Description | Framework는 외부 산출물인 Workload와 제어 요청을 신뢰하지 않고, 검증 실패 이미지의 실행과 비인가 요청을 전부 거부해야 한다. (ISO 25010: Security / Authenticity, Integrity) |
| Source | 변조된 Workload 이미지를 공급하는 공격자, 비인가 Host 프로세스 |
| Stimulus | 서명이 없거나 변조된 Workload 이미지의 탑재 요청, 인가되지 않은 주체의 파이프라인 제어(시작/종료/자원) 요청 |
| Environment | 파이프라인 기동 단계 및 정상 운용 중 |
| Artifact | Framework의 Workload 이미지 검증/탑재 모듈, 파이프라인 제어 API의 인증/인가 모듈 |
| Response | 검증 실패 이미지는 탑재/실행이 중단되고, 비인가 요청은 거부되며, 모든 거부가 감사 기록으로 남는다 |
| Response Measure | KPI-1: 변조/미서명 이미지 실행 차단율 100% (X = 차단 건수 / 변조 이미지 주입 건수 = 1.0) [근거-3]. KPI-2: 비인가 제어 요청 거부율 100% [근거-1]. 측정 방법: 서명 훼손/바이트 변조/버전 롤백 이미지 셋과 비인가 주체 요청 셋을 자동화 주입 |

## QS-COMP-01: 상호운용성 - 외부 SW 자산 무수정 수용

| Field | Specification |
|-------|---------------|
| Description | 시스템 외부인 Camera/AI Framework와 기존 TrustZone Secure OS 기반 자산(GP TEE API 클라이언트/TA)이 원본 수정 없이 Framework의 실행 환경에서 동작해야 한다. (ISO 25010: Compatibility / Interoperability) |
| Source | 외부 SW 공급자 (Camera/AI Framework 벤더, Secure OS 벤더, 기존 TA 자산 보유 제조사) |
| Stimulus | 기존 Camera/AI Framework를 포함한 Workload와 GP TEE API 기반 기존 자산을 본 Framework 환경에 통합 |
| Environment | 제품 통합/포팅 단계 |
| Artifact | Framework가 pVM 내부에 제공하는 게스트 실행 환경 인터페이스, Secure OS 연동(GP API 라우팅) 인터페이스 |
| Response | 외부 SW는 자신이 기대하는 표준 인터페이스(디바이스 노드, GP TEE Client API 등)를 그대로 사용하여 동작한다 |
| Response Measure | KPI-1: 외부 SW 원본 소스 수정 0 LoC (X = 수정된 외부 SW 파일 수 / 전체 외부 SW 파일 수 = 0) [근거-4]. 측정 함수(ISO 25023 상호운용성 변형): 교환 대상 인터페이스 중 표준 규격대로 동작한 비율 = 1.0 [근거-1]. 측정 방법: 통합 전후 외부 SW 소스 diff 및 GP TEE Client API 적합성 시험 |

## QS-PERF-01: 성능 효율성 - 격리 하 실시간 파이프라인

| Field | Specification |
|-------|---------------|
| Description | 격리 중개(pVM 간 프레임 공유, HW 접근 경로)가 파이프라인의 실시간성을 해치지 않아야 한다. (ISO 25010: Performance Efficiency / Time Behaviour, Resource Utilization) |
| Source | Camera pVM (프레임 단위 대용량 영상을 반복 전달) |
| Stimulus | 1080p 30fps 영상 프레임의 지속적 캡처-공유-추론-결과 전달 반복 |
| Environment | 파이프라인 정상 운용 중 (Host 일반 기능 동시 동작) |
| Artifact | Framework의 pVM 간 데이터 공유 채널, HW IP 접근 중개 경로, pVM 생명주기 관리자(기동 경로) |
| Response | 프레임은 복사 없이 도메인 간 이동하고, 파이프라인은 카메라 프레임레이트를 유지한다 |
| Response Measure | KPI-1: 프레임 전달 경로의 CPU memcpy 0회 (dma-buf 기반 zero-copy, bounce buffer 미사용) [근거-5][근거-6]. KPI-2: 도메인 간 채널 유효 처리량 >= 0.75 Gbps (1080p30 NV12 = 1920x1080x1.5B x 30fps = 약 93.3MB/s 산출치; virtio-vsock 실측 상한 G2H 약 40Gbps 대비 여유 확인) [근거-7]. KPI-3: 프레임당 End-to-End 처리 시간 <= 33.3ms (30fps 프레임 주기) [근거-8]. KPI-4: pVM 콜드 스타트 시간 목표치 TBD(근거 조사 필요) — 참고 분포: VM 기반 격리 플랫폼 기동 실측 100~800ms [근거-9]. 측정 방법: 프레임 타임스탬프 추적, perf/ftrace로 복사 호출 계수, iperf-vsock 처리량 측정 |

## QS-REL-01: 신뢰성 - pVM 장애 격리와 자동 복구

| Field | Specification |
|-------|---------------|
| Description | 통제 불가능한 외부 Workload의 장애가 Host, 타 pVM, 로봇 기본 동작으로 전파되지 않고 Framework에 의해 자동 복구되어야 한다. (ISO 25010: Reliability / Fault Tolerance, Recoverability, Availability) |
| Source | 오동작하는 Workload (외부 산출물) |
| Stimulus | pVM 내 Workload의 크래시, 무응답(행), 자원 고갈성 오동작 |
| Environment | 다중 pVM 운용 중인 로봇 통상 동작 상태 |
| Artifact | Framework의 pVM 생명주기 관리자, 장애 감지/자원 회수/재시작 모듈 |
| Response | 장애는 해당 pVM에 한정되고, Framework가 자원을 회수한 뒤 파이프라인을 재기동하며, 장애 이력이 기록된다 |
| Response Measure | KPI-1: 장애 주입 시험 전 케이스에서 Host/타 pVM으로의 장애 전파 0건. KPI-2: 자동 복구 성공률 100% (X = 자동 복구 성공 수 / 장애 발생 수, ISO 25023 회복성 측정 함수) [근거-1]. KPI-3: 평균 복구 시간(MTTR) 목표치 TBD(근거 조사 필요) — 복구 시간은 pVM 재기동 시간에 종속되므로 KPI-4(QS-PERF-01)와 함께 확정, 참고 분포 [근거-9]. 측정 방법: 크래시/행/메모리 고갈 장애 주입 자동화, 복구 시간 로그 측정 |

## QS-FLEX-01: 유연성 - SoC/도메인 확장 이식성

| Field | Specification |
|-------|---------------|
| Description | 신규 커스텀 SoC, 신규 Linux 배포판, 신규 보안 도메인(제3의 pVM) 추가 시 Framework 코어의 수정 없이 포팅 계층 작업만으로 대응해야 한다. (ISO 25010: Flexibility / Adaptability, Scalability) |
| Source | SoC 벤더, 로봇 제품 개발자 |
| Stimulus | HW IP 구성이 다른 신규 SoC로의 포팅, 신규 배포판(Yocto/Ubuntu 계열) 적용, 신규 보안 도메인 추가 |
| Environment | 제품 개발/포팅 단계 |
| Artifact | Framework 코어 모듈과 HW/플랫폼 추상화 계층의 분리 구조 |
| Response | 변경은 사전 정의된 추상화 계층(HW 접근 백엔드, 플랫폼 어댑터)에 국한되고 코어 모듈은 재사용된다 |
| Response Measure | KPI-1: 포팅 시 코어 모듈 수정 0건 (ISO 25023 적응성 측정 함수 변형: X = 수정된 코어 모듈 수 / 전체 코어 모듈 수 = 0) [근거-1]. KPI-2: 신규 도메인 추가 시 기존 도메인 코드 수정 0건. 근거: 코어/벤더 모듈 분리 구조의 대규모 실증 사례(Android GKI: 분리 이전 커널 코드 최대 50%가 SoC별 out-of-tree 커스텀이었으며 GKI가 코어 불변+벤더 모듈 구조로 해소) [근거-10]. 측정 방법: 포팅 전후 모듈별 diff 집계 |

## QS-MAINT-01: 유지보수성 - mainline 커널 추종

| Field | Specification |
|-------|---------------|
| Description | pKVM은 upstream에서 진화 중인 메커니즘이므로, Framework의 커널 의존부가 mainline 커널 업그레이드를 저비용으로 추종할 수 있어야 한다. (ISO 25010: Maintainability / Modifiability) |
| Source | Framework 유지보수 개발자 |
| Stimulus | pKVM을 포함한 mainline 커널의 신규 LTS 버전으로의 업그레이드 |
| Environment | 제품 유지보수 단계 |
| Artifact | Framework의 커널 의존 계층 (pVM 관리/HW 공유용 커널 드라이버 및 모듈) |
| Response | 커널측 기능은 로더블 모듈로 국소화되어 있어 코어 커널 소스에 대한 사설 패치 없이 업그레이드가 완료된다 |
| Response Measure | KPI-1: 코어 커널 소스에 대한 out-of-tree 수정 패치 0건 (Framework 커널 기능은 전부 로더블 모듈로 제공) [근거-10][근거-11]. KPI-2: 커널 업그레이드 시 수정 범위가 커널 의존 계층 모듈 내로 한정 (코어 Framework 수정 0건). 측정 방법: 커널 트리 diff 검사(사설 패치 검출), LTS 업그레이드 리허설 후 수정 파일 목록 집계 |

## KPI 근거

| 번호 | 출처 |
|------|------|
| 근거-1 | ISO/IEC 25023:2016, Systems and software engineering - Measurement of system and software product quality. 접근 통제성/회복성/적응성/상호운용성 측정 함수(X = A/B 형태)의 출전 |
| 근거-2 | Linux Kernel documentation, "Protected KVM (pKVM)" — pKVM의 설계 목표가 침해된 Host로부터 pVM 메모리 기밀성/무결성 보호임을 명시 (https://www.kernel.org/doc/html/next/virt/kvm/arm/pkvm.html). 보조: Google pKVM SESIP Level 5 인증 (2025) — 격리 보증의 제3자 평가 선례 |
| 근거-3 | AOSP, "Microdroid" / "AVF architecture" — pvmfw가 pVM payload를 검증하며 검증 실패 시 부팅을 진행하지 않는 verified boot 체인(dm-verity 포함)이 업계 구현 표준임 (https://source.android.com/docs/core/virtualization/microdroid) |
| 근거-4 | GlobalPlatform, "TEE Client API Specification v1.0" (GPD_SPE_007) — 표준 API 준수 시 클라이언트/TA의 소스 수준 호환성이 규격의 목적. 기존 자산 무수정 수용(0 LoC)의 판정 기준 |
| 근거-5 | Linux Kernel documentation, "Buffer Sharing and Synchronization (dma-buf)" — dma-buf는 디바이스/서브시스템 간 버퍼를 복사 없이 공유하기 위한 커널 표준 프레임워크 (https://docs.kernel.org/driver-api/dma-buf.html) |
| 근거-6 | Wei et al., "Measuring and Optimizing the Performance of the Android Virtualization Framework", ACM SAC 2024 (https://dl.acm.org/doi/10.1145/3605098.3636097) — pVM의 주된 I/O 오버헤드 원인이 SWIOTLB bounce buffering(이중 복사)임을 실측으로 규명. 최적화로 네트워크 8~22%, 블록 I/O 최대 7% 개선. zero-copy 경로가 성능 기준선임을 뒷받침 |
| 근거-7 | S. Garzarella, "virtio-vsock in QEMU, Firecracker and Linux", KVM Forum 2019 (https://stefano-garzarella.github.io/posts/2019-11-08-kvmforum-2019-vsock/) — Linux v5.4 기준 virtio-vsock 실측 처리량 Guest→Host 약 40Gbps(64KB payload), Host→Guest 약 25Gbps(4KB 수신 버퍼). 요구 처리량은 시나리오 산술치(1080p30 NV12 약 0.75Gbps)로 도출 |
| 근거-8 | 프레임 주기 산술치 — 레퍼런스 시나리오(`02_reference_scenario.md`)의 카메라 표준 동작점 30fps에서 프레임 주기 = 1/30s = 33.3ms. 실시간 유지의 정의상 프레임당 처리 시간이 프레임 주기를 넘을 수 없음 |
| 근거-9 | van Rijn & Rellermeyer, "A Fresh Look at the Architecture and Performance of Contemporary Isolation Platforms" (arXiv:2110.11462) — 격리 플랫폼 기동 시간 300회 반복 실측: Docker 약 100ms, gVisor 약 190ms, Kata(VM 기반) 약 600ms, LXC 약 800ms. pVM 기동 목표치 확정 전 참고 분포 |
| 근거-10 | AOSP, "Generic Kernel Image (GKI) project" (https://source.android.com/docs/core/architecture/kernel/generic-kernel-image) — GKI 이전 디바이스 커널 코드의 최대 50%가 out-of-tree였고, GKI는 코어 커널 불변 + 벤더 로더블 모듈 + 안정 KMI 구조로 이를 해소. 코어/모듈 분리와 out-of-tree 패치 0건 정책의 대규모 실증 사례 |
| 근거-11 | LWN.net, "The intersection of modules, GKI, and rocket science" (https://lwn.net/Articles/872209/) — 벤더는 GKI 코어 커널을 교체/수정할 수 없고 모듈만 공급 가능하다는 정책과 그 유지보수 효과에 대한 커뮤니티 분석 |
