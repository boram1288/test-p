# 요구사항 도출

> 본 문서는 `01_requirements_collection.md`에서 수집한 VOS-01~VOS-16을 입력으로, 요구사항을 **FR(기능 요구사항), QA(품질 속성 요구사항), CONST(제약사항)** 로 분류/정제한다.
> 진행 순서: 요구사항 수집 → **요구사항 도출(본 문서)** → 품질 속성 선정(QAW) → Architectural Driver 선정
>
> FR에서 도출된 유즈케이스 명세는 [`01_use_case.md`](01_use_case.md) 및 [`01_use_case_spec.md`](01_use_case_spec.md) 참조.

---

## 1. 도출 원칙

- **추적성**: 모든 요구사항은 하나 이상의 VOS에서 도출되며, 출처 VOS를 명시한다. 역으로 모든 VOS는 하나 이상의 요구사항에 반영함을 원칙으로 하되, 요구사항으로 도출하지 않는 VOS는 그 사유를 명시한다(5절 추적 매트릭스).
- **분류 기준**:
  - **FR**: 시스템이 "무엇을 하는가". Framework가 제공해야 할 기능.
  - **QA**: 시스템이 "얼마나 잘 하는가". 자극(stimulus)과 응답(response)으로 표현 가능한 품질 속성. 측정 가능한 응답 측정치(response measure)는 3단계 QAW에서 구체화한다.
  - **CONST**: 설계 선택지를 제한하는 기술/조직/규제상 제약. 협상 대상이 아닌 전제 조건.
- **명명**: FR-nn / QA-nn / CONST-nn.

## 3. QA (품질 속성 요구사항)

각 QA는 "자극 → 기대 응답" 형태로 기술한다. 정량적 응답 측정치는 3단계 QAW에서 시나리오로 구체화한다.

| ID | 품질 속성 | 요구사항 | 측정 방법 | 출처 VOS |
|----|----------|----------|-----------|----------|
| QA-01 | **보안 (기밀성)** | Host가 완전히 침해되어도 pVM 내 영상/모델/추론 데이터는 노출되지 않는다. | 격리 메모리 노출 0건 (차단율 100%) | VOS-01, VOS-15 |
| QA-02 | **성능 (실시간 처리)** | 영상 입력이 유입될 때 Camera/AI HW 가속으로 격리 환경에서도 실시간 처리 속도를 만족한다. | 30fps 유지, E2E 지연 100ms 이하, 비격리 대비 성능 저하 10% 이내 (가정치) | VOS-02 |
| QA-03 | **성능 (통신 오버헤드)** | 도메인 간 대용량 데이터 전달 시 통신 오버헤드가 실시간성을 해치지 않는다. | zero-copy 또는 프레임당 전달 지연 5ms 이하 (가정치) | VOS-09 |
| QA-04 | **자원 효율** | 보안 기능 활성화 시 메모리/전력 오버헤드가 탑재 가능 한도 이내로 유지된다. | 추가 메모리 256MB 이하, 전력 증가 5% 이내 (가정치) | VOS-04 |
| QA-05 | **확장성** | 신규 Workload 추가 시 소스 수정 없이 패키징/탑재만으로 수용된다. | Framework 코어 수정 0 LoC, Workload 패키지 작성/탑재만으로 동작 | VOS-05 |
| QA-06 | **변경 용이성 (Secure OS 이식)** | Secure OS/Framework 버전이 변경될 때 정의된 인터페이스만으로 대응하고 재이식하지 않는다. | Secure OS 외 모듈의 수정 파일 0개 | VOS-11 |
| QA-07 | **가용성** | pVM이 비정상 종료/오동작해도 Host/다른 pVM/로봇 기본 동작은 유지된다. | Host/타 pVM 다운타임 0, 장애 pVM 재시작 3초 이내 (가정치) | VOS-16 |
| QA-08 | **시험 용이성** | 격리 검증이 필요할 때 Host 침해를 모사한 시험으로 객관적으로 검증할 수 있다. | QA-01~03 자동화 시험 커버 100%, 회귀 시험 반복 실행 가능 | VOS-13 |
| QA-09 | **성능 (HW IP 공유 오버헤드)** | Host/pVM이 HW IP를 동시 사용할 때 중재 오버헤드가 양쪽의 실시간 요구를 해치지 않는다. | 목표 프레임율 유지(저하 10% 이내), 사용 주체 전환 오버헤드 1ms 이하 (가정치) | VOS-02 |

## 4. CONST (제약사항)

| ID | 제약사항 | 설명 | 출처 VOS |
|----|---------|------|----------|
| CONST-01 | **Linux 네이티브** | Android 스택에 의존하지 않고 Linux에서 네이티브로 동작한다. | VOS-03 |
| CONST-02 | **pKVM 커널 전제** | 기 포팅된 pKVM(EL2)을 전제로, EL2 수정 없이 hypercall 범위 내에서 설계한다. | VOS-10 |
| CONST-03 | **기존 TEE 경로 무회귀** | 기존 TrustZone SMC 경로/TEE 기능(키 관리/인증)은 변경 없이 그대로 동작해야 한다. | VOS-12 |
| CONST-04 | **규제 준수** | GDPR/개인정보보호법이 요구하는 기술적 격리를 충족/증빙해야 한다. | VOS-15 |
| CONST-05 | **타겟 HW** | Camera/AI HW 탑재 ARM 기반 로봇용 SoC 대상이며, 두 IP는 단일 Context HW로 HW 설계 변경은 불가하다. | VOS-02, VOS-08 |
| CONST-06 | **Secure OS 신규 개발 제외** | Secure OS는 신규 개발하지 않고 기존 Secure OS의 이식/수정만 다룬다. | - |

| VOS | 도출된 요구사항 |

| VOS-11 (Secure OS 이식 인터페이스 명확성) | FR-06, QA-06 |
| VOS-12 (기존 TrustZone 기능 무회귀) | FR-06, CONST-03 |
| VOS-13 (격리 보장의 객관적 검증) | QA-08 |
| VOS-14 (단순/문서화된 API) | (도출 제외 — 본 단계의 아키텍처 요구사항에서 제외, 상세 설계 단계에서 검토) |
| VOS-15 (시장의 프라이버시/규제 요구, GDPR 등) | QA-01, CONST-04 |
| VOS-16 (pVM 장애 시 무영향) | QA-07 |
