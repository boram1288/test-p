# 요구사항 도출

> 본 문서는 `01_vos_collection.md`와 유즈케이스 명세를 입력으로 하여 Secure Vision AI Platform의 기능 요구사항(FR), 품질 속성 요구사항(QA), 제약사항(CONST)을 도출한다.
>
> 진행 순서: 요구사항 수집 → **요구사항 도출(본 문서)** → 품질 속성 명세 → Architectural Driver 선정

관련 문서: [`01_use_case_spec.md`](01_use_case_spec.md)

---

## 1. 도출 원칙

- **FR (Functional Requirement)**: 시스템이 제공해야 하는 기능. 유즈케이스의 Main/Alternative/Exception Flow에서 도출한다.
- **QA (Quality Attribute Requirement)**: 기능이 만족해야 하는 품질 조건. VOS에서 반복적으로 등장한 보안, 성능, 확장성, 자원, 변경성, 가용성, 시험성을 도출한다.
- **CONST (Constraint)**: 설계 선택지를 제한하는 제약. 과제 전제, 기존 포팅 상태, 표준/호환성 요구에서 도출한다.
- **명명**: FR-nn / QA-nn / CS-nn.

## 2. FR (기능 요구사항)

| ID | 요구사항 | 설명 | 관련 UC | 출처 VOS |
|----|---------|------|--------|---------|
| FR-01 | **pVM 생성/시작/정지/종료** | pVM의 생성/시작/정지/종료와 자원 할당/회수를 관리하는 기능을 제공한다. | UC-01 | VOS-01, VOS-07 |
| FR-02 | **다중 pVM 동시 운용** | Secure Camera, Secure AI 등 복수 pVM을 독립적으로 동시 운용하는 기능을 제공한다. | UC-02 | VOS-07 |
| FR-03 | **Camera/AI HW 공유 사용** | Camera/AI HW를 Host/pVM이 SW 중재로 공유하고, 사용 주체 전환 시 격리를 보장하는 기능을 제공한다. | UC-03 | VOS-02, VOS-08 |
| FR-04 | **도메인 간 보안 데이터 전송** | pVM↔pVM, pVM↔Host 간 데이터를 비신뢰 주체에 노출 없이 전달하는 보안 채널 기능을 제공한다. | UC-04 | VOS-01, VOS-09 |
| FR-05 | **보안 Workload 동적 탑재** | 신규 보안 Workload를 재배포 없이 서명 검증 후 동적으로 pVM에 탑재하는 기능을 제공한다. | UC-05 | VOS-05 |
| FR-06 | **Secure OS ENC/DEC 명령 전송** | pVM 내에서 ENC/DEC를 Secure OS에 전달하는 기능을 제공한다. | UC-06 | VOS-11, VOS-12 |

## 3. QA (품질 속성 요구사항)

QA 번호는 `03_utility_tree.md`의 리프 시나리오 우선순위 순서와 일치시킨다.

| ID | 품질 속성 | 요구사항 | 측정 방법 | 출처 VOS |
|----|----------|---------|----------|---------|
| QA-01 | **보안 (기밀성)** | Host가 완전히 침해되어도 pVM 내 영상/모델/추론 데이터는 노출되지 않는다. | 격리 메모리 노출 0건 (차단율 100%) | VOS-01, VOS-15 |
| QA-02 | **보안 (HW 접근 격리)** | Camera/AI HW 사용 주체 전환 시 배타적 접근이 항상 보장된다. | S2MPU 권한 상태 로그에서 두 주체의 권한 중첩 구간 없음 (위반 횟수 = 0) | VOS-02 |
| QA-03 | **성능 (실시간 처리)** | 영상 입력이 유입될 때 Camera/AI HW 가속으로 격리 환경에서도 실시간 처리 속도를 만족한다. | 30fps 유지, E2E 지연 100ms 이하, 비격리 대비 성능 저하 10% 이내 (가정치) | VOS-02 |
| QA-04 | **확장성** | 신규 Workload 추가 시 소스 수정 없이 패키징/탑재만으로 수용된다. | Framework 코어 수정 0 LoC, Workload 패키지 작성/탑재만으로 동작 | VOS-05 |
| QA-05 | **성능 (통신 오버헤드)** | 도메인 간 대용량 데이터 전달 시 통신 오버헤드가 실시간성을 해치지 않는다. | zero-copy 또는 프레임당 전달 지연 5ms 이하 (가정치) | VOS-09 |
| QA-06 | **가용성** | pVM이 비정상 종료/오동작해도 Host/다른 pVM/로봇 기본 동작은 유지된다. | Host/타 pVM 다운타임 0, 장애 pVM 재시작 3초 이내 (가정치) | VOS-16 |
| QA-07 | **자원 효율** | 보안 기능 활성화 시 메모리/전력 오버헤드가 탑재 가능 한도 이내로 유지된다. | 추가 메모리 256MB 이하, 전력 증가 5% 이내 (가정치) | VOS-04 |
| QA-08 | **시험 용이성** | 격리 검증이 필요할 때 Host 침해를 모사한 시험으로 객관적으로 검증할 수 있다. | 주요 격리 요구사항(QA-01, QA-02, QA-05) 자동화 시험 커버 100%, 회귀 시험 반복 실행 가능 | VOS-13 |
| QA-09 | **변경 용이성 (Secure OS 이식)** | Secure OS/Framework 버전이 변경될 때 정의된 인터페이스만으로 대응하고 재이식하지 않는다. | Secure OS 외 모듈의 수정 파일 0개 | VOS-11 |

## 4. CONST (제약사항)

| ID | 제약사항 | 설명 | 출처 VOS |
|----|---------|------|---------|
| CS-01 | **Linux 네이티브** | Android 스택에 의존하지 않고 Linux에서 네이티브로 동작한다. | VOS-03 |
| CS-02 | **pKVM 커널 전제** | 기 포팅된 pKVM(EL2)을 전제로, EL2 수정 없이 hypercall 범위 내에서 설계한다. | VOS-10 |
| CS-03 | **TrustZone 기능 무회귀** | 기존 OP-TEE/TrustZone 기반 ENC/DEC 기능은 유지되어야 한다. | VOS-12 |
| CS-04 | **개인정보/규제 고려** | 영상/AI 데이터는 개인정보 및 규제 요구를 고려하여 보호되어야 한다. | VOS-15 |
| CS-05 | **단일 Context HW IP** | Camera/AI HW가 다중 Context를 하드웨어적으로 제공하지 않는 경우 SW 중재로 공유해야 한다. | VOS-02, VOS-08 |
| CS-06 | **서명된 Workload만 탑재** | 동적 탑재되는 Workload는 서명 검증을 통과해야 한다. | VOS-05 |
