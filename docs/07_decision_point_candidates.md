> 본 문서는 레퍼런스 시나리오(`99_reference_scenario_flow.md`)와 과제 요구사항(`00_overview.md`의 R-1~R-5, `02_requirements.md`의 FR/QA/CS)을 기준으로, Linux pKVM 기반 가상화 보안 Framework 구조 설계에서 결정이 필요한 지점(Decision Point)을 새로 도출한 것이다.

- 레퍼런스 시나리오 13단계의 실행 경로에서 구조적 대안이 존재하는 지점

## 3. A. 제어 구조 관점

### DP-A1. pVM 관리 골격(제어 평면) 구조

- **결정 질문**: pVM 생명주기를 관장하는 주체를 단일 중앙 데몬으로 둘 것인가, pVM별 관리 인스턴스로 분산할 것인가. 제어 평면(생성/정책)과 데이터 평면(프레임 전달)을 분리할 것인가.
- **대안 방향**: 중앙집중형 Manager / pVM별 위임형(per-instance monitor) / 제어·데이터 평면 분리형
- **품질속성**: 기밀성(관리 주체가 TCB에 포함되는 범위 최소화), 확장성(신규 도메인 추가 비용), 가용성(관리 데몬이 단일 장애점이 되는가)

### DP-A2. Host 노출 API 계층과 권한 결정 지점

- **결정 질문**: Framework API를 데몬+IPC로 제공할 것인가 라이브러리 직결로 제공할 것인가. 시나리오 2단계의 "요청 권한·정책 확인"에서 정책 결정 지점(PDP)과 집행 지점(PEP)을 어디에 둘 것인가.
- **대안 방향**: 데몬+IPC 게이트웨이 / 라이브러리 직결 / 혼합(제어는 데몬, 데이터는 직결)
- **품질속성**: 사용 편의성(VOS-14, 비보안 전문가용 API), 보안(요청 주체 인증), 기존 GP Client API와의 공존(CS-03)

### DP-A3. 장애 검출·복구 정책 구조

- **결정 질문**: pVM 헬스 검출 방식(heartbeat, watchdog, exit-reason 감시)과 복구 단위를 무엇으로 할 것인가 — pVM 단위 재시작인가, Camera+AI 파이프라인 전체 재구성인가. 장애 전파를 어디서 차단하는가.
- **대안 방향**: pVM 단위 재시작 / 파이프라인 단위 재구성 / 단계적(escalation) 복구
- **품질속성**: 가용성(시나리오 13단계 "장애를 해당 pVM에 한정"), 성능(복구 시간), 기밀성(장애 시 자원 회수·잔류 데이터 처리)

## 4. B. 실행 환경과 신뢰 체인 관점

### DP-B1. pVM 게스트 실행 환경(페이로드 OS) 선택

- **결정 질문**: Workload를 담는 pVM 내부 실행 환경을 무엇으로 할 것인가 — 최소 Linux(Microdroid류), 경량 유니커널/RTOS, 이식된 Secure OS 직접 탑재.
- **대안 방향**: 최소 Linux 게스트 / 경량 유니커널·RTOS / 이식 Secure OS 직접 탑재
- **품질속성**: 기밀성(pVM 내부 TCB 규모), 확장성(다양한 Workload 수용, R-4), 성능(부팅 시간·메모리 풋프린트), 범위 제약(신규 Secure OS from-scratch 개발 제외)

### DP-B2. pVM 프로비저닝 시점 전략

- **결정 질문**: pVM을 부팅 시 사전 생성해 풀로 유지할 것인가, 요청 시(시나리오 4단계) 동적 생성할 것인가, 웜 풀 하이브리드로 갈 것인가.
- **대안 방향**: 부팅 시 사전 생성 풀 / 요청 시 동적 생성 / 웜 풀 하이브리드
- **품질속성**: 성능(파이프라인 시작 지연), 자원 효율(유휴 메모리 점유), 가용성(재시작 시간), 동적 확장성(R-4)

### DP-B3. Workload 신뢰 체인(검증 아키텍처)

- **결정 질문**: 시나리오 3단계의 이미지 서명·manifest·무결성 검증을 누가 수행하는가 — Host Framework(비신뢰 영역), pVM 내부 초기 로더, 또는 이중 검증. 서명 형식·버전 정책·폐기(revocation) 관리는 어디에 두는가.
- **대안 방향**: Host Framework 단일 검증 / pVM 내부 검증 / Host 사전 검증 + pVM 최종 검증(이중)
- **품질속성**: 무결성/기밀성(CS-06 "서명된 Workload만 탑재", Host 비신뢰 원칙 R-1과의 정합), 확장성(신규 Workload 배포 절차)

## 5. C. 데이터 경로와 HW 공유 관점

### DP-C1. 도메인 간 프레임 전달 채널 구조

- **결정 질문**: Camera pVM에서 AI pVM으로 영상 프레임(시나리오 9단계)을 어떤 메커니즘으로 넘길 것인가 — pKVM lend/share 기반 공유 메모리 zero-copy, virtio/vsock 스트림, 하이퍼바이저 중재 복사.
- **대안 방향**: lend/share 공유 메모리 zero-copy / virtio·vsock 스트림 / 하이퍼바이저 중재 복사
- **품질속성**: 성능(실시간 프레임률, 복사 횟수), 기밀성(전달 구간에서 Host 매핑 차단, VOS-09), 확장성(도메인 수 증가 시 채널 토폴로지)

### DP-C2. HW IP 중재(Mediation) 위치와 공유 방식

- **결정 질문**: 단일 Context인 Camera/AI HW(CS-05)를 Host 일반 기능과 pVM이 동시 사용(R-2)하도록 중재하는 주체를 어디에 두는가 — Host 커널 드라이버의 시분할 중재, 전용 디바이스 소유 pVM(driver VM), 요청별 컨텍스트 스위칭 직접 할당.
- **대안 방향**: Host 커널 드라이버 시분할 중재 / 전용 디바이스 소유 pVM(driver VM) / 요청별 직접 할당 + 컨텍스트 스위칭
- **품질속성**: 성능(중재 오버헤드, fps), 기밀성(중재자가 보안 프레임에 접근 가능한가), 가용성(중재자 장애 시 Host·pVM 양쪽 영향)

### DP-C3. DMA 격리와 메모리 소유권 전환 프로토콜

- **결정 질문**: HW 사용 주체 전환 시 S2MPU 프로그래밍 책임을 누가 지는가(Host 드라이버, Framework, EL2 위임). 버퍼 대여→사용→회수 라이프사이클과 잔류 데이터 소거(VOS-08) 시점을 어떻게 정의하는가.
- **대안 방향**: Host 드라이버 책임 / Framework 커널 드라이버 책임 / EL2 hypercall 위임
- **품질속성**: 기밀성(DMA 우회 공격 차단), 성능(소유권 전환 빈도가 프레임 주기와 결합), 안전성(회수 실패 시 자원 누수)

### DP-C4. 민감 데이터 암복호화 실행 위치와 키 관리 경로

- **결정 질문**: 시나리오 8·11단계의 영상/AI 데이터 암호화 저장을 어디서 수행하는가 — Secure OS(TEE)에 SMC로 위임, pVM 내 자체 crypto + 키 sealing, 데이터는 pVM·키는 TEE의 혼합형.
- **대안 방향**: Secure OS(TEE) 전면 위임 / pVM 내 자체 crypto + 키 sealing / 혼합형(키는 TEE, 벌크 암호화는 pVM)
- **품질속성**: 기밀성(키 노출 면적), 성능(프레임당 ENC/DEC 지연), 상호운용성(R-5, GP API 경로 유지)

## 6. D. 기존 자산 공존 관점

### DP-D1. TrustZone Secure OS 공존 토폴로지

- **결정 질문**: pVM 도메인과 기존 TrustZone TEE의 역할 분담 구조 — pVM에 Secure OS를 이식해 병렬 TEE로 운용할 것인가, pVM은 TrustZone TEE의 프록시/클라이언트로 동작할 것인가. 두 경로가 공존할 때 SMC 경로 중재는 누가 하는가.
- **대안 방향**: pVM 병렬 TEE 운용 / pVM→TrustZone 프록시 / 역할 분담형(신규 기능은 pVM, 기존 기능은 TZ 유지)
- **품질속성**: 상호운용성(R-5, CS-03, VOS-12 "기존 SMC 경로 유지"), 기밀성(두 보안 세계 간 신뢰 경계), 확장성(Secure OS 이식 인터페이스 안정성, VOS-11)

## 7. E. 운영 품질과 진화 관점

### DP-E1. pVM 자원 할당·스케줄링(QoS) 정책 구조

- **결정 질문**: pVM vCPU를 물리 코어에 고정 할당(pinning)할 것인가, Host 스케줄러와 공유할 것인가. 메모리는 정적 예약인가 동적 조정인가. Camera/AI 파이프라인의 실시간성을 Host 일반 부하와 어떻게 경합 없이 보장하는가.
- **대안 방향**: 코어 정적 분할 / 우선순위 기반 공유 스케줄링 / 파이프라인 구간별 혼합(캡처 경로만 고정)
- **품질속성**: 성능(30fps·E2E 지연 100ms, QA-02 — 스케줄링 경합이 실시간성을 깨는 지점), 자원 효율(추가 메모리 256MB·전력 5% 한도, QA-06), 가용성(Host 과부하 시 보안 파이프라인 보호)

### DP-E2. 격리 증빙(Attestation)과 감사 증적 구조

- **결정 질문**: "Host 침해 시에도 격리가 유지된다"를 검증 부서와 규제 대응 관점에서 어떻게 객관적 증거로 만들 것인가 — pVM 부팅·Workload 탑재 상태에 대한 측정/attestation 체계를 둘 것인가, 침해 모사 시험 훅(fault injection 인터페이스)을 Framework에 내장할 것인가, 감사 로그는 어디에 어떤 형태(변조 방지 여부)로 남기는가.
- **대안 방향**: 부팅 측정 기반 로컬 attestation / 시험 전용 빌드에만 검증 훅 포함 / 상시 감사 로깅 + 민감 데이터 배제 정책
- **품질속성**: 시험 용이성(QA-07, VOS-13 객관적 검증), 규제 증빙(VOS-15, CS-04), 기밀성(검증 훅·로그 자체가 새로운 공격 표면이 되지 않도록 하는 트레이드오프)

### DP-E3. 인터페이스 버저닝과 호환성 유지 전략

- **결정 질문**: Framework↔Workload, Framework↔Secure OS, Framework↔Host App의 세 경계 인터페이스를 어떤 방식으로 버전 관리하는가 — 안정 ABI 고정, manifest 기반 버전 협상, 어댑터 계층 삽입. Secure OS나 Framework 버전이 바뀔 때 어느 쪽이 맞추는가.
- **대안 방향**: 경계별 고정 ABI + 하위 호환 보장 / 탑재 시점 버전 협상(capability negotiation) / Secure OS Adapter 계층에 변환 책임 집중
- **품질속성**: 변경 용이성(QA-08 "Secure OS 외 모듈 수정 파일 0개", VOS-11 이식 인터페이스 안정성), 확장성(QA-03 코어 수정 0 LoC — 인터페이스 계약이 전제 조건), 상호운용성(GP API 경로와의 정합)

## 8. P. 성능 관점 (추가)

### DP-P1. 프레임 버퍼 풀과 Stage-2 매핑 수명주기 전략

- **결정 질문**: 30fps로 순환하는 영상 프레임 버퍼를 매 프레임 할당/해제할 것인가, 사전 할당된 버퍼 풀을 순환 재사용할 것인가. 도메인 간 소유권이 오가는 버퍼의 Stage-2 매핑을 매번 map/unmap(hypercall) 할 것인가, 상주 매핑으로 유지할 것인가. HW DMA용 연속 메모리(CMA) 확보는 어느 시점에 하는가.
- **대안 방향**: 프레임별 동적 할당 + 매번 매핑 전환 / 사전 할당 버퍼 풀 + 상주 매핑 / 풀 기반 + 소유권 전환 시에만 매핑 갱신
- **품질속성**: 성능(hypercall·TLB 관리 비용이 프레임 주기마다 발생하는가, QA-02·QA-04), 기밀성(상주 매핑이 넓어질수록 노출 창 확대 — 잔류 데이터 소거 시점과의 결합)

### DP-P2. 파이프라인 실행 모델과 도메인 간 알림 메커니즘

- **결정 질문**: 캡처→전달→추론(시나리오 7~12단계)을 프레임 단위 동기 RPC로 직렬 실행할 것인가, 단계별 비동기 파이프라이닝(이중/삼중 버퍼링)으로 겹쳐 실행할 것인가. 프레임 도착 알림은 인터럽트 주입으로 할 것인가, 공유 링 폴링으로 할 것인가 — VM exit 빈도와 지연의 트레이드오프를 어디에 두는가.
- **대안 방향**: 프레임 단위 동기 RPC / 비동기 파이프라이닝 + 인터럽트 알림 / 비동기 파이프라이닝 + 공유 링 폴링(또는 하이브리드 도어벨)
- **품질속성**: 성능(E2E 지연 100ms vs 처리량 30fps의 균형, VM exit·컨텍스트 스위치 빈도, QA-02·QA-04)

### DP-P3. AI 모델 가중치 로딩·캐싱 전략

- **결정 질문**: 암호화된 모델 가중치(시나리오 10단계)를 세션마다 검증+복호화+로드할 것인가, 최초 1회 로드 후 pVM 메모리에 상주 캐시할 것인가. 복호화된 가중치를 AI HW가 직접 읽는 메모리에 두는 방식과 모델 교체(hot swap) 시의 재로딩 경로를 어떻게 설계하는가.
- **대안 방향**: 세션마다 재로드 / pVM 상주 캐시 / 부분 캐시(핫 모델만 상주) + 교체 시 재검증
- **품질속성**: 성능(추론 시작 지연, 모델 교체 시간, QA-02), 기밀성(복호화된 가중치의 상주 시간·위치), 자원 효율(pVM 메모리 점유, QA-06)

## 9. S. 확장성 관점 (추가)

### DP-S1. Workload 패키지·Manifest 스키마 설계

- **결정 질문**: Workload 패키지가 선언해야 하는 정보의 범위를 어디까지로 하는가 — HW IP 요구, 자원 요구(CPU/메모리), 통신 채널 요구, Secure OS 서비스 의존성. Framework가 manifest 해석만으로 신규 Workload를 수용하는 선언적 구조로 갈 것인가, 일부는 코드 확장(플러그인)을 허용할 것인가.
- **대안 방향**: 완전 선언적 manifest(코어 수정 0) / manifest + 제한적 플러그인 / Workload 유형별 템플릿 고정
- **품질속성**: 확장성(QA-03 "코어 수정 0 LoC, 패키징/탑재만으로 동작"의 실현 수단), 무결성(manifest 자체가 서명 대상, CS-06)

### DP-S2. 파이프라인 토폴로지 구성 방식

- **결정 질문**: Camera→AI 2-도메인 파이프라인을 고정 구조로 구현할 것인가, 도메인 수와 연결 관계를 선언적 그래프로 기술해 N개 도메인 파이프라인(예: 신규 센서 도메인, 다단 추론)을 구성 변경만으로 수용할 것인가. 토폴로지 해석과 채널 배선은 누가 수행하는가.
- **대안 방향**: 레퍼런스 2-도메인 고정 / 선언적 파이프라인 그래프(구성 파일 기반) / 고정 골격 + 확장 슬롯
- **품질속성**: 확장성(R-4 신규 보안 시나리오의 Framework 수정 없는 수용, FR-02 다중 pVM), 성능(범용 그래프 해석이 데이터 경로에 오버헤드를 더하지 않는가)

### DP-S3. HW IP 추상화 계층과 신규 가속기 수용 구조

- **결정 질문**: Camera/AI HW 중재를 디바이스 종류별 개별 구현으로 둘 것인가, 디바이스 클래스 추상화(capture류/inference류)를 정의해 신규 HW IP(차세대 NPU, ISP 변경, 제3의 가속기)를 중재 드라이버 플러그인 추가만으로 수용할 것인가. SoC 세대 간 S2MPU 차이는 어느 계층에서 흡수하는가.
- **대안 방향**: 디바이스별 개별 중재 구현 / 디바이스 클래스 추상화 + 플러그인 드라이버 / 클래스 추상화 + SoC HAL 분리(2계층)
- **품질속성**: 확장성(신규 HW IP 수용 비용, QA-03), 변경 용이성(SoC 라인업 확장 시 수정 범위), 성능(추상화 계층이 HW 접근 경로에 더하는 간접 비용)

## 10. 우선순위 소견

- **DP-C1, DP-C2, DP-C4**: 레퍼런스 시나리오 반복 구간(7~12단계)의 성능과 기밀성을 동시에 결정하므로 최우선 확정 대상.
- **DP-P1, DP-P2**: DP-C1의 하위 상세 결정이지만 QA-02(30fps, 100ms)·QA-04(5ms) 수치 달성 여부를 직접 좌우.
- **DP-S1, DP-E3**: QA-03(코어 수정 0 LoC)·QA-08(수정 파일 0개)의 전제 조건이 되는 인터페이스 계약 결정으로, 늦게 바꾸면 비용이 가장 크게 증가.
- **DP-E2**: 기능 경로는 아니지만 검증·규제 증빙 단계에서 소급 적용이 어려워 초기 확정 가치가 높음.
