# Q&A 기록: 신뢰 경계 확정은 왜 별도 작업인가

> 일자: 2026-06-11
> 관련 문서: `04_architectural_drivers.md` 5.2절 설계 착수 순서
> 성격: 설계 의사결정 질의/답변 기록

---

## 배경

`04_architectural_drivers.md`의 설계 착수 순서(5.2절)에서 "신뢰 경계 확정"을 순서 1의 별도 작업으로 두었다. 이에 대해 다음 질의가 제기되었다.

## 질문

**신뢰 경계 확정은 왜 별도 작업이 필요한가?**

신뢰 경계의 큰 틀은 pKVM 아키텍처가 이미 결정해 놓았다(신뢰 주체는 EL2 Hypervisor뿐, Host는 비신뢰). 이미 결정된 사항이라면 별도 작업 없이 각 핵심 기술(T-1~T-5) 설계 안에서 전제로 삼으면 되지 않는가?

## 답변

큰 틀은 정해져 있으나, **경계의 세부를 확정하는 결정들이 남아 있고 그 결정이 T-1~T-5 전체의 구조 후보를 가르기 때문에** 별도 작업이 필요하다. 논거는 네 가지다.

### 1. Framework 자신의 TCB 포함 여부 결정이 남아 있다

본 과제가 개발하는 Middleware/커널 드라이버는 Host에 위치한다. "Host 비신뢰" 전제를 끝까지 밀면 **Framework 자신도 TCB 밖**이어야 하고, 보안 보장은 Framework 코드의 올바름이 아니라 Stage-2/SMMU 메커니즘에만 의존해야 한다.

이 결정의 대가도 함께 수용해야 한다. Host(Framework 포함)가 침해되면 pVM의 기밀성/무결성은 지켜지지만 **가용성(스케줄링 거부, DoS)은 보호 범위 밖**이 된다. "무엇을 보호하고 무엇을 포기하는가"는 누군가 명시적으로 결정해야 하는 사안이다.

### 2. T-2(HW IP 중재자)의 배치 선택지를 직접 가른다

- 중재자를 **Host 커널**에 두면: 중재자는 비신뢰이므로 중재는 스케줄링 역할만 하고, 격리는 SMMU(EL2 통제)가 보장하는 구조여야 한다.
- 중재자에 **신뢰가 필요**하다면: 별도 서비스 pVM에 둬야 한다.
- **EL2 배치**: CONST-02(EL2 수정 불가)로 금지된다.

즉 신뢰 경계 세부를 먼저 정하지 않으면 T-2 후보 구조를 평가할 기준 자체가 없다. T-3(보안 채널의 중개자를 누가 검증하는가), T-4(Workload 패키지 서명을 누가 검증하는가)도 동일하다.

### 3. 병렬 설계 시 신뢰 가정 불일치가 가장 큰 실패 요인이다

T-1~T-5는 Security 파트(국내/SRCX)와 HV 파트가 나눠 병렬로 설계한다. 예컨대 T-2 팀은 "Host 드라이버는 신뢰할 수 있다"고 가정하고 T-3 팀은 "Host는 비신뢰"로 가정하면, 전체 격리 주장(QA-01)은 **가장 약한 가정 쪽으로 무너진다**. 그래서 공통 전제를 한 번, 가장 먼저, 문서로 확정해야 한다.

### 4. 산출물이 있는 작업이다

결과물은 **신뢰 모델 문서**다.

- TCB 구성 요소 목록
- 공격자 모델
- 보호 속성별 의존 메커니즘 매핑

이것이 이후 후보 구조 평가의 기준이 되고, QA-10(시험 용이성)의 검증 대상 정의와 CONST-04(규제 증빙)의 기준선이 된다.

## 요약

신뢰 경계 확정은 코드를 만드는 개발 작업이 아니라 **전역 설계 결정(decision) 작업**이다. 어느 단일 핵심 기술(T)에도 속하지 않으면서 모든 T의 선택지를 한정하고, 사후 변경 비용이 가장 크기 때문에 별도 항목으로 가장 앞에 둔다.
