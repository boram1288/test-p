# QS 별점 평가 기준

본 문서는 `09_qs.md`에 정의한 QS 7건의 별점(3개 척도) 평가 기준을 정의한다. `qs-writer` 스킬 규칙에 따라 평가 기준은 QS 본문과 분리하며, 별 3/2/1개를 가르는 경계값에는 객관적 근거를 병기한다.

## 1. 평가 원칙

- 별 3개: KPI 목표치 완전 충족
- 별 2개: 부분 충족 — 허용 구간을 정량적으로 정의
- 별 1개: 미충족 — 하한선을 정량적으로 정의
- 경계값의 근거는 `09_qs.md`의 "KPI 근거" 표(근거-n)를 공유하며, 본 문서에서 추가된 근거는 말미 "경계값 추가 근거" 절에 정리한다.

## 2. QS별 평가 기준

| QS ID | 별 3개 기준 | 별 2개 기준 | 별 1개 기준 | 경계값 근거 |
|-------|------------|------------|------------|-------------|
| QS-SEC-01 | 침투 시험에서 pVM 데이터 노출 0건, 그리고 정의된 공격 벡터 목록의 시험 커버리지 100% | 노출 0건이나 공격 벡터 시험 커버리지 100% 미달 (미시험 벡터 존재) | 노출 1건 이상 | 기밀성 침해는 발생 건수 1건이 곧 보증 실패 — 보안성 평가 방법론(CC AVA_VAN, SESIP)은 악용 가능한 취약점이 1건이라도 확인되면 해당 보증 수준을 부여하지 않음 [추가근거-A]. 따라서 노출 건수로는 부분 점수가 없고, 부분 충족은 "검증의 완전성(커버리지)" 축으로만 정의 |
| QS-SEC-02 | 변조 이미지 실행 차단율 100% 및 비인가 요청 거부율 100%, 모든 거부의 감사 기록 존재 | 차단/거부율 100%이나 감사 기록 누락 존재 | 차단 또는 거부 실패 1건 이상 | verified boot 모델은 검증 실패 시 실행 중단이 전부-아니면-무의미(fail-closed)인 구조 — 1건의 우회는 신뢰 체인 전체 붕괴 [근거-3]. 감사 기록은 ISO 25010 Accountability에 해당하는 부가 축이므로 부분 충족 축으로 사용 |
| QS-COMP-01 | 외부 SW(Camera/AI Framework, GP TEE API 자산) 원본 수정 0 LoC로 동작 | 원본 수정 0 LoC이나, Framework 측 어댑터/설정(매니페스트, 배선) 작업이 필요한 경우 | 외부 SW 원본 소스 수정이 1 LoC라도 필요 | GP TEE Client API 규격의 목적이 표준 준수 클라이언트/TA의 소스 수준 호환성이므로 "원본 무수정"이 규격 충족선 [근거-4]. 어댑터 계층 작업은 외부 SW 자산을 훼손하지 않으므로 허용 구간(별 2개), 원본 수정은 상호운용성 실패(별 1개) |
| QS-PERF-01 | 프레임당 E2E 처리 33.3ms 이내(30fps 유지) 그리고 프레임 전달 경로 memcpy 0회(zero-copy) | 프레임당 E2E 처리 66.7ms 이내(15fps 유지) 또는 복사 1회(single-copy) 경로 | 프레임당 66.7ms 초과 또는 복사 2회 이상(bounce buffer 이중 복사) 경로 | 프레임레이트 경계는 카메라 표준 동작점의 산술치: 30fps=33.3ms, 그 절반인 15fps=66.7ms까지는 영상 인식 기능이 저하된 채 유지, 그 미만은 실시간 파이프라인 실패 [근거-8]. 복사 횟수 경계는 실측 연구에서 오버헤드의 주원인으로 규명된 구조가 SWIOTLB 이중 복사(bounce buffering)라는 사실에 근거: zero-copy(설계 목표) / single-copy(중간) / double-copy(회피 대상 구조) [근거-5][근거-6] |
| QS-REL-01 | 장애 주입 전 케이스에서 전파 0건, 자동 복구 성공률 100% | 전파 0건이나 자동 복구 성공률 100% 미달(수동 개입 복구 존재) | Host 또는 타 pVM으로의 장애 전파 1건 이상 | 장애 격리는 pKVM 격리 구조가 보장해야 할 1차 속성으로, 전파 1건은 격리 실패와 동치이므로 부분 점수 없음 [근거-2]. 부분 충족은 복구의 자동화 수준(ISO 25023 회복성 측정 함수의 자동 복구 비율 축)으로 정의 [근거-1] |
| QS-FLEX-01 | 포팅/도메인 추가 시 코어 모듈 수정 0건 (변경이 사전 정의된 추상화 계층 파일 목록 내로 한정) | 코어 모듈 수정이 발생했으나 인터페이스(모듈 간 계약) 변경 없이 구현 내부에 한정 | 코어 모듈의 인터페이스 변경이 필요 | 경계 구분은 GKI의 실증 구조를 따름: 코어 불변+모듈 교체가 목표 상태(별 3), 인터페이스(KMI에 해당) 안정성이 유지되면 수정 비용이 국소화되므로 허용 구간(별 2), 인터페이스가 깨지면 의존 모듈 전체로 변경이 전파되어 분리 구조의 이득이 소멸(별 1) [근거-10][근거-11] |
| QS-MAINT-01 | 코어 커널 out-of-tree 수정 패치 0건 (커널측 기능 전부 로더블 모듈) | out-of-tree 패치가 존재하나 upstream 제출 중이거나 식별 태그 하에 관리됨 (ACK의 `ANDROID:` 태그 방식의 관리 프로세스) | 관리 프로세스 없는 사설 패치 존재 | 경계는 Android 커널 생태계의 실제 등급 구조를 따름: GKI 정책은 코어 커널 수정 0건이 목표 상태이고 [근거-10], ACK는 불가피한 out-of-tree 패치를 upstream-first 원칙과 식별 태그로 관리하는 중간 상태를 운용하며 [추가근거-B], 비관리 사설 패치는 GKI 이전 최대 50% out-of-tree 상태로의 회귀로 유지보수 비용 폭증이 실증됨 [근거-10] |

## 3. 경계값 추가 근거

| 번호 | 출처 |
|------|------|
| 추가근거-A | Common Criteria (ISO/IEC 15408) 취약성 평가 클래스 AVA_VAN 및 GlobalPlatform SESIP 방법론 — 평가 대상에서 공격 잠재력 내 악용 가능한 취약점이 확인되면 해당 보증 수준 불합격. 보안 침해 건수에 부분 점수를 두지 않는 경계 설정의 근거. 보조: Google pKVM SESIP Level 5 인증 사례(2025) |
| 추가근거-B | AOSP, "Develop kernel code for GKI" (https://source.android.com/docs/core/architecture/kernel/kernel-code) — 업스트림에 수용되지 못한 패치를 `ANDROID:` 태그로 식별하고 upstream-first로 최소화하는 ACK의 out-of-tree 패치 관리 정책 |
