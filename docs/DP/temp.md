# DP-C-03~DP-C-07 후보구조 요약

## DP-C-03: 기밀성, 확장성, 가용성을 위한 pVM 관리 골격 설계

| 후보구조 ID | 문제점 | 제안된 후보구조 | 짧은 설명 |
|---|---|---|---|
| A | pVM 생명주기 관리자, 제어 인터페이스, 자원 할당자 책임 경계를 정해야 한다. | 중앙 집중 단일 관리 데몬 | 하나의 시스템 데몬이 모든 pVM 생명주기, 자원, 상태 모니터링을 통합 관리한다. |
| B | pVM 생명주기 관리자, 제어 인터페이스, 자원 할당자 책임 경계를 정해야 한다. | pVM별 VMM 프로세스 + 얇은 공용 관리 서비스 | pVM마다 독립 VMM을 두고 공용 서비스는 레지스트리, 예산, 설치만 담당한다. |
| C | pVM 생명주기 관리자, 제어 인터페이스, 자원 할당자 책임 경계를 정해야 한다. | 커널 중심 관리 | 커널 드라이버가 생명주기 상태 머신과 자원 관리를 보유하고 사용자 공간은 얇게 둔다. |

## DP-C-04: 무결성 보장을 위한 보안 워크로드 실행 방식 설계

| 후보구조 ID | 문제점 | 제안된 후보구조 | 짧은 설명 |
|---|---|---|---|
| A | 워크로드 패키지, 부팅 규약, 통신 규약, 설치·로딩·교체 검증 지점을 표준화해야 한다. | 단일 표준 게스트 ABI | 모든 워크로드와 Secure OS가 동일한 부팅 규약, 가상 디바이스, 채널 ABI를 따른다. |
| B | 워크로드 패키지, 부팅 규약, 통신 규약, 설치·로딩·교체 검증 지점을 표준화해야 한다. | 프로파일 분리형 표준 | 패키지 포맷과 검증 절차는 단일화하고 실행 환경은 경량 워크로드와 Secure OS 프로파일로 나눈다. |
| C | 워크로드 패키지, 부팅 규약, 통신 규약, 설치·로딩·교체 검증 지점을 표준화해야 한다. | 최소 표준 | manifest, 서명, 암호화 구획만 표준화하고 부팅·통신 규약은 워크로드별 VMM 구성에 맡긴다. |

## DP-C-05: GP API Backward Compatibility를 위한 TrustZone 연동 구조 설계

| 후보구조 ID | 문제점 | 제안된 후보구조 | 짧은 설명 |
|---|---|---|---|
| A | pVM이 기존 Host→TEE 경로를 깨지 않으면서 TEE를 호출하고 호출 주체를 식별해야 한다. | Host 프록시 경유 | pVM 요청을 Host 프레임워크가 대신 받아 기존 TEE 클라이언트 드라이버로 SMC를 발행한다. |
| B | pVM이 기존 Host→TEE 경로를 깨지 않으면서 TEE를 호출하고 호출 주체를 식별해야 한다. | EL2 SMC 트랩·포워딩 | pVM의 SMC를 EL2가 트랩해 EL3·TEE로 전달하고 기존 Host 경로와 병존시킨다. |
| C | pVM이 기존 Host→TEE 경로를 깨지 않으면서 TEE를 호출하고 호출 주체를 식별해야 한다. | TEE 프록시 서비스 pVM | 전용 서비스 pVM이 다른 pVM의 TEE 호출을 모아 중계한다. |

## DP-C-06: 성능(통신 오버헤드)을 위한 zero-copy 보안 채널 구조 설계

| 후보구조 ID | 문제점 | 제안된 후보구조 | 짧은 설명 |
|---|---|---|---|
| A | Secure Camera와 Secure AI 사이의 영상 데이터를 Host 접근 없이 zero-copy로 전달해야 한다. | 정적 공유 영역 + 프레임 링버퍼 | 채널 수립 시 양단 pVM만 접근 가능한 공유 영역을 만들고 프레임 링버퍼로 전달한다. |
| B | Secure Camera와 Secure AI 사이의 영상 데이터를 Host 접근 없이 zero-copy로 전달해야 한다. | 프레임 단위 소유권 이전 | 프레임 버퍼마다 Camera pVM에서 AI pVM으로 소유권을 이전해 한 시점에 한 도메인만 접근한다. |
| C | Secure Camera와 Secure AI 사이의 영상 데이터를 Host 접근 없이 zero-copy로 전달해야 한다. | Host 중계 암호화 채널 | Host와 공유한 전달 경로 위에서 pVM 간 세션 키로 데이터를 암호화해 전달한다. |

## DP-C-07: 성능(실시간 처리) 및 자원 효율을 위한 HW IP 중재 구조 설계

| 후보구조 ID | 문제점 | 제안된 후보구조 | 짧은 설명 |
|---|---|---|---|
| A | HW IP 사용 주체 전환 시 SMMU 제어, 잔류 데이터 소거, 전환 오버헤드 기준을 정해야 한다. | Host 커널 중재 + EL2 SMMU 게이트 | Host 커널은 사용 순서만 정하고 SMMU 바인딩과 소거 보장은 EL2 게이트가 집행한다. |
| B | HW IP 사용 주체 전환 시 SMMU 제어, 잔류 데이터 소거, 전환 오버헤드 기준을 정해야 한다. | 중재 서비스 pVM | HW IP 드라이버와 중재자를 전용 서비스 pVM에 두고 Host와 pVM이 모두 이 pVM에 사용을 요청한다. |
| C | HW IP 사용 주체 전환 시 SMMU 제어, 잔류 데이터 소거, 전환 오버헤드 기준을 정해야 한다. | 정적 모드 전환 | 보안 파이프라인 구간과 Host 일반 사용 구간을 나누고 모드 전환 시에만 소유권을 바꾼다. |
