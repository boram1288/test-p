## 4. 보안 솔루션 공백
- 기존 Linux 가상화 솔루션
(대표 이미지를 포함한 테이블로 아래 정리)
  * Hypervisor(Xen): Runtime에 VM생성 불가능
  * KVM/QEMU: Host OS와 VM간 메모리 격리 안됨
  * Docker/LXC:  Host OS와 Linux Kernel 공유. Kernel은 표면이 너무 넓어 비신뢰영역.
  * AWS Nitro: 가상화 전용 하드웨어 사용. CPU/Memory를 고객에게 100%제공

- Host OS와 VM간 메모리 가능한 기존 격리 솔루션
  * Android AVF는 Android SW Stack에 종속적
   (AVF SW stack 다이어그램)

  * Protected KVM패치는 공개되어 있으나 이를 user단에서 사용할 framework의 부재
    (Framework이 비어있는 다이어그램)

핵심 문구: "Linux Custom SoC용 보안 Framework 부재"

## 5. 레퍼런스 시나리오 선정
- 설계 및 개발을 위한 검증(타겟) 시나리오 필요
  * 선정 기준
    보안 요구 대표: 고객사 보안 요구 사례
    HW IP 포괄성: 핵심 HW IP 사용
    기존 구조 한계 노출성: 다중 격리 도메인, 고성능 연산, HW IP 접근 동시 요구
    확장 구조 검증성: 멀티미디어와 AI, 도 도메인의 조합으로 다중 도메인 확장 구조를 검증하는 출발점
- Secure Vision AI 시나리오
  (App포함한 시나리오 전체 대표 다이어그램)

핵심 문구: "로봇용 Secure Vision AI 레퍼런스 시나리오를 사용한 설계 및 검증"

# 요구분석
(다른 ppt들과 양식 맞춰서 정리)

## 기능 요구사항
FR-01	pVM 생명주기 관리	선정	프레임워크 본체(미들웨어·드라이버)의 책임 분할과 컴포넌트 경계를 결정
FR-02	다중 pVM 동시 운용	선정	다중 격리 도메인 구조(도메인 단위 독립 관리)를 강제. TrustZone 이진 구조와의 차별화 지점
FR-03	HW IP 보안 공유(할당·중재)	선정	HW IP 중재 계층이라는 신규 구조 요소를 강제. 중재자의 위치(Host 커널/전용 pVM)가 전체 구조를 좌우
FR-04	격리 도메인 간 보안 통신	선정	도메인 간 데이터 경로(공유 메모리·RPC 채널) 구조를 결정
FR-06	보안 워크로드 동적 탑재	선정	플러그인형 패키징·로딩 구조와 패키지 검증 체계를 강제
FR-07	Secure OS pVM 실행 지원	선정	프레임워크-게스트 간 ABI(부팅 규약, 가상 디바이스, 통신 규약)의 초기 고정을 강제
FR-08	TrustZone 공존·연동	선정	pVM→TEE 요청 라우팅(SMC 포워딩)과 기존 경로 무회귀를 만족하는 세계 간 연동 계층을 결정
