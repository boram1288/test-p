## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-07)

- 감사 로그는 어떤 포맷과 저장 위치로 관리해야 하는가?
- 격리 상태를 외부에 증명하기 위해 attestation 구조를 둘 것인가?
- 증빙 범위는 내부 품질 검증용인가, 고객·규제기관 제출용인가?
- DP-11 로그, 침투 시험 결과, 설정 스냅샷을 어떤 단위의 감사 패키지로 묶을 것인가?
- 규제 증빙을 위해 설계 단계에서 어떤 문서와 시험 산출물을 고정해야 하는가?

### 1.2 선행 DP 결정이 주는 전제

- **DP-11**: 감사 입력은 "로그 pVM 해시 체인 + TEE 주기 앵커 + 누락 탐지 필드(G2)"다. 미앵커 구간의 보증 한계(G4)를 증빙 문서에 명시해야 한다. 증빙 형식·서명 키 수명주기·로그 검증 절차의 정의는 본 DP의 책임이다(G6).
- **DP-01**: AT 제외 항목(물리·부채널·부트체인)을 증빙 문서에 한계로 명시해야 한다(품질·검증 조직 협의 사항).
- **DP-06 G5**: attestation 등 신규 TEE 기능은 TA 자원 예산표 등록을 전제로 한다.

## 2. 후보 구조 (생성: Claude)

### 후보 A: 정적 증빙 패키지 (문서·시험·봉인 로그 중심)

릴리스 단위로 "감사 패키지"를 구성한다: (1) 설계 증빙 — 신뢰 모델(06 문서), DP 결정 기록, 보호 속성-메커니즘 매핑, (2) 시험 증빙 — QA-10 침해 모사 시험 결과(QS-01~03 차단율), 회귀 시험 이력, (3) 운영 증빙 — DP-11 봉인 로그와 그 검증 절차, 설정 스냅샷(manifest·권한 테이블 해시). 런타임 attestation은 두지 않는다.

- **장점**: 추가 런타임 컴포넌트가 없어 구현 비용이 가장 작고, GDPR류 규제 증빙의 통상 형태(기술·조직적 조치 문서 + 시험 결과)와 부합한다. TEE 부하 없음.
- **단점**: "지금 이 디바이스가 실제로 격리 상태로 동작 중"임을 외부에 실시간 증명할 수 없다. 증빙이 릴리스 시점 기준이라 배포 후 변조(승인되지 않은 워크로드 탑재)는 DP-11 로그로만 사후 추적된다.

### 후보 B: 정적 패키지 + TEE attestation (동적 증빙)

후보 A에 더해, TEE가 부팅 측정값(DP-09)·탑재 워크로드 목록·격리 설정 다이제스트를 서명한 attestation 보고서를 요청 시 발급한다. 고객·검증 조직은 디바이스 상태를 원격·현장에서 검증할 수 있다.

- **장점**: 배포 후 상태 증명이 가능해 제3자 감사·고객 수용 검사에 강하다. DP-09 측정값과 DP-10 아이덴티티 인프라를 재사용한다.
- **단점**: DP-09(측정) 완성이 선행되어야 하고, attestation TA·키 체계가 TEE에 추가된다(TA 예산·CONST-03 회귀 시험 부담). 검증자 측 도구(보고서 파서·기준값 관리)까지 범위가 늘어난다.

### 후보 C: 외부 감사 서버 연동 (지속 증빙)

후보 B에 더해, 디바이스가 주기적으로 attestation 보고서·로그 다이제스트를 제조사 감사 서버로 전송하고, 서버가 함대(fleet) 단위 규제 보고서를 생성한다.

- **장점**: 규제 대응을 제품군 단위로 자동화할 수 있고 사고 시 포렌식 자료가 풍부하다.
- **단점**: 본 과제(프레임워크) 범위를 넘는 서버·운영 인프라가 필요하다. 로봇의 오프라인 구간, 전송 데이터 자체의 프라이버시(감사 데이터가 개인정보가 되는 역설) 등 제품 운영 이슈가 크다. 전송 경로는 비신뢰 Host 경유라서 DP-11의 보조 복제 경로 결정과 같은 한계를 가진다.

### 공통 설계 요소

- **감사 패키지 단위**: 릴리스 버전 × 디바이스 모델. 내용 — 설계 증빙(고정 문서 세트), 시험 증빙(QA-10 산출물), 운영 증빙(DP-11 로그 + 검증 절차 + 보존 정책).
- **설계 단계 고정 문서**: `06_dp01_trust_model.md`(신뢰 모델·AT 제외 한계), DP 결정 기록(07~ 문서), 보호 속성-메커니즘 매핑, 침투 시험 계획.
- **증빙 한계 명시 의무**: AT-4·5·6 제외(06 문서), DP-11 미앵커 구간 보증 한계(G4), 가용성(b) 비보장.

### 3.2 종합 판정과 권고

- **A: 1차 기준선 적합** — 단 "현재 디바이스 상태" 증명은 불가.
- **B: 가장 강한 확장안** — 제3자 감사·고객 수용 검사 고려 시.
- **C: 기준선 아님** — 제품 운영/fleet 옵션.

**권고안**: A를 1차 릴리스 기준으로 채택하고, DP-09 측정·DP-10 아이덴티티·TEE TA 예산이 준비되면 B를 2단계로 추가하는 하이브리드. C는 TEE 서명 요약·봉인 로그의 보조 복제 경로로만 두고 규제 증빙의 원천으로 삼지 않음.

### 3.3 지적된 결함·누락

1. DP-11 G6가 위임한 증빙 형식·서명 키 수명주기·로그 검증 절차가 아직 미정의.
2. 감사 패키지에 DP-11 G2 필드, G4 미앵커 구간 한계, G5 생명주기 상태/heartbeat 교차 대조 결과를 포함해야 함.
3. 설정 스냅샷은 Host가 만든 파일이면 신뢰 증빙이 아님 — manifest hash·권한 테이블 hash·측정값의 TEE 서명 바인딩 필요.
4. attestation(2단계)에는 nonce, freshness, 기준 측정값 관리, 키 폐기·회전, 보고서 검증 도구가 필요.
5. AT-4·5·6 제외와 가용성(b) 비보장을 감사 문서의 명시적 한계로 고정해야 함.

## 5. 결정

**후보 A(정적 증빙 패키지)를 1차 릴리스 기준선으로 채택하고, 후보 B(TEE attestation)를 DP-09·DP-10 완성 후 2단계 확장으로 예약한다.** 후보 C는 TEE 서명 요약·봉인 로그의 보조 복제 경로로만 사용하며 규제 증빙의 원천으로 삼지 않는다(DP-11의 외부 전송 결정과 동일 한계).
