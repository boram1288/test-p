## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-06)

- pVM이 기존 TEE를 호출할 때 SMC 포워딩·라우팅 경로를 어디에 둘 것인가?
- 기존 Host→TEE SMC 경로를 변경 없이 유지하면서 pVM→TEE 경로를 어떻게 추가할 것인가?
- pVM이 TEE를 호출할 때 호출 주체의 신뢰성을 TEE가 어떻게 확인할 것인가?
- DP-08의 키 관리, DP-09의 이미지 검증, DP-10의 아이덴티티 검증에 TrustZone TEE를 사용할 것인가?
- TrustZone 연동 실패가 pVM, Host, 기존 TEE 기능에 어떤 영향을 주도록 제한할 것인가?

### 1.2 신뢰 모델 전제 (`06_dp01_trust_model.md` 6.5절)

- TEE는 1차 신뢰 앵커다. 신규 TA는 저빈도·소용량 작업으로 한정하고 기존 기능 회귀 시험을 전제한다.
- Host(프레임워크 포함)는 비신뢰이므로, Host를 경유하는 경로는 기밀성·무결성을 스스로 보장할 수 없다.
- EL2는 기 제공 hypercall·트랩 메커니즘 범위 안에서만 사용한다(CONST-02).

## 2. 후보 구조 (생성: Claude)

### 후보 A: Host 프록시 경유

pVM이 보안 채널로 Host 프레임워크에 TEE 호출을 요청하면, Host의 기존 TEE 클라이언트 드라이버가 대신 SMC를 발행하고 결과를 반환한다.

- **구성**: pVM → (보안 채널 RPC) → Host 프레임워크 TEE 프록시 → 기존 TEE 커널 드라이버 → SMC → EL3 → TEE
- **장점**: 기존 Host→TEE 경로를 그대로 재사용하므로 SW 추가가 가장 적고 CONST-03 위험이 없다. EL2와 무관하다.
- **단점**: 비신뢰 Host가 호출 파라미터·결과를 열람·변조·재전송할 수 있다. TEE는 호출 주체를 Host로만 인식하므로 pVM 단위 호출 주체 식별이 불가능하다. 종단 간 암호화를 덧붙여도 키 교환의 신뢰 앵커가 다시 필요해지는 순환이 생긴다.

### 후보 B: EL2 SMC 트랩·포워딩

pVM이 직접 SMC를 발행하면 pKVM(EL2)이 이를 트랩하여 EL3로 포워딩한다. pKVM은 게스트의 SMC를 트랩하는 메커니즘을 기본적으로 가지고 있으므로, 기 포팅된 pKVM이 제공하는 포워딩 정책 범위 안에서 사용한다.

- **구성**: pVM → SMC → EL2 트랩(기 제공 포워딩 정책) → EL3 SPD → TEE. 기존 Host→TEE 경로(EL1 SMC → EL3)는 변경 없이 병존.
- **장점**: Host를 완전히 우회하므로 호출 내용의 기밀성·무결성이 신뢰 모델과 정합한다. EL2가 트랩 시점에 VMID를 식별할 수 있어 호출 주체 정보(pVM 식별자)를 EL3·TEE까지 전달할 근거가 생긴다(DP-09 측정값, DP-10 아이덴티티와 연계 가능).
- **단점**: 기 포팅 pKVM의 SMC 포워딩 지원 범위·정책에 의존한다(HV 파트 확인 필요, CONST-02상 수정 불가). EL3 SPD와 TEE가 새 호출 출처(pVM)를 구분·수용하도록 TEE 측 수용 작업이 필요할 수 있어 CONST-03 회귀 시험 범위가 늘어난다.

### 후보 C: TEE 프록시 서비스 pVM

전용 서비스 pVM이 다른 pVM들의 TEE 호출을 모아 중계한다.

- **구성**: pVM → (pVM 간 보안 채널) → TEE 프록시 pVM → SMC(B 방식) → TEE
- **장점**: 호출 정책(쿼터, 필터링)을 Linux 환경에서 유연하게 구현할 수 있고, TEE에 노출되는 호출 출처를 프록시 하나로 단일화할 수 있다.
- **단점**: 프록시 pVM이 모든 TEE 호출 내용을 보게 되어 TCB 편입이 불가피하다(DP-01의 조건부 2차 앵커 규칙 적용 대상). 부트스트랩(프록시 자신의 검증), 추가 전환 지연, DP-04 채널 완성 이전에는 동작 불가라는 순서 문제가 있다.

### 평가 기준

| 기준 | 설명 |
|------|------|
| E1. 신뢰 모델 정합 | Host 비신뢰 전제에서 호출 기밀성·무결성이 성립하는가 |
| E2. CONST-03 무회귀 | 기존 Host→TEE 경로·기능에 변경이 없는가, TEE 측 수정량은 |
| E3. CONST-02 적합 | EL2 수정 없이 기 제공 메커니즘 범위에서 실현 가능한가 |
| E4. 호출 주체 식별 | TEE가 호출한 pVM을 식별·인증할 수 있는가 (DP-09·10 연계) |
| E5. 성능 | SMC 왕복 지연, 저빈도 키 관리·인증 용도 기준 |
| E6. 실패 격리 | 연동 실패가 pVM·Host·기존 TEE에 미치는 영향 범위 |
