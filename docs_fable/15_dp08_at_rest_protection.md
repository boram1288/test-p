## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-08)

- AI 모델 가중치, 워크로드 패키지, 설정 파일 중 어떤 자산을 저장 시 암호화할 것인가?
- 암호화 키는 TrustZone TEE, pKVM EL2, 별도 키 관리 서비스 중 누가 생성·보관·사용할 것인가?
- 암호화 범위는 패키지 단위, 파일 단위, 파일시스템 전체 중 무엇으로 할 것인가?
- 복호화는 어느 시점에 수행하고, 복호화 키는 어떤 경로로 pVM에 전달할 것인가?
- 키 회전, 폐기, 워크로드 삭제 시 저장 자산을 어떻게 무효화할 것인가?

### 1.2 선행 DP 결정이 주는 전제

- **DP-12**: 키 방출은 TEE가 manifest 권한·측정값을 대조한 후 수행하는 "최후의 공통 게이트"다 — 다른 게이트가 뚫려도 자산 복호화는 불가해야 한다. 본 DP는 이 게이트의 구체 구조를 설계한다.
- **DP-06**: TEE가 1차 신뢰 앵커이며, pVM→TEE 호출은 EL2 SMC 트랩·포워딩 경로다(Host 프록시 금지). 키 관리 TA는 저빈도·소용량 규칙과 G5 예산표 등록을 따른다. EL2 확장(키 관리 기능 추가)은 CONST-02로 원천 배제된다.
- **DP-04 G3**: 복호화 키의 pVM 전달은 Secure World 전용 버퍼 모델(Host unmapped)을 사용한다 — 일반 채널 경유 금지.
- **DP-09 (미결, 후순위)**: 키 방출 조건인 "부팅 검증 통과·측정값 대조"의 구체 형태는 DP-09에서 확정된다. DP-10과 동일하게 단계적 확정이 필요할 수 있다.

## 2. 후보 구조 (생성: Claude)

### 후보 A: 패키지 단위 암호화 + TEE 봉인 키 + pVM 내 복호화

보호 자산(AI 모델 가중치, 워크로드 패키지의 보호 구획)을 패키지 단위로 암호화하여 Host 파일시스템에 저장한다. 자산 암호화 키(per-workload)는 TEE가 봉인(seal) 보관하고, pVM이 부팅 검증을 통과한 뒤 EL2 SMC 경로로 키 방출을 요청하면 TEE가 manifest 권한·측정값을 대조하여 해당 pVM의 Secure World 전용 버퍼(DP-04 G3 서브셋)로만 방출한다. 복호화는 pVM 내부에서 수행되어 평문은 pVM 격리 메모리에만 존재한다.

- **저장·로딩 흐름**: 배포 시 패키지 보호 구획 암호화 → Host는 암호문 파일의 운반·저장만 담당 → pVM 부팅·검증 통과 → TEE 키 방출(게이트) → pVM 내 복호화·사용
- **장점**: Host는 전 구간에서 암호문만 취급한다(QA-01 at-rest 정합). DP-12의 키 방출 게이트 결정과 정확히 일치하며, 기존 TEE 키 관리 기능(FR-08) 재사용이라 신규 TCB가 없다. 키 방출은 pVM 부팅 시 1회로 저빈도 규칙에 부합한다.
- **단점**: 복호화가 pVM 부팅·로딩 시간에 추가된다(대형 모델 가중치에서 체감 가능). 키 회전 시 자산 재암호화 절차가 필요하다. DP-09 미결로 "측정값 대조"의 구체 형태가 아직 열려 있다.

### 후보 B: 키 관리 서비스 pVM (key broker)

전용 키 관리 서비스 pVM이 자산 키의 보관·방출을 담당하고, TEE는 이 서비스 pVM의 마스터 키 봉인만 담당한다.

- **장점**: 키 정책(회전·폐기·버전 관리)을 Linux 환경에서 유연하게 구현할 수 있고, TEE TA 추가가 최소화된다. 다수 워크로드의 키를 대량 관리할 때 TEE 자원 제약을 우회한다.
- **단점**: 서비스 pVM의 TCB 편입(DP-01 조건부 규칙 + DP-11의 상한 정책 — 로그 pVM에 이어 두 번째 편입)이 필요하고, 부트스트랩(이 pVM 자신의 검증·자기 키 수급)이 DP-09·TEE에 결국 의존하므로 계층만 늘어난다. 키 방출 게이트가 TEE에서 서비스 pVM으로 이동하면 DP-12의 "최후의 공통 게이트 = TEE" 결정과 어긋난다.

### 후보 C: Host 파일시스템 전체 암호화 (dm-crypt 류, 비교 기준선)

Host가 관리하는 파일시스템 암호화(dm-crypt/fscrypt)로 저장 자산을 보호한다.

- **장점**: 표준 기능 재사용으로 구현 비용이 거의 없고 모든 파일에 일괄 적용된다.
- **단점**: 마운트 후에는 Host 커널이 평문을 본다 — Host 커널 권한 공격자(AT-2)에게 무력하여 QA-01의 자극 자체를 방어하지 못한다. 오프라인 도난(AT-4 인접) 방어에만 유효하다. 단독 후보로 성립 불가, 심층 방어의 보조 계층으로만 의미.

### 공통 설계 요소

- **암호화 대상 분류**: (1) 기밀 — AI 모델 가중치, 워크로드 보호 구획, 봉인 로그 본문: 암호화 필수. (2) 무결성만 — manifest, pVM 이미지(코드): 서명·측정(DP-09·DP-12)으로 보호하되 암호화는 선택(코드 기밀성 요구 시). (3) 공개 — 설정 메타데이터: 서명만.
- **키 수명주기**: per-workload 키, 워크로드 삭제 시 TEE 봉인 키 폐기로 저장 자산 즉시 무효화(crypto-erase). 키 회전은 패키지 갱신 주기와 동기화.
- **DP-09 연계**: 키 방출 조건의 측정값 대조 형태는 DP-09 확정 후 역반영(DP-10 G1과 동일 패턴 예상).

### 평가 기준

| 기준 | 설명 |
|------|------|
| E1. at-rest 기밀성 | Host 커널 권한 공격자(AT-2)가 저장 자산 평문·키에 접근 불가한가 (QA-01) |
| E2. 로딩 경로 평문 비노출 | 복호화 시점·키 전달 경로에서 평문·키가 Host에 노출되지 않는가 |
| E3. 키 수명주기 완결성 | 회전·폐기·워크로드 삭제 시 무효화(crypto-erase)가 성립하는가 |
| E4. 선행 결정 정합 | DP-12 키 방출 게이트, DP-04 G3 버퍼 모델, DP-06 경로·TA 규칙과의 정합 |
| E5. 성능·TEE 부하 | 로딩 시간 영향(대형 가중치), TA 예산(저빈도·소용량) 적합성 |
| E6. 확장성 | 신규 워크로드 추가 시 키 체계 수정 없이 수용 (QA-07, DP-05 패키지 연계) |
