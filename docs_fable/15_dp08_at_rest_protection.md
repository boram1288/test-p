## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-08)

- AI 모델 가중치, 워크로드 패키지, 설정 파일 중 어떤 자산을 저장 시 암호화할 것인가?
- 암호화 키는 TrustZone TEE, pKVM EL2, 별도 키 관리 서비스 중 누가 생성·보관·사용할 것인가?
- 암호화 범위는 패키지 단위, 파일 단위, 파일시스템 전체 중 무엇으로 할 것인가?
- 복호화는 어느 시점에 수행하고, 복호화 키는 어떤 경로로 pVM에 전달할 것인가?
- 키 회전, 폐기, 워크로드 삭제 시 저장 자산을 어떻게 무효화할 것인가?

### 1.2 선행 DP 결정이 주는 전제

- **DP-12**: 키 방출은 TEE가 manifest 권한·측정값을 대조한 후 수행하는 "최후의 공통 게이트"다 — 다른 게이트가 뚫려도 자산 복호화는 불가해야 한다. 본 DP는 이 게이트의 구체 구조를 설계한다.
- **DP-06**: TEE가 1차 신뢰 앵커이며, pVM→TEE 호출은 EL2 SMC 트랩·포워딩 경로다(Host 프록시 금지). 키 관리 TA는 저빈도·소용량 규칙과 G5 예산표 등록을 따른다. EL2 확장(키 관리 기능 추가)은 CONST-02로 원천 배제된다.
- **DP-04 G3**: 복호화 키의 pVM 전달은 Secure World 전용 버퍼 모델(Host unmapped)을 사용한다 — 일반 채널 경유 금지.
- **DP-09 (미결, 후순위)**: 키 방출 조건인 "부팅 검증 통과·측정값 대조"의 구체 형태는 DP-09에서 확정된다. DP-10과 동일하게 단계적 확정이 필요할 수 있다.

## 2. 후보 구조 (생성: Claude)

### 후보 A: 패키지 단위 암호화 + TEE 봉인 키 + pVM 내 복호화

보호 자산(AI 모델 가중치, 워크로드 패키지의 보호 구획)을 패키지 단위로 암호화하여 Host 파일시스템에 저장한다. 자산 암호화 키(per-workload)는 TEE가 봉인(seal) 보관하고, pVM이 부팅 검증을 통과한 뒤 EL2 SMC 경로로 키 방출을 요청하면 TEE가 manifest 권한·측정값을 대조하여 해당 pVM의 Secure World 전용 버퍼(DP-04 G3 서브셋)로만 방출한다. 복호화는 pVM 내부에서 수행되어 평문은 pVM 격리 메모리에만 존재한다.

- **저장·로딩 흐름**: 배포 시 패키지 보호 구획 암호화 → Host는 암호문 파일의 운반·저장만 담당 → pVM 부팅·검증 통과 → TEE 키 방출(게이트) → pVM 내 복호화·사용
- **장점**: Host는 전 구간에서 암호문만 취급한다(QA-01 at-rest 정합). DP-12의 키 방출 게이트 결정과 정확히 일치하며, 기존 TEE 키 관리 기능(FR-08) 재사용이라 신규 TCB가 없다. 키 방출은 pVM 부팅 시 1회로 저빈도 규칙에 부합한다.
- **단점**: 복호화가 pVM 부팅·로딩 시간에 추가된다(대형 모델 가중치에서 체감 가능). 키 회전 시 자산 재암호화 절차가 필요하다. DP-09 미결로 "측정값 대조"의 구체 형태가 아직 열려 있다.

### 후보 B: 키 관리 서비스 pVM (key broker)

전용 키 관리 서비스 pVM이 자산 키의 보관·방출을 담당하고, TEE는 이 서비스 pVM의 마스터 키 봉인만 담당한다.

- **장점**: 키 정책(회전·폐기·버전 관리)을 Linux 환경에서 유연하게 구현할 수 있고, TEE TA 추가가 최소화된다. 다수 워크로드의 키를 대량 관리할 때 TEE 자원 제약을 우회한다.
- **단점**: 서비스 pVM의 TCB 편입(DP-01 조건부 규칙 + DP-11의 상한 정책 — 로그 pVM에 이어 두 번째 편입)이 필요하고, 부트스트랩(이 pVM 자신의 검증·자기 키 수급)이 DP-09·TEE에 결국 의존하므로 계층만 늘어난다. 키 방출 게이트가 TEE에서 서비스 pVM으로 이동하면 DP-12의 "최후의 공통 게이트 = TEE" 결정과 어긋난다.

### 후보 C: Host 파일시스템 전체 암호화 (dm-crypt 류, 비교 기준선)

Host가 관리하는 파일시스템 암호화(dm-crypt/fscrypt)로 저장 자산을 보호한다.

- **장점**: 표준 기능 재사용으로 구현 비용이 거의 없고 모든 파일에 일괄 적용된다.
- **단점**: 마운트 후에는 Host 커널이 평문을 본다 — Host 커널 권한 공격자(AT-2)에게 무력하여 QA-01의 자극 자체를 방어하지 못한다. 오프라인 도난(AT-4 인접) 방어에만 유효하다. 단독 후보로 성립 불가, 심층 방어의 보조 계층으로만 의미.

### 공통 설계 요소

- **암호화 대상 분류**: (1) 기밀 — AI 모델 가중치, 워크로드 보호 구획, 봉인 로그 본문: 암호화 필수. (2) 무결성만 — manifest, pVM 이미지(코드): 서명·측정(DP-09·DP-12)으로 보호하되 암호화는 선택(코드 기밀성 요구 시). (3) 공개 — 설정 메타데이터: 서명만.
- **키 수명주기**: per-workload 키, 워크로드 삭제 시 TEE 봉인 키 폐기로 저장 자산 즉시 무효화(crypto-erase). 키 회전은 패키지 갱신 주기와 동기화.
- **DP-09 연계**: 키 방출 조건의 측정값 대조 형태는 DP-09 확정 후 역반영(DP-10 G1과 동일 패턴 예상).

**후보 B (키 관리 서비스 pVM)**: E4 **하** — 키 방출 최종 판단이 broker로 이동하면 DP-12 "TEE 최후 공통 게이트"와 충돌. E1 중(대량 키 보유 신규 핵심 TCB, 침해 시 영향 반경 큼), E3 상(조건부, 수명주기 관리 유연), E6 상. 대형 모델 복호화 문제는 그대로 남음.

**후보 C (Host 파일시스템 전체 암호화)**: E1·E2·E3·E4 모두 **하** — 마운트 후 Host 커널이 평문을 보므로 AT-2 방어 불가. E5만 상. 단독 부적합.

### 3.2 종합 판정

- **A: 조건부 채택 대상.** 보안 구조는 가장 올바르나, 현재 문안은 대형 모델 로딩 비용, crypto-erase 의미, DP-09 미결 범위를 과소기술.
- **B: 기준안 부적합, 보조 확장 후보.** key lifecycle 관리 서비스로는 가능하나 최종 키 방출 게이트를 대체하면 DP-12와 충돌.
- **C: 단독 부적합.** 오프라인 저장장치 도난 방어용 심층 방어 계층으로만 의미. QA-01 대응책으로 계산 금지.

### 3.3 권고안

후보 A를 기준안으로 채택하되 **"패키지 단위 암호화 + 내부 chunk/tensor 단위 AEAD 복호화"**로 보강:

- TEE는 대용량 복호화를 하지 않고 per-workload/per-asset **DEK unwrap·방출만** 수행한다.
- pVM은 암호문 전체를 메모리에 올린 뒤 평문을 다시 만드는 방식이 아니라, **1~수 MB 단위 chunk를 인증·복호화해 최종 모델 arena에 직접 적재**한다. naive 복호화는 `암호문 N + 평문 N + 임시 버퍼`로 피크 메모리가 거의 2배가 되고 부팅 시간이 수 초 단위로 증가한다. 목표: "피크 메모리 = 모델 크기 + 작은 window + metadata".
- 추가 대안: lazy/on-demand tensor decrypt, prefetch, pVM 내 하드웨어 crypto 사용, 최소 부팅과 모델 로딩을 분리하는 2단계 startup.
- **crypto-erase 정의 교정**: "배포된 암호문을 삭제한다"가 아니라 "해당 암호문을 해독할 키 접근을 영구 차단한다"로 정의해야 한다. Host·백업에 남은 암호문 사본은 계속 존재하며, old key, rollback된 sealed key, 다른 디바이스의 동일 fleet key가 살아 있으면 무효화가 깨진다. key version/epoch, anti-rollback secure storage, old key 폐기, 재암호화 절차, 백업·스냅샷 정책이 결정문에 필요하다.

### 3.4 결함·누락·불일치 지적

1. 후보 A의 "대형 모델 가중치" 영향이 단점 한 줄 처리 — 부팅 시간, peak memory, chunked decrypt 요구, 성능 측정 게이트 명시 필요.
2. crypto-erase 과대 주장 — 배포된 암호문 사본, 백업, old key, rollback copy, 이미 실행 중 pVM에 방출된 키/평문은 한계로 별도 기술.
3. DP-09 미결 상태에서 확정 가능한 범위는 TEE 최종 게이트, Host 비노출 경로, fail-closed 원칙, VMID 단독 인증 금지까지 — "측정값 대조" 구체 조건은 확정 불가.
4. DP-06 G1/G3, DP-04 G3가 조건부인데 후보 A는 확정 전제처럼 기술 — DP-08 결정 상태도 조건부여야 함.
5. AEAD 무결성, manifest hash와 ciphertext binding, key id/version, rollback 방지, rotation 중 dual-key 기간 누락.
6. 후보 B 도입 시에도 broker는 메타데이터·운영 보조 역할에 한정하고 TEE가 최종 unwrap/release를 수행해야 함.
7. 후보 C는 "보조 방어 계층"으로 명확히 격하 필요.
8. DP-06 G5 TEE TA 예산표에 DP-08 key TA의 호출 빈도, 세션 수, secure storage 사용량, boot storm rate limit 등록 필요.

## 4. 검수 (검수: Claude)

> 수행: critic 에이전트 (별도 컨텍스트, 전제 문서 5종 교차 대조)

### 4.1 판정

**조건부 동의.** Codex의 "후보 A + chunked AEAD 보강" 권고안은 방향이 타당하고 선행 DP(DP-12 키 방출 게이트, DP-04 G3 버퍼, DP-06 TA 규칙)와 정합한다. 단, Codex가 식별한 결함을 "게이트로 승격할 것"과 "한계 명시로 충분한 것"으로 구분하지 않고 결정문을 닫으면 DP-10이 회피한 함정(DP-09 미결 침범)에 다시 빠진다.

### 4.2 주요 검수 의견

1. **Codex 등급·근거 타당** — 과대/과소 평가 없음. 후보 B E4 "하"(DP-12 충돌), 후보 C 전 기준 "하"(AT-2 무력, AT-4는 공격자 집합 밖) 모두 전제 문서와 일치 확인.
2. **(CRITICAL) DP-10 이중 의존 누락** — Codex는 DP-09 의존만 지적했으나, DP-08의 키 방출 대상 판정 근거는 "DP-09 측정값 → DP-10 measured identity → DP-08 키 방출"로 **DP-10을 경유**한다. DP-08 골격은 DP-09(측정값 형태)와 DP-10 G1(자격 발급 바인딩) **양쪽** 역반영에 걸린다. 키 방출 대상 오판은 보호 자산 전체 노출(QA-01 핵심)로 직결되므로 게이트 승격 필수.
3. **(MAJOR) 실행 중 인스턴스 무효화 누락** — crypto-erase 시 이미 키/평문을 메모리에 적재한 실행 중 pVM은 키 폐기로 막을 수 없다. 단순 한계가 아니라 DP-02 자원 회수 + DP-04 G4 채널 강제 해제 + DP-10 G3 시간 경계와 연동되는 구조적 철회 게이트다.
4. **(MAJOR) DP-05 패키지 포맷 경계 필요** — "패키지 단위 암호화 + chunked AEAD"는 패키지 포맷(청크 레이아웃, AEAD 태그 배치, manifest↔ciphertext 바인딩 위치)을 직접 규정하게 된다. DP-08은 암호화 단위·키 방출 게이트·복호화 위치까지만 확정하고, 포맷 구체 정의는 DP-05로 핸드오프해야 설계 자유도 침범을 피한다.
5. **E5 책임 주체 교정** — 메모리 2배 문제는 TEE 부하가 아니라 pVM 내부 메모리 압박이다. 성능 게이트의 책임 주체는 pVM 내부 로딩 설계(DP-05 연계)임을 명확히 한다.
6. **결함 분류** — 게이트 승격: 결함 1(성능)·3(DP-09 범위)·4(상태 일치)·5(키 수명주기)·8(예산표). 한계 명시: 결함 2 중 분산 사본·이미 방출된 키. 결정문 본문 처리: 결함 6(B 격하)·7(C 격하).

### 4.3 미해결 질문 (후속 확정 대상)

- per-workload 키가 **per-device인지 per-fleet인지** 미명시 — 한 디바이스 폐기가 fleet 전체 crypto-erase 보증에 미치는 영향. DP-05 또는 DP-08 상세 설계에서 확정.
- DP-09 measured boot 체인의 종료 지점(워크로드 코드까지 vs pVM 이미지까지)에 따라 키 방출이 "변조된 워크로드"를 걸러내는 범위가 달라진다 — G2 역반영 시 재확인.

## 5. 결정

### 5.1 채택 구조

**후보 A를 골격 채택한다 — "패키지 단위 암호화 + TEE 봉인 키(DEK unwrap·방출 전용) + pVM 내 chunked AEAD 복호화".**

- 보호 자산(AI 모델 가중치, 워크로드 보호 구획)은 배포 시 패키지 단위로 암호화하고, Host는 전 구간에서 **암호문 운반·저장만** 담당한다.
- 키 방출은 **TEE가 최후의 공통 게이트**(DP-12 결정 상속)로서 per-workload DEK unwrap·방출만 수행한다(대용량 복호화는 TEE가 하지 않는다). 방출 경로는 EL2 SMC(DP-06) + Secure World 전용 버퍼(DP-04 G3 소비자 참조)이며 fail-closed, **VMID 단독 인증 금지**.
- 복호화는 pVM 내부에서 **1~수 MB 단위 chunk를 인증·복호화해 모델 arena에 직접 적재**한다(naive 전체 복호화 금지 — 피크 메모리 목표: 모델 크기 + 작은 window + metadata).
- **crypto-erase 정의**: "배포된 암호문 삭제"가 아니라 "해당 암호문을 해독할 키 접근의 영구 차단"이다.
- **후보 B**는 키 lifecycle 메타데이터·운영 보조 역할로만 보존 가능하다(최종 unwrap/release는 항상 TEE — DP-12 충돌 방지). **후보 C**는 오프라인 도난 방어용 보조 심층 방어 계층으로 격하한다(QA-01 대응책으로 계상 금지).

확정 범위는 **골격까지**다: TEE 최종 키 방출 게이트, Host 비노출 경로, fail-closed, VMID 단독 인증 금지, pVM 내 chunked 복호화. "측정값 대조"의 구체 형태와 키 방출 대상 인증 형태는 G1·G2 역반영 대상이다.

### 5.3 한계 명시 (게이트 아님)

- crypto-erase는 분산 암호문 사본·백업·다른 디바이스의 fleet key를 물리적으로 제거하지 못한다 — 잔여 위험은 AT-4(공격자 모델 제외) 영역으로 귀속, 제품 차원 대응(케이스 봉인 등) 권고.
- 본 체계가 막는 것은 "Host의 저장 자산 평문 접근"이며, 침해된 pVM이 자기 권한으로 받은 평문 모델을 오용하는 것은 막지 못한다(DP-10 G4와 동일 층위 — 탐지·증빙은 DP-07·DP-11).

### 5.4 후속 DP에 주는 전제

- **DP-05**: 패키지 컨테이너 포맷에 암호문 청크 레이아웃 + AEAD 태그 배치 + manifest hash↔ciphertext 바인딩 위치 포함(포맷 구체는 DP-05 책임). 로딩 절차는 "TEE 키 방출 → pVM 내 chunked 복호화 → arena 직접 적재"의 2단계 startup 수용. 삭제 절차 = 키 폐기 + 실행 중 인스턴스 강제 종료(G6). 키 회전과 패키지 갱신 주기 동기화. **per-device vs per-fleet 키 단위 확정**(4.3절 질문).
- **DP-09**: 측정값 대조 구체 형태는 DP-09가 확정하고 DP-08에 역반영(G1). measured boot 체인 종료 지점(워크로드 코드까지 vs pVM 이미지까지)이 키 방출의 변조 워크로드 차단 범위를 결정함을 입력으로 전달.
- **DP-06 G5**: key TA 예산 등록(G7).

### 5.5 상태

**골격 채택 (조건부)** — G1·G2(DP-09·DP-10 역반영)와 G3(선행 조건부 게이트) 충족 시 확정으로 전환한다.
