## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-02)

- 생명주기 관리자, pVM 제어 인터페이스, 자원 할당자의 책임 경계를 어떻게 나눌 것인가?
- 도메인별 메모리·CPU·디바이스 자원을 어떤 단위로 분리·회수할 것인가?
- KVM ioctl, VFIO, dma-buf 등 어떤 Linux 네이티브 인터페이스를 기본 제어면으로 사용할 것인가?
- HV 파트와 확정해야 할 최소 pKVM hypercall 계약은 무엇인가?
- pVM 생성, 메모리 할당, hypercall 최소 계약 중 어떤 항목이 정해지면 DP-03 PoC를 병행 착수할 수 있는가?

### 1.2 선행 DP 결정이 주는 전제

- **DP-01**: 생명주기 관리자는 비신뢰 영역(Host)에 있다. 관리자는 "가용성·관리 기능 제공자"일 뿐 보안 보장 주체가 아니며, 보안이 걸리는 동작은 hypercall 계약을 경계로 EL2 메커니즘에 위임된다.
- **DP-12**: 관리자는 권한의 운반자·요청자다. pVM 생성·메모리 공유 요청 시 manifest를 운반하지만 권한 판단은 TCB 게이트가 수행한다(fail-closed).
- **DP-11 G5**: 가용성(b) 거부 시도(생성 거부, 기아, 강제 종료)의 감지를 위해 생명주기 상태 보고가 로그 pVM heartbeat와 교차 대조 가능해야 한다 — 관리자의 상태 모니터링 출력이 감사 입력의 일부가 된다.

## 2. 후보 구조 (생성: Claude)

### 후보 A: 중앙 집중 단일 관리 데몬

하나의 시스템 데몬이 모든 pVM의 생명주기·자원·상태 모니터링을 통합 관리하고, 자체 커널 모듈이 KVM 제어와 메모리 예약을 담당한다.

- **구성**: 관리 데몬(단일 프로세스, 전체 pVM 상태 머신 보유) + 통합 커널 모듈 → KVM ioctl·hypercall
- **장점**: 자원 예산(메모리·CPU)의 전역 관리가 자연스럽다. 구성 요소가 적어 초기 개발이 빠르다.
- **단점**: 데몬 크래시 시 모든 pVM의 관리면이 동시에 상실된다(QA-09 관리면 단일 장애점). 도메인별 상태가 한 프로세스에 결합되어 워크로드 추가·교체(QA-07) 시 데몬 영향 범위가 넓다. pVM별 권한 분리(최소 권한)가 어렵다.

### 후보 B: pVM별 VMM 프로세스 + 얇은 공용 관리 서비스

pVM마다 독립 VMM 프로세스(KVM ioctl로 해당 pVM만 소유)를 두고, 공용 관리 서비스는 레지스트리(어떤 pVM이 떠 있는가), 자원 예산 조정, 패키지 설치 관리만 담당한다. AVF의 crosvm 인스턴스 모델을 Android 스택 없이 Linux 네이티브(데몬 + 프로세스 모델)로 재구성한 형태다.

- **구성**: pVM당 VMM 프로세스(vCPU 스레드, 가상 디바이스, 해당 pVM 메모리 소유) + 공용 관리 서비스(레지스트리·예산·설치) + 공용 커널 드라이버(예약 메모리 풀, hypercall 래핑, dma-buf 연계)
- **장점**: VMM 크래시가 해당 pVM에만 한정된다(QA-09·FR-09와 정합 — 도메인 단위 장애 격리). pVM별 자원이 프로세스 단위로 분리되어 회수가 명확하다(프로세스 종료 = 자원 회수 트리거). cgroup·seccomp 등 Linux 표준 격리로 VMM 자체를 최소 권한화할 수 있다. 도메인 추가가 프로세스 추가라서 확장(QA-07)이 구조적으로 열려 있다.
- **단점**: 프로세스 간 조정(예산 합의, 시작 순서)이 필요해 공용 서비스와 VMM 간 프로토콜이 생긴다. pVM 수만큼 프로세스 상주 비용(소규모)이 든다.

### 후보 C: 커널 중심 관리 (커널 드라이버가 상태 머신 보유)

생명주기 상태 머신과 자원 관리를 커널 드라이버에 두고, 사용자 공간은 얇은 라이브러리·CLI만 제공한다.

- **구성**: 통합 커널 드라이버(생명주기 상태 머신, 메모리 풀, KVM 내부 API 직접 사용) + 사용자 공간 라이브러리
- **장점**: 호출 경로가 짧고 사용자 공간 의존이 적다. 부팅 초기 등 데몬이 없는 시점에도 동작 가능하다.
- **단점**: 커널 내 복잡도가 커져 결함이 곧 Host 커널 패닉으로 이어진다(로봇 기본 동작 영향, QA-09 역행). 가상 디바이스·패키지 처리 같은 정책성 로직이 커널에 들어가 유지보수·확장이 어렵다(QA-07). KVM의 사용자 공간 VMM 모델(ioctl)과 어긋나 Linux 네이티브 정합성이 떨어진다.

| # | 계약 항목 | 용도 |

### 3.2 종합 판정과 권고

- **A**: 초기 구현은 빠르나 최종 구조로는 단일 장애점·권한 분리 약점이 큼.
- **B: 채택 적합** — DP-01·DP-12 전제와 가장 정합, Linux 네이티브 구조가 자연스러움.
- **C: 부적합** — 얇은 예약 메모리 풀·hypercall 래퍼 수준만 커널에 두고, 상태 머신을 커널에 두면 안 됨.

**권고안**: 후보 B를 기준안으로 채택. DP-03 PoC를 위해 H1(생성)·H2(메모리)·H6(SMMU)을 1차로 고정하는 단계적 접근. 공용 관리 서비스는 레지스트리·예산·설치만 담당하고 권한 판단은 TCB 게이트 fail-closed로.

### 3.3 지적된 결함·누락 (hypercall 계약 보강 요구)

1. H1~H6은 항목 선정은 맞으나 계약으로는 부족 — 시그니처, 상태 전이, 실패 원자성, 오류 코드, 권한 토큰/manifest 바인딩이 필요.
2. H1은 pVM ID·측정값·manifest/capability handle 바인딩을 포함해야 Host의 임의 인스턴스와 승인 pVM을 구분 가능.
3. H2는 소유권 상태 머신, pVM accept 의미론, unshare 실패 처리, 회수 시 소거 완료 조건 명시 필요.
4. H4는 DP-11 G2 요구 필드(단조 시퀀스, 드롭 카운터, overflow, heartbeat) + VMID + SMMU fault source 포함 필요.
5. H6은 "SMMU 의미론" 수준으로는 DP-12 G3를 닫기에 부족 — EL2/TCB의 SMMU 집행 주체성, Host 직접 설정 차단, stream ID/device ID 바인딩, 권한 대조, revoke, sanitize, atomic rebind 정의 필요.
6. DP-11 G5 대응으로 생명주기 상태 보고와 로그 pVM heartbeat의 교차 대조 포맷이 본 DP에 포함되어야 함.

## 5. 결정

**후보 B(pVM별 VMM 프로세스 + 얇은 공용 관리 서비스)를 채택한다.** 공용 관리 서비스는 레지스트리·자원 예산·패키지 설치만 담당하고, 권한 판단은 DP-12의 TCB 게이트(fail-closed)에 위임한다. 후보 C 요소는 예약 메모리 풀·hypercall 래퍼 수준의 얇은 커널 드라이버로만 사용한다.

### 채택 조건 (게이트)

| # | 조건 | 처리 |
|---|------|------|
| G1 | **H1~H6 계약 명세 승격**: 각 항목에 시그니처·상태 전이·실패 원자성·오류 코드·권한 토큰(manifest) 바인딩을 포함한 계약 문서로 승격. H1: pVM ID+측정값+manifest handle 바인딩. H2: 소유권 상태 머신·accept 의미론·unshare 실패 처리·회수 시 소거 완료 조건. H4: DP-11 G2 필드(시퀀스·드롭·overflow·heartbeat)+VMID+SMMU fault source. H5: VMID 전달 경로만 확정하고 "VMID 단독 인증 금지"(DP-06 G2) 명문화. H6: DP-03 G2의 8항목과 동일 명세 | 계약 문서 작성 |
| G2 | **HV 파트 확정 게이트**: H1~H6 전체가 기 포팅 pKVM 제공 범위인지 확인 전까지 조건부. 미제공 항목은 해당 DP의 폴백 트리거(DP-06→후보 C, DP-11→후보 A, DP-03→후보 B). **H6은 4개 DP가 걸린 최대 리스크이므로 단건 조기 질의로 분리** | HV 파트 협의 |
| G3 | **가용성(b) 상태 보고 인터페이스**: 공용 관리 서비스의 pVM 상태 보고를 로그 pVM heartbeat와 교차 대조하는 포맷·경로를 골격 구조 요소로 추가. 비신뢰 Host 보고의 신뢰 한계(단절 추론 의존)를 명시 — DP-11 G5 폐쇄 | 상세 설계, DP-11 |
| G4 | **DP-03 병행 착수 1차 고정**: H1·H2 + H6 시그니처 수준 합의를 1차 고정 항목으로 확정 (설계질문 5 답) | HV 파트 협의 |

### 후속 DP에 주는 전제
