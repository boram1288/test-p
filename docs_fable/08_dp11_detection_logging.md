## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-11)

- 이상 접근 이벤트(Stage-2 fault, SMMU fault)는 누가 감지하고 기록하는가?
- 로그 저장소는 EL2 직접 기록, 별도 로그 pVM, TrustZone TEE, 외부 저장소 중 어디에 둘 것인가?
- Host가 로그를 삭제·변조·재정렬하지 못하도록 어떤 무결성 보호를 적용할 것인가?
- 실시간 탐지 알림과 사후 감사용 로그를 같은 경로로 처리할 것인가, 분리할 것인가?
- DP-07에서 사용할 감사 입력으로 어떤 이벤트 필드와 보존 기간을 보장할 것인가?

### 1.2 신뢰 모델 전제 (`06_dp01_trust_model.md`)

- 이상 접근 이벤트의 원천은 EL2다 — Stage-2 fault와 SMMU fault는 구조적으로 EL2로 트랩되며, 이 사실 자체는 Host가 위조할 수 없다.
- EL2는 수정 불가이므로(CONST-02), 이벤트를 EL2 밖으로 노출하는 방식은 기 포팅 pKVM이 제공하는 인터페이스 범위에 의존한다(HV 파트 잔여 협의 사항).
- 로그 저장소는 TCB 요소여야 한다(6.5절). 가용성(b) 거부 시도도 기록 대상이다.
- Host는 전달자 역할은 가능하나 기록의 무결성·완전성을 보장하는 주체가 될 수 없다.

## 2. 후보 구조 (생성: Claude)

세 후보 모두 "이벤트 원천 = EL2 (기 제공 노출 인터페이스)"는 공통이며, 수집·봉인·저장 주체가 다르다.

### 후보 A: Host 수집 + TEE 봉인 저장

Host의 수집 데몬이 EL2 이벤트 버퍼를 읽어 TEE에 전달하고, TEE가 해시 체인으로 연결·서명하여 변조 불가 저장소(RPMB 등 TEE 전용 스토리지)에 봉인한다.

- **구성**: EL2 이벤트 버퍼 → Host 수집 데몬(비신뢰 전달자) → TEE 로그 봉인 TA → RPMB/Secure Storage
- **무결성 보호**: TEE 해시 체인 + 서명. Host의 변조는 체인 검증으로 탐지된다.
- **누락(드롭) 대응**: Host가 이벤트를 버릴 수 있으므로, EL2 이벤트에 단조 증가 시퀀스 번호가 포함되어야 탐지 가능(기 제공 인터페이스에 포함 여부 확인 필요).
- **장점**: 신규 신뢰 컴포넌트 없음. TEE는 저빈도 봉인 작업만 수행(DP-01 TA 한정 규칙 부합).
- **단점**: 전달 지연·누락이 Host에 달려 있어 실시간 알림 경로로는 부적합. fault 폭주 시 SMC 빈도 부담.

### 후보 B: 전용 로그 pVM 수집 + TEE 앵커

전용 로그 pVM이 EL2 이벤트를 (기 제공 vIRQ/공유 버퍼 인터페이스로) 직접 수신·축적하고, 주기적으로 체인 다이제스트만 TEE에 봉인(앵커링)한다.

- **구성**: EL2 이벤트 버퍼 → 로그 pVM(수집·해시 체인 구성·로컬 축적) → 주기적 다이제스트 → TEE 앵커 → RPMB
- **무결성 보호**: 로그 본문은 로그 pVM의 격리 메모리·암호화 저장에, 위조 방지 앵커는 TEE에.
- **장점**: Host 개입 없이 수집하므로 누락 저항성과 실시간성이 가장 좋다. 고빈도 이벤트를 TEE 부담 없이 처리.
- **단점**: 로그 pVM은 TCB 편입 대상(DP-01 조건부 규칙: 이미지 검증의 TEE 앵커 연결 + 신뢰 모델 문서 개정 필요). 상시 pVM 1개의 메모리·CPU 비용(QA-06). EL2가 이벤트를 특정 pVM에 직접 전달하는 인터페이스가 기 제공 범위인지가 관건.

### 후보 C: Host 수집 + 외부 전송 (원격 로그 서버)

Host 수집 데몬이 이벤트를 서명 없이 외부 서버로 즉시 전송하고, 무결성은 서버 측에서 관리한다.

- **장점**: 디바이스 내 저장소 부담이 없고 구현이 단순하다.
- **단점**: Host가 전송 자체를 차단·선별할 수 있어 변조 불가 요건을 충족하지 못한다. 로봇의 오프라인 동작 구간에서 공백이 생긴다. 단독 후보로는 신뢰 모델과 정합하지 않는다.

### 평가 기준

| 기준 | 설명 |
|------|------|
| E1. 변조·삭제 저항성 | Host 커널 권한 공격자가 기록을 위조·삭제·재정렬할 수 없는가 |
| E2. 누락 탐지 | 전달 주체가 이벤트를 버렸을 때 탐지 가능한가 |
| E3. CONST-02 적합 | EL2 기 제공 인터페이스 범위에서 실현 가능한가 (필요 인터페이스의 종류) |
| E4. 성능·자원 | fault 폭주 시 처리량, TEE 호출 빈도, 상시 자원 비용(QA-06) |
| E5. 실시간 알림 | 침해 시도를 운영 주체에 적시에 알릴 수 있는가 (QS-01 응답) |
| E6. DP-07 적합성 | 감사 패키지 입력(필드·보존·증명 가능성)으로 적합한가 |

### 3.1 후보별 평가 (상/중/하)

| 기준 | A: Host 수집 + TEE 봉인 | B: 로그 pVM + TEE 앵커 | C: Host 수집 + 외부 전송 |
|------|:---:|:---:|:---:|
| E1 변조·삭제 저항성 | 중 — 전달된 로그는 보호되나 Host가 전달 전 삭제 가능 | 상 — 직접 수집 + TEE 앵커링 성립 시 변조·삭제 곤란 | 하 — Host가 선별·차단·위조 가능 |
| E2 누락 탐지 | 중 — EL2 단조 시퀀스·드롭 카운터가 있어야 탐지 가능 | 상 — 직접 전달 + 시퀀스 + 오버플로 카운터 시 가장 강함 | 하 — 버린 이벤트를 알 방법 없음 |
| E3 CONST-02 적합 | 중 — Host가 읽는 이벤트 인터페이스·시퀀스 필드 기 제공 필요 | 중 — 특정 pVM 직접 전달(vIRQ·공유버퍼) 기 제공 필요, 없으면 불가 | 상 — EL2 요구 최소 |
| E4 성능·자원 | 중 — fault 폭주 시 TEE 호출 비용 증가 | 중 — 고빈도 유리, 상시 pVM 자원 비용 | 중 — 네트워크·오프라인 병목 |
| E5 실시간 알림 | 하 — 침해 Host가 지연·차단 가능 | 상 — 직접 수신으로 실시간 경로 구성 가능 | 하 — Host·네트워크 의존 |
| E6 DP-07 적합성 | 중 — 봉인은 가능하나 완전성 증명이 조건부 | 상 — 원천성·순서성·앵커링을 갖춘 감사 입력 | 하 — 변조 불가 원천 로그 아님 |

### 3.2 종합 판정과 권고

- **A: 부분 적합** — 봉인은 가능하나 Host 누락 문제로 완전한 해법이 아님.
- **B: 최선 후보** — 단 EL2의 직접 이벤트 전달 인터페이스 존재 여부가 결정적.
- **C: 단독 부적합** — 외부 미러·운영 편의 기능으로만 사용 가능.

**권고안**: 하이브리드. 기본 구조는 B(로그 pVM 직접 수집 + 해시 체인 + TEE 주기 앵커링), 원격 전송은 TEE 서명 요약·봉인 로그의 보조 복제 경로로만(C 격하). EL2가 로그 pVM 직접 전달을 지원하지 않으면 A를 임시 기준선으로 하되, EL2 이벤트에 단조 시퀀스 번호·드롭 카운터·버퍼 오버플로 표시·주기적 heartbeat가 반드시 포함되어야 함 — 이 조건이 없으면 Host 누락을 탐지할 수 없어 DP-11 요구 미충족.

### 3.3 지적된 결함·누락

1. 후보 A는 Host 삭제·지연을 막지 못하므로 "변조 불가 로깅"이 "전달된 로그의 변조 방지"로 축소됨을 명시해야 함.
2. 후보 B는 로그 pVM의 TCB 편입 조건·이미지 검증·신뢰 모델 문서 개정 절차 필요.
3. 가용성(b) 거부 시도(pVM 생성 거부, 스케줄링 기아, 강제 종료)를 어떤 이벤트로 기록할지 정의 부족.
4. 신뢰 가능한 시간원, 로그 필드 스키마, 보존 기간, 저장소 full 처리, fault 폭주 시 샘플링·드롭 정책 미정의.
5. DP-07에 필요한 증빙 형식, 서명 키 수명주기, 로그 검증 절차 미정의.

**교차 검수 지적 (핵심 공백)**: 가용성(b) 거부 시도(pVM 생성 거부, 스케줄링 기아, 강제 종료)는 EL2 fault가 아니라 **Host 동작의 부재·방해**라서 EL2 이벤트 원천에 잡히지 않을 수 있다. 현재 권고안에 이 감지 메커니즘이 비어 있다.

## 5. 결정

**하이브리드를 채택한다.** 기본 구조는 후보 B(전용 로그 pVM이 EL2 이벤트 직접 수집, 해시 체인 구성, TEE 주기 앵커링), 외부 전송(후보 C)은 TEE 서명 요약·봉인 로그의 보조 복제 경로로만 사용한다. EL2 직접 전달 미지원 시 후보 A를 임시 기준선으로 한다.

### 채택 조건 (게이트)

| # | 조건 | 처리 |
|---|------|------|
| G1 | **성립 게이트**: EL2가 특정 pVM에 이벤트를 직접 전달하는 인터페이스(vIRQ·공유 버퍼)가 기 제공 범위인지 HV 파트에 확인. 미지원 시 후보 A 폴백 | HV 파트 협의 |
| G2 | **공통 이벤트 요구**: EL2 이벤트에 단조 시퀀스 번호, 드롭 카운터, 버퍼 오버플로 표시, 주기적 heartbeat 포함 — 후보 A·B 공통(검수 이견 2). 이 조건 미충족 시 누락 탐지가 불가하여 DP-11 요구 미충족 | HV 파트 협의 |
| G3 | **로그 pVM TCB 편입 절차**: (i) 로그 pVM 이미지 검증을 TEE 앵커·Secure Boot 체인에 연결(DP-09 연계), (ii) `06_dp01_trust_model.md` 개정 + 영향 평가. DP-01의 조건부 2차 앵커 규칙 적용 | DP-09, 신뢰 모델 개정 |
| G4 | **보증 시간 경계 명시**: 실시간 알림은 로그 pVM이 직접 발행(TEE 무관). TEE 앵커링은 저빈도 주기로 수행하며, 미앵커 구간 로그의 무결성 보증 범위는 로그 pVM 격리 메모리에 한정됨을 증빙 문서(DP-07)에 명시 | DP-07 |
| G5 | **가용성(b) 거부 이벤트 감지 정의**: EL2 fault로 잡히지 않는 거부 시도는 로그 pVM의 heartbeat 단절 감지와 생명주기 관리자(DP-02) 상태 보고의 교차 대조로 감지하는 구조를 DP-02 설계에 요구 사항으로 전달 | DP-02 |
| G6 | **미정의 항목 분담**: 신뢰 시간원·로그 필드 스키마·보존 기간·저장소 full 처리·폭주 시 샘플링 정책은 본 DP 상세 설계에서, 증빙 형식·서명 키 수명주기·로그 검증 절차는 DP-07에서 정의 | 상세 설계, DP-07 |

### 후속 DP에 주는 전제

- DP-07의 감사 입력은 "로그 pVM 해시 체인 + TEE 주기 앵커 + G2 누락 탐지 필드"다.
- 로그 pVM은 본 과제 최초의 서비스 pVM TCB 편입 사례가 되며, 검수의 교차 지적에 따라 추가 서비스 pVM 편입은 통합 영향 평가를 거친다(서비스 pVM 상한 정책).
- TEE 앵커링 TA는 DP-06 G5의 TA 자원 예산표에 등록 후 도입한다.

> 상태: **조건부 채택 (검수 완료)** — G1·G2 확인(HV 파트) 결과에 따라 확정
