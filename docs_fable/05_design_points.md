# Design Point 목록

> 본 문서는 `04_architectural_drivers.md` 5.2절의 설계 착수 순서(T-1~T-5 연계 6단계)와 `99_additional_design_points.md`의 교차(cross-cutting) Design Point(DP-A~E)를 통합하여, 아키텍처 설계 착수 전 확정해야 할 **모든 Design Point**를 정리한다.

> 진행 순서: 요구사항 수집 → 요구사항 도출 → QAW → Architectural Driver 선정 → **Design Point 목록(본 문서)**

---

## 1. 개요

Design Point(DP)는 Architectural Driver로부터 도출되는 구체적인 설계 결정 단위다. DP를 해결하기 전까지는 후보 구조를 평가할 기준이 성립하지 않는다.

본 문서의 DP는 두 가지 유형으로 구성된다.

| 유형 | 설명 | 건수 |
|------|------|:----:|
| **T-연계 DP** | 핵심 기술 T-1~T-5(또는 공통 전제)와 직결된 설계 결정. 착수 순서 1~6에 배정 | 6건 |
| **교차 DP** | 단일 T에 속하지 않고 Driver에서 직접 요구되는 설계 결정. 특정 착수 순서에 병행 배치 | 5건 |

## 3. 착수 순서별 Design Point 상세

### 착수 순서 1

---

#### DP-01: 신뢰 모델 확정

- **유형**: T-연계 (T-1~T-5 공통 전제)
- **관련 Driver**: QA-01, QA-02, CONST-02

**설계 결정 사항**
- TCB 범위: Host 측 프레임워크(미들웨어·커널 드라이버)를 TCB 안에 포함하는가 외에 두는가
- 공격자 모델: 공격자의 권한 수준(Host 커널 권한, pVM 내부 권한) 및 가정 범위
- 보호 속성 범위: 기밀성·무결성만 보장하고 가용성(DoS)은 보호 범위 밖으로 명시하는가
- 신뢰 주체 목록: EL2(pKVM), TrustZone EL3·S-EL1, 그 외 신뢰 주체 포함 여부

**산출물**: 신뢰 모델 문서 (TCB 목록, 공격자 모델, 보호 속성별 의존 메커니즘)

> 상세 논거: `99_trust_boundary_qna.md` 참조

---

#### DP-07: 격리 증빙·감사 체계

- **유형**: 교차 (CONST-04 유일 미배정 Driver)
- **관련 Driver**: CONST-04

**설계 결정 사항**
- 감사 로그 저장 위치와 포맷 (비신뢰 Host가 위·변조할 수 없어야 함)
- 격리 상태를 외부에 증명하는 attestation 구조 채택 여부
- 증빙 요구 수준: 내부 검증용인가, 제3자 감사용인가

**연계 DP**: DP-11(변조 불가 로그)이 이 DP의 기반 입력

---

#### DP-11: 침해 시도 탐지·변조 불가 로깅

- **유형**: 교차
- **관련 Driver**: QA-01(QS-01 응답 요소), CONST-04

**설계 결정 사항**
- 이상 접근 이벤트(Stage-2 fault, SMMU fault) 기록 주체와 저장소
- 기록 저장소를 Host로부터 격리하는 방법 (EL2 직접 기록 vs 별도 로그 pVM)
- 실시간 탐지 알림과 사후 감사용 로그 분리 여부

**연계 DP**: DP-07(감사 체계)의 핵심 입력

---

### 착수 순서 2

---

#### DP-02: 생명주기·다중 pVM 관리 골격 설계

- **유형**: T-연계 (T-1)
- **관련 Driver**: FR-01, FR-02, CONST-01

**설계 결정 사항**
- 프레임워크 컴포넌트 분할: 생명주기 관리자, pVM 제어 인터페이스, 자원 할당자의 경계
- 다중 도메인 독립 관리 구조: 도메인별 메모리·자원 분리 단위
- Linux 네이티브 인터페이스 선택: KVM ioctl·VFIO·dma-buf 등 구체적 인터페이스 결정(CONST-01 이행)
- pKVM hypercall 인터페이스 확정: HV 파트와 조기 인터페이스 계약 필요(CONST-02 이행)

---

### 착수 순서 3

---

#### DP-03: HW IP SW 중재 구조 설계

- **유형**: T-연계 (T-2)
- **관련 Driver**: FR-03, QA-03, QA-11, CONST-05

**설계 결정 사항**
- 중재자 위치: Host 커널 드라이버에 두는가 vs 별도 서비스 pVM에 두는가
  - Host 커널 배치: 중재자는 비신뢰이므로 격리는 SMMU(EL2 통제)가 보장해야 함
  - 서비스 pVM 배치: 중재자에 신뢰가 필요한 경우 선택
- SMMU/IOMMU 제어 구조: pVM 사용 구간 동안 DMA 경로 격리 방법
- 잔류 데이터 소거 메커니즘: 사용 주체 전환 시 이전 도메인 데이터 소거 방법 및 시점
- 중재·전환 오버헤드 측정 방법: QA-11 수치 목표 검증을 위한 PoC 설계

---

### 착수 순서 4

---

#### DP-04: zero-copy 보안 채널 구조 설계

- **유형**: T-연계 (T-3)
- **관련 Driver**: FR-04, QA-04, QA-05

**설계 결정 사항**
- 공유 메모리 채널 구조: Secure Camera→Secure AI 대용량 영상 데이터 zero-copy 전달 방법
- 제어 경로와 데이터 경로 분리: RPC(제어)와 공유 메모리(데이터) 역할 분리 구조
- 채널 수립·해제 프로토콜: pVM 간 채널 협상 절차, 실패 처리
- 격리 보장 방법: Host(비신뢰)가 채널 데이터를 열람할 수 없도록 하는 메커니즘

---

#### DP-10: pVM 아이덴티티·채널 상대 인증

- **유형**: 교차
- **관련 Driver**: QA-02 × FR-04

**설계 결정 사항**
- pVM 아이덴티티(도메인 ID, 인증서) 발급 주체와 검증 방법
- 채널 수립 시 상호 인증 주체: 프레임워크 vs EL2 vs 앱 계층
- 아이덴티티와 DP-09(부팅 무결성 검증) 연계 여부

---

### 착수 순서 5

---

#### DP-05: 확장 인터페이스 계약 고정

- **유형**: T-연계 (T-4)
- **관련 Driver**: FR-06, FR-07, QA-07, QA-08, CONST-06

**설계 결정 사항**
- 워크로드 패키지 형식: 메타데이터·이미지·설정의 패키지 구조 정의
- 게스트 ABI 고정: 부팅 규약, 가상 디바이스 인터페이스, pVM↔프레임워크 통신 규약
- 플러그인 로딩 구조: 펌웨어 재배포 없이 동적 탑재를 가능하게 하는 로딩 메커니즘
- Secure OS 추상화 인터페이스: Secure OS 교체 시 관련 없는 SW 무수정을 보장하는 추상화 경계

---

#### DP-08: 보호 자산 저장(at-rest) 보호

- **유형**: 교차
- **관련 Driver**: QA-01 × FR-08

**설계 결정 사항**
- AI 모델 가중치·워크로드 패키지 저장 시 암호화 키 관리 주체 (TrustZone TEE, pKVM EL2, 별도 키 관리 서비스)
- 암호화 범위: 패키지 단위인가, 파일시스템 전체인가
- 복호화 시점과 복호화 키 전달 경로

---

#### DP-09: pVM 부팅 무결성 검증

- **유형**: 교차
- **관련 Driver**: QA-01 × FR-06

**설계 결정 사항**
- pVM 이미지 서명 검증 주체: EL2 vs Host 드라이버 vs 별도 서비스 pVM
- measured boot 체인 범위: 어디서 시작하고 어디서 끊는가
- CONST-02(EL2 수정 불가) 제약 하에서 이미지 진위 보장 방법

**연계 DP**: DP-10(상대 인증)과 아이덴티티 연계

---

### 착수 순서 6

---

#### DP-06: TrustZone 연동 구조 배치

- **유형**: T-연계 (T-5)
- **관련 Driver**: FR-08, CONST-03

**설계 결정 사항**
- SMC 포워딩·라우팅 구조: pVM이 기존 TEE(키 관리·인증)를 호출하는 경로 설계
- 기존 TrustZone SMC 경로 무회귀 보장: 연동 계층 배치 방법
- pVM→TEE 신뢰 연결 방법: pVM이 TEE를 호출할 때 신뢰 연결 고리 확보

## 4. 우선 결정 필요 Design Point

DP-01, DP-07, DP-11은 **모든 후속 DP의 전제**가 되며, 이 세 DP가 미결 상태이면 나머지 DP의 후보 구조를 평가할 기준이 성립하지 않는다.

| 우선순위 | DP | 이유 |
|:--------:|----|------|
| 1 | DP-01 (신뢰 모델) | TCB 범위와 보호 속성이 미결이면 DP-02~06 전체의 구조 선택지를 평가할 수 없음 |
| 1 | DP-07 (격리 증빙 체계) | 유일한 미배정 Driver(CONST-04). GDPR 규제 증빙 구조를 결정하지 않으면 아키텍처가 규제 요건을 충족하는지 판단 불가 |
| 1 | DP-11 (탐지·로깅) | QS-01 응답 요소의 공백. DP-07의 입력이 됨 |
| 2 | DP-03 (HW IP 중재) | 착수 순서 3에서 PoC로 실현 가능성을 먼저 검증해야 전체 구조 방향을 확정할 수 있음 |
| 3 | DP-08, DP-09, DP-10 | 해당 착수 순서(4·5) 설계 시 병행 검토 |
