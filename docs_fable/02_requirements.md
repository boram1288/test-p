# 요구사항 도출

> 본 문서는 `01_requirements_collection.md`에서 수집한 VOS-01~VOS-18을 입력으로, 요구사항을 **FR(기능 요구사항), QA(품질 속성 요구사항), CONST(제약사항)** 로 분류·정제한다.
> 진행 순서: 요구사항 수집 → **요구사항 도출(본 문서)** → 품질 속성 선정(QAW) → Architectural Driver 선정

---

## 1. 도출 원칙

- **추적성**: 모든 요구사항은 하나 이상의 VOS에서 도출되며, 출처 VOS를 명시한다. 역으로 모든 VOS는 하나 이상의 요구사항에 반영된다(4절 추적 매트릭스).
- **분류 기준**:
  - **FR**: 시스템이 "무엇을 하는가". 프레임워크가 제공해야 할 기능.
  - **QA**: 시스템이 "얼마나 잘 하는가". 자극(stimulus)과 응답(response)으로 표현 가능한 품질 속성. 측정 가능한 응답 측정치(response measure)는 3단계 QAW에서 구체화한다.
  - **CONST**: 설계 선택지를 제한하는 기술·조직·규제상 제약. 협상 대상이 아닌 전제 조건.
- **명명**: FR-nn / QA-nn / CONST-nn.

## 2. FR (기능 요구사항)

| ID | 요구사항 | 설명 | 출처 VOS |
|----|---------|------|----------|
| FR-01 | **pVM 생명주기 관리** | 프레임워크는 pVM(protected VM)의 생성·시작·정지·종료와 메모리 등 자원의 할당·회수를 관리해야 한다. | VOS-01, VOS-07 |
| FR-02 | **다중 pVM 동시 운용** | 프레임워크는 Secure Camera, Secure AI 등 복수의 격리 도메인(pVM)을 서로 독립적으로 동시에 생성·운용해야 한다. | VOS-07 |
| FR-03 | **HW IP 보안 공유(할당·중재)** | 프레임워크는 다중 컨텍스트를 지원하지 않는 ISP·NPU 등 HW IP를 Host와 pVM이 동시에 사용할 수 있도록 SW 중재 구조로 공유·할당하고, pVM이 사용하는 동안에는 SMMU/IOMMU를 설정하여 해당 pVM만 HW IP의 DMA 경로를 사용하도록 해야 한다. | VOS-02, VOS-08 |
| FR-04 | **격리 도메인 간 보안 통신** | 프레임워크는 pVM↔pVM(예: Secure Camera→Secure AI), pVM↔Host 간 데이터를 제3자(Host 포함 비신뢰 주체)에게 노출하지 않고 전달하는 보안 채널(공유 메모리·RPC)을 제공해야 한다. | VOS-09 |
| FR-05 | **최소 결과 전달** | 프레임워크는 격리 영역 밖(Host OS)으로는 AI 판단 결과만 전달하고, 영상 원본·모델 가중치·추론 중간 데이터는 격리 영역 내에만 유지해야 한다. | VOS-01 |
| FR-06 | **보안 워크로드 동적 탑재** | 프레임워크는 신규 보안 워크로드를 패키징하여 펌웨어 재배포 없이 동적으로 pVM으로 탑재(설치·로딩·교체)하는 기능을 제공해야 한다. | VOS-05, VOS-15 |
| FR-07 | **Secure OS pVM 실행 지원** | 프레임워크는 이식·수정된 기존 Secure OS를 pVM 격리 도메인에서 게스트로 실행하기 위한 실행 환경(부팅, 가상 디바이스, 통신 경로)을 제공해야 한다. | VOS-11, VOS-15 |
| FR-08 | **TrustZone 공존·연동** | 프레임워크는 기존 TrustZone Secure OS 기반 TEE 기능(키 관리·인증)과 공존하며, pVM 워크로드가 기존 TEE 기능을 호출·연동할 수 있는 인터페이스를 제공해야 한다. | VOS-12 |
| FR-09 | **pVM 상태 모니터링·장애 처리** | 프레임워크는 pVM의 상태를 모니터링하고, 비정상 종료 시 자원을 안전하게 회수(메모리 반환 전 격리 해제 처리 포함)하며 해당 pVM을 재시작할 수 있어야 한다. | VOS-18 |
| FR-10 | **개발자 API 제공** | 프레임워크는 pVM 생성·실행·통신·워크로드 탑재를 위한 공개 API와 문서를 제공해야 한다. | VOS-14 |
| FR-11 | **HW IP 사용 주체 전환 처리** | 프레임워크는 HW IP의 사용 주체가 전환될 때(pVM→Host, pVM→타 pVM 등) 이전 사용 주체의 잔류 데이터(내부 버퍼·레지스터·매핑)를 소거하고 격리 설정을 해제·재구성한 후 다음 사용 주체에 할당해야 한다. | VOS-08 |

## 3. QA (품질 속성 요구사항)

각 QA는 "자극 → 기대 응답" 형태로 기술한다. 정량적 응답 측정치는 3단계 QAW에서 시나리오로 구체화한다.

| ID | 품질 속성 | 요구사항 (자극 → 기대 응답) | 출처 VOS |
|----|----------|---------------------------|----------|
| QA-01 | **보안 (기밀성)** | Host OS(Linux 커널 포함)가 완전히 침해되어도 → pVM 내 영상 원본·AI 모델 가중치·추론 중간 데이터는 노출되지 않는다. | VOS-01, VOS-16, VOS-17 |
| QA-02 | **보안 (도메인 간 격리)** | 하나의 pVM(예: Secure Camera)이 침해되어도 → 다른 pVM(예: Secure AI)의 메모리·데이터는 노출되지 않는다. | VOS-07 |
| QA-03 | **보안 (DMA 경로 격리)** | HW IP가 pVM에 할당된 동안 Host 또는 비할당 pVM이 HW IP의 DMA를 이용해 격리 메모리 접근을 시도해도 → SMMU/IOMMU에 의해 차단되며, 사용 주체 전환 후에는 이전 도메인의 잔류 데이터에 접근할 수 없다. | VOS-08 |
| QA-04 | **성능 (실시간 처리)** | 카메라 영상 입력이 지속 유입될 때 → ISP·NPU HW 가속을 통해 격리 환경에서도 실시간 영상 처리·AI 추론 요구 속도(프레임율·지연)를 만족한다. | VOS-02 |
| QA-05 | **성능 (통신 오버헤드)** | 도메인 간(Camera→AI, pVM↔Host) 대용량 영상 데이터 전달 시 → 통신 오버헤드가 파이프라인 실시간성을 해치지 않는 수준으로 유지된다. | VOS-09 |
| QA-06 | **자원 효율** | 보안 프레임워크 활성화 시 → 비격리 구성 대비 메모리·전력 오버헤드가 제품 탑재 가능한 한도 이내로 유지된다. | VOS-04 |
| QA-07 | **확장성** | 신규 보안 워크로드(예: 개인정보 처리, 펌웨어 보호)가 추가될 때 → 프레임워크 소스 수정 없이 패키징·탑재만으로 수용된다. | VOS-05 |
| QA-08 | **변경 용이성 (Secure OS 교체)** | 탑재된 Secure OS를 다른 Secure OS로 교체할 때 → Secure OS와 관련 없는 SW(프레임워크, Host 미들웨어, 타 워크로드)는 수정이 발생하지 않는다. | VOS-15 |
| QA-09 | **이식성 (Secure OS 이식)** | 기존 Secure OS를 pVM 환경으로 이식할 때 → 이식 대상 인터페이스(가상 디바이스, 통신, 부팅 규약)가 명확히 정의되어 있어, 프레임워크 변경이 이식 작업의 재수행을 유발하지 않는다. | VOS-11 |
| QA-10 | **가용성** | pVM이 비정상 종료되거나 보안 워크로드가 오동작해도 → Host OS, 다른 pVM, 로봇의 기본 동작은 영향받지 않고 계속 동작한다. | VOS-18 |
| QA-11 | **시험 용이성** | 격리 보장(QA-01~03)에 대한 검증 요구 시 → Host 침해를 모사한 시험 등 객관적·재현 가능한 방법으로 격리 유지 여부를 검증할 수 있다. | VOS-13 |
| QA-12 | **사용성 (개발자)** | 보안 비전문 개발자가 보안 워크로드를 탑재하려 할 때 → 공개 API와 문서만으로 pVM 생성·통신·탑재를 수행할 수 있다. | VOS-14 |
| QA-13 | **성능 (HW IP 공유 오버헤드)** | Host와 pVM이 HW IP를 동시에 사용(SW 중재)할 때 → 중재·전환 오버헤드가 Host 일반 기능과 보안 파이프라인 양쪽의 실시간 요구를 해치지 않는 수준으로 유지된다. | VOS-02 |

## 4. CONST (제약사항)

| ID | 제약사항 | 설명 | 출처 VOS |
|----|---------|------|----------|
| CONST-01 | **Linux 네이티브** | 프레임워크는 Android 스택(VirtualizationService 등)에 의존하지 않고 Linux(Yocto, Ubuntu 등)에서 네이티브로 동작해야 한다. | VOS-03 |
| CONST-02 | **pKVM 커널 전제** | 기 포팅된 pKVM 커널(EL2 Hypervisor)을 전제로 하며, EL2 코드 수정이 필요한 설계는 불가하다. 제공되는 hypercall 인터페이스 범위 내에서 설계한다. | VOS-10 |
| CONST-03 | **일정·인력** | 2026-10-30까지 Secure Vision AI End-to-End 데모를 완성해야 하며, 가용 인력은 Security 파트 14명(국내 9 + SRCX 5), HV 파트 3명이다. | VOS-06 |
| CONST-04 | **기존 TEE 경로 무회귀** | 기존 TrustZone Secure OS의 SMC 경로와 TEE 기능(키 관리·인증)은 프레임워크 도입 후에도 변경 없이 그대로 동작해야 한다. | VOS-12 |
| CONST-05 | **규제 준수** | GDPR·개인정보보호법이 요구하는 영상·생체 데이터 처리의 기술적 격리를 충족하고 증빙 가능해야 한다. | VOS-17 |
| CONST-06 | **타겟 HW** | 타겟은 ISP·NPU 등 전용 HW IP를 탑재한 ARM 기반 로봇용 커스텀 SoC이다. ISP·NPU는 다중 컨텍스트(HW 가상화)를 지원하지 않는 단일 컨텍스트 HW이며 Host와 격리 도메인이 공유하는 자원이다. HW IP의 하드웨어 설계 변경은 불가하다(기존 SoC HW를 전제). | VOS-02, VOS-08, `00_overview.md` 5.2절 |
| CONST-07 | **Secure OS 신규 개발 제외** | Secure OS를 처음부터 개발(from scratch)하지 않으며, 기존 Secure OS의 이식·수정만 범위에 포함한다. | `00_overview.md` 5.2절 |
