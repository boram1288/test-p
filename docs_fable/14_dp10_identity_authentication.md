## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-10)

- pVM 아이덴티티는 도메인 ID, 인증서, 측정값 중 무엇으로 표현할 것인가?
- 아이덴티티 발급과 검증은 프레임워크, EL2, TrustZone TEE, 앱 계층 중 누가 수행할 것인가?
- 채널 수립 시 상호 인증은 어떤 프로토콜과 실패 처리로 수행할 것인가?
- DP-09의 pVM 이미지 측정·검증 결과를 아이덴티티와 연계할 것인가?
- 권한 정책(DP-12)에 따라 어떤 pVM 조합만 채널 수립을 허용할 것인가?

### 1.2 선행 DP 결정이 주는 전제

- **DP-06 G2**: VMID는 전달 경로 식별자일 뿐이며 단독으로 인증 근거로 사용 금지. 신뢰 아이덴티티는 DP-09 측정값과의 결합으로 완성되고, 완성 후 DP-06 경로에 역반영된다.
- **DP-12**: 채널 수립 허용 조합은 manifest의 정적 선언이며, 인증 성공 이후의 허용 판단은 manifest 대조로 수행한다. manifest 권한은 부팅 측정값에 바인딩된다(G5).
- **DP-01/DP-06**: 아이덴티티 발급·검증의 신뢰 앵커 우선순위는 EL2 기 제공 메커니즘 → TEE(1차) → 조건부 서비스 pVM(2차). TEE TA는 저빈도·소용량 한정(TA 예산표 등록).
- **DP-04 (병행)**: 상호 인증 프로토콜은 채널 수립 절차에 삽입되므로 DP-04와 결과를 상호 반영한다.

## 2. 후보 구조 (생성: Claude)

### 후보 A: 측정값 바인딩 TEE 발급 자격 (measured identity)

pVM 아이덴티티를 "부팅 측정값(DP-09)에 바인딩되어 TEE가 발급하는 자격(per-instance 키·토큰)"으로 정의한다. pVM 부팅 시 검증·측정 절차의 일부로 TEE가 해당 인스턴스에만 자격을 방출하고(측정값 불일치 시 미방출), 채널 수립 시 양단이 이 자격으로 상호 인증한다(challenge-response). 인증 성공 후 manifest의 허용 조합을 대조하여 채널을 허가한다.

- **발급**: TEE — 부팅 검증 통과 시에만, pVM 전용 공유 영역(DP-04 정의)으로 방출
- **검증**: 채널 수립 시 양단 상호 challenge-response, 검증 기준(승인 측정값·발급 체인)은 TEE 서명으로 보증
- **장점**: "지금 실행 중인 코드가 승인된 그 코드"임을 측정값으로 보증 — Host가 만든 임의 인스턴스는 자격을 받지 못해 구조적으로 배제된다(DP-06 G2의 요구를 정면으로 해결). DICE류 업계 패턴과 부합.
- **단점**: DP-09(측정·검증)와 TEE 발급 TA가 선행 완성되어야 한다. 발급은 pVM 부팅 시 1회(저빈도)지만 TA 예산표 등록이 필요하다.

### 후보 B: VMID + 프레임워크 레지스트리 (비교 기준선)

EL2 VMID와 공용 관리 서비스의 레지스트리("VMID 7 = Secure Camera")로 상대를 식별한다.

- **장점**: 추가 인프라 없이 즉시 구현 가능.
- **단점**: 레지스트리는 비신뢰 Host가 관리하므로 침해된 Host가 임의 pVM을 "Secure AI"로 등록하면 우회된다. DP-06 G2(VMID 단독 인증 금지)에 의해 단독 채택이 금지된 기준선이다. 부팅 초기·PoC 한정 임시 수단으로만 의미.

### 후보 C: 워크로드 내장 인증서 (앱 계층 상호 인증)

워크로드 이미지에 내장된 키·인증서로 TLS류 상호 인증을 앱 계층에서 수행한다.

- **장점**: 프레임워크·TEE 의존이 없어 워크로드가 자체 완결적이다. 표준 프로토콜 재사용 가능.
- **단점**: 키가 이미지에 내장되면 이미지가 유출·복제될 때 아이덴티티도 복제된다(인스턴스 구분 불가). 측정값과 연계되지 않으면 "어떤 코드가 실행 중인지"를 보증하지 못해 변조 이미지도 올바른 키만 가지면 인증을 통과한다. 워크로드마다 인증 구현이 달라져 QA-07(프레임워크 수정 없는 수용)의 공통 기반이 되지 못한다.

### 공통 설계 요소

- **실패 처리**: 인증 실패·manifest 불허 조합 시 채널 수립 거부(fail-closed) + DP-11 이벤트 기록. 재시도 정책은 지수 백오프, 반복 실패는 침해 시도 의심 이벤트로 격상.
- **자격 수명**: pVM 인스턴스 수명과 동일(재부팅 시 재발급). 철회(DP-12 G4)는 TEE의 자격 무효화 + 진행 중 채널 강제 해제(DP-04)로 연동.

### 평가 기준

| 기준 | 설명 |
|------|------|
| E1. 위조 저항성 | 침해된 Host(AT-2)·침해된 pVM(AT-3)이 타 도메인 아이덴티티를 위조·도용할 수 없는가 |
| E2. 코드 진위 보증 | 아이덴티티가 "승인된 코드의 실행 중 인스턴스"를 보증하는가 (DP-09 연계, DP-06 G2 폐쇄) |
| E3. 정책 연계 | manifest 허용 조합 대조(DP-12)와 자연스럽게 결합되는가 |
| E4. 채널 절차 통합 | DP-04 수립 절차에 삽입 가능한 프로토콜 복잡도·지연 (수립 시 1회) |
| E5. TEE 부하 | 발급·검증의 TA 예산 적합성 (저빈도·소용량 규칙) |
| E6. 확장성 | 신규 워크로드 추가 시 아이덴티티 체계 수정 없이 수용 (QA-07) |
