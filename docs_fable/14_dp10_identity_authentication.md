## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-10)

- pVM 아이덴티티는 도메인 ID, 인증서, 측정값 중 무엇으로 표현할 것인가?
- 아이덴티티 발급과 검증은 프레임워크, EL2, TrustZone TEE, 앱 계층 중 누가 수행할 것인가?
- 채널 수립 시 상호 인증은 어떤 프로토콜과 실패 처리로 수행할 것인가?
- DP-09의 pVM 이미지 측정·검증 결과를 아이덴티티와 연계할 것인가?
- 권한 정책(DP-12)에 따라 어떤 pVM 조합만 채널 수립을 허용할 것인가?

### 1.2 선행 DP 결정이 주는 전제

- **DP-06 G2**: VMID는 전달 경로 식별자일 뿐이며 단독으로 인증 근거로 사용 금지. 신뢰 아이덴티티는 DP-09 측정값과의 결합으로 완성되고, 완성 후 DP-06 경로에 역반영된다.
- **DP-12**: 채널 수립 허용 조합은 manifest의 정적 선언이며, 인증 성공 이후의 허용 판단은 manifest 대조로 수행한다. manifest 권한은 부팅 측정값에 바인딩된다(G5).
- **DP-01/DP-06**: 아이덴티티 발급·검증의 신뢰 앵커 우선순위는 EL2 기 제공 메커니즘 → TEE(1차) → 조건부 서비스 pVM(2차). TEE TA는 저빈도·소용량 한정(TA 예산표 등록).
- **DP-04 (병행)**: 상호 인증 프로토콜은 채널 수립 절차에 삽입되므로 DP-04와 결과를 상호 반영한다.

## 2. 후보 구조 (생성: Claude)

### 후보 A: 측정값 바인딩 TEE 발급 자격 (measured identity)

pVM 아이덴티티를 "부팅 측정값(DP-09)에 바인딩되어 TEE가 발급하는 자격(per-instance 키·토큰)"으로 정의한다. pVM 부팅 시 검증·측정 절차의 일부로 TEE가 해당 인스턴스에만 자격을 방출하고(측정값 불일치 시 미방출), 채널 수립 시 양단이 이 자격으로 상호 인증한다(challenge-response). 인증 성공 후 manifest의 허용 조합을 대조하여 채널을 허가한다.

- **발급**: TEE — 부팅 검증 통과 시에만, pVM 전용 공유 영역(DP-04 정의)으로 방출
- **검증**: 채널 수립 시 양단 상호 challenge-response, 검증 기준(승인 측정값·발급 체인)은 TEE 서명으로 보증
- **장점**: "지금 실행 중인 코드가 승인된 그 코드"임을 측정값으로 보증 — Host가 만든 임의 인스턴스는 자격을 받지 못해 구조적으로 배제된다(DP-06 G2의 요구를 정면으로 해결). DICE류 업계 패턴과 부합.
- **단점**: DP-09(측정·검증)와 TEE 발급 TA가 선행 완성되어야 한다. 발급은 pVM 부팅 시 1회(저빈도)지만 TA 예산표 등록이 필요하다.

### 후보 B: VMID + 프레임워크 레지스트리 (비교 기준선)

EL2 VMID와 공용 관리 서비스의 레지스트리("VMID 7 = Secure Camera")로 상대를 식별한다.

- **장점**: 추가 인프라 없이 즉시 구현 가능.
- **단점**: 레지스트리는 비신뢰 Host가 관리하므로 침해된 Host가 임의 pVM을 "Secure AI"로 등록하면 우회된다. DP-06 G2(VMID 단독 인증 금지)에 의해 단독 채택이 금지된 기준선이다. 부팅 초기·PoC 한정 임시 수단으로만 의미.

### 후보 C: 워크로드 내장 인증서 (앱 계층 상호 인증)

워크로드 이미지에 내장된 키·인증서로 TLS류 상호 인증을 앱 계층에서 수행한다.

- **장점**: 프레임워크·TEE 의존이 없어 워크로드가 자체 완결적이다. 표준 프로토콜 재사용 가능.
- **단점**: 키가 이미지에 내장되면 이미지가 유출·복제될 때 아이덴티티도 복제된다(인스턴스 구분 불가). 측정값과 연계되지 않으면 "어떤 코드가 실행 중인지"를 보증하지 못해 변조 이미지도 올바른 키만 가지면 인증을 통과한다. 워크로드마다 인증 구현이 달라져 QA-07(프레임워크 수정 없는 수용)의 공통 기반이 되지 못한다.

### 공통 설계 요소

- **실패 처리**: 인증 실패·manifest 불허 조합 시 채널 수립 거부(fail-closed) + DP-11 이벤트 기록. 재시도 정책은 지수 백오프, 반복 실패는 침해 시도 의심 이벤트로 격상.
- **자격 수명**: pVM 인스턴스 수명과 동일(재부팅 시 재발급). 철회(DP-12 G4)는 TEE의 자격 무효화 + 진행 중 채널 강제 해제(DP-04)로 연동.

### 3.2 종합 판정과 권고

- **A: 채택 적합** — 단 발급·방출 경로 조건부.
- **B: 부적합** — DP-06 G2(VMID 단독 인증 금지)에 정면으로 걸림. 디버그 라벨·PoC 로그 용도 이상 금지.
- **C: 단독 부적합** — 앱 계층 프로토콜은 재사용 가능하나 identity root가 되면 안 됨.

**후보 A 발급·방출 경로 성립 조건** (Codex 검토 결과):
- pVM→TEE 호출은 DP-06의 EL2 SMC 트랩·포워딩 경로여야 하며 Host 프록시 금지.
- 측정값·manifest handle은 Host 주장 값이 아니라 TCB 경로에서 검증된 값이어야 함.
- 자격 방출 버퍼는 Host unmapped pVM↔TEE 전용 버퍼로, DP-04 일반 채널보다 먼저 사용 가능한 **최소 SMC ABI**로 별도 정의(부트스트랩 순환 회피).
- TEE가 private key를 방출하지 말고, pVM이 생성한 ephemeral 공개키에 measurement-bound certificate/token을 서명하는 방식 권장.

**권고안**: A 단일 기준안 채택. 프로토콜 구현은 표준(challenge-response/mTLS/Noise)을 A가 발급한 측정값 바인딩 자격 위에 얹는다. C 폐기, B는 디버그 용도 한정.

### 3.3 지적된 결함·누락

1. DP-09가 사실상 hard dependency — 게이트로 더 강하게 명시 필요.
2. 부트스트랩 순환: DP-04 채널 수립에 DP-10 자격이 필요한데 자격 방출이 "DP-04 정의 공유 영역"을 참조 — pVM↔TEE 최소 credential buffer를 별도 선행 정의해야 함.
3. "TEE 자격 무효화"만으로는 이미 열린 채널·캐시된 세션을 막지 못함 — 짧은 수명, epoch, nonce, revocation check, DP-04 강제 해제 연동 필요.
4. 침해된 pVM은 자기 자격을 사용할 수 있음 — 후보 A가 막는 것은 "타 도메인 아이덴티티 위조"이지 "자기 신원의 악성 행위"가 아님(한계 명시).
5. TEE TA 예산, boot storm, 호출 rate limit은 DP-06 G4/G5와 연결해 수치화 필요.

## 4. 검수 (검수: Claude)

> 수행: critic 에이전트 (별도 컨텍스트, 읽기 전용). 판정: **REVISE** — 아래 두 사항을 결정문에서 해소하는 조건으로 골격 채택 진행

**동의**: 후보 A 채택·B 부적합(DP-06 G2 정면 충돌)·C 단독 부적합 판정, Codex의 발급 경로 성립 조건 4개(EL2 SMC 경로, TCB 검증 측정값, 최소 SMC ABI 별도 정의, private key 미방출) 전부 타당.

**이견 1 (CRITICAL)**: 후보 A가 hard dependency로 삼는 **DP-09가 아직 미결정**이다(착수 단계상 DP-10보다 후순위). 지금 바인딩 구체 형태까지 확정하면 DP-09 설계 자유도를 역방향으로 침해하고, DP-09 결정이 전제와 어긋나면 본 DP 재설계가 확정적으로 발생한다. DP-06 G2의 "단계적 확정" 패턴을 차용하여, 본 DP는 **프로토콜 골격만 확정하고 측정값↔자격 바인딩의 구체 형태는 DP-09 완료 후 역반영**해야 한다.

**이견 2 (MAJOR)**: 부트스트랩 순환 해소책(최소 credential buffer 선행 정의)이 2절 후보 A의 "DP-04 정의 공유 영역으로 방출" 표현과 자기모순이며, pVM↔TEE 버퍼 소유권의 정의 책임이 DP-04(G3 위임)와 본 DP 사이에서 떠돈다 — DP-04를 정본으로 일원화하고 본 DP의 credential buffer는 그 부트스트랩 서브셋으로 참조해야 한다.

**이견 3 (MAJOR)**: 자격 무효화와 채널 강제 해제 사이의 시간 창(무효화된 자격으로 이미 열린 채널이 동작하는 구간)이 미설계 — DP-11 G4의 보증 시간 경계 패턴 차용 필요.

**이견 4 (MINOR)**: boot storm(다수 pVM 동시 부팅 시 발급 부하)·호출 rate limit의 DP-06 G4·G5 연결 수치화 필요.

## 5. 결정

**후보 A(측정값 바인딩 TEE 발급 자격)의 프로토콜 골격을 채택한다** — pVM 아이덴티티는 부팅 측정값에 바인딩되어 TEE가 발급하는 인스턴스 자격이며, 채널 수립 시 표준 프로토콜(challenge-response/mTLS/Noise 중 택일)을 이 자격 위에 얹어 상호 인증하고, 성공 후 manifest 허용 조합(DP-12)을 대조한다. TEE는 private key를 방출하지 않고 pVM이 생성한 ephemeral 공개키에 measurement-bound 인증 토큰을 서명한다. 후보 B는 디버그·PoC 라벨 용도로 한정하고(인증 근거 사용 금지), 후보 C는 identity root로 사용 금지하되 앱 계층 프로토콜 형식은 재사용한다.

**측정값↔자격 바인딩의 구체 형태(측정값 포맷, 측정 주체, 발급 TA 인터페이스)는 본 결정에서 확정하지 않는다** — DP-09 결정 완료 후 본 문서 개정으로 역반영한다(G1).

### 채택 조건 (게이트)

| # | 조건 | 처리 |

| G1 | **(CRITICAL) DP-09 단계적 확정**: 바인딩 구체 형태는 DP-09 결정에 위임하고 완료 후 역반영. | G2 | **credential buffer 일원화**: 자격 방출 버퍼는 DP-04 G3 정본(Secure World 전용 소유권 모델)의 부트스트랩 서브셋으로 정의하며, 일반 pVM↔pVM 채널(DP-04 후보 A)보다 선행 가용해야 한다. 2절 후보 A의 "DP-04 정의 공유 영역" 표현은 본 게이트로 교정한다 (부트스트랩 순서는 DP-04 5절과 공통) | DP-04 G3 |
| G3 | **무효화-해제 시간 경계**: 자격에 짧은 수명·epoch·nonce를 두고, 철회 시 TEE revocation check + DP-04 채널 강제 해제(DP-12 G4)를 양방향 연동. 무효화~해제 사이 시간 창의 보증 한계를 DP-11 G4 패턴으로 명시 | DP-04, DP-12 |
| G4 | **한계 명시**: 본 체계가 막는 것은 "타 도메인 아이덴티티 위조"이며, 침해된 pVM의 자기 자격 악성 사용은 막지 못함 — DP-07 증빙 한계 절에 기록 | DP-07 G4 |
| G5 | **TA 예산·boot storm**: 자격 발급 TA를 DP-06 G5 예산표에 등록 후 도입. boot storm 발급 부하와 호출 rate limit을 DP-06 G4와 연결해 수치화하고, DP-04의 pVM↔TEE 버퍼 호출 빈도와 합산하여 DP-06에 공동 통지 | DP-06 G4·G5 |
| G6 | **발급 경로 조건**: pVM→TEE 호출은 DP-06의 EL2 SMC 트랩·포워딩 경로만 사용(Host 프록시 금지). 측정값·manifest handle은 TCB 경로에서 검증된 값만 사용(H5 "VMID 단독 인증 금지" 준수) | DP-06 G1, DP-02 H5 |

### 후속 DP에 주는 전제

- DP-09: 본 골격(TEE 발급·측정값 바인딩·ephemeral 키 서명)이 입력 전제다. DP-09 결정 후 본 문서 G1 역반영과 DP-06 G2 역반영을 함께 수행한다.
- DP-05: 신규 워크로드의 아이덴티티 수용은 "signed manifest/측정값 등록"으로 수렴한다(체계 수정 불요, QA-07).

> 상태: **골격 채택 (검수 완료)** — 바인딩 구체 형태는 DP-09 완료 후 역반영으로 확정
