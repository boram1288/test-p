# Voices of Stakeholders (VoS)

| # | 이해관계자 | 배경 인식 | 주요 요구사항 | 주요 우려사항 |
|---|-----------|----------|------------|------------|
| 1 | Legacy CA/TA 앱 개발자 | - TrustZone 기반 TA·CA 자산 운용 중<br/>- pKVM 도입으로 SMC 경로 영향 가능성 인식 | - GP Client API 코드 수정 없이 사용<br/>- REE↔TEE SMC 경로 유지<br/>- 기존 바이너리 최소 수정 이식 | - Hypervisor의 SMC 개입 시 TA/CA 부작용<br/>- 호환성 보장 없으면 재개발 비용 발생 |
| 2 | 보안 시나리오 앱 개발자 | - 복수 독립 격리 실행 환경 필요<br/>- 기존 TrustZone은 단일 Secure World만 지원<br/>- pVM 생명주기 제어 및 양방향 통신 요구 | - pVM 생명주기 API(생성·실행·중단·삭제)<br/>- 상태 조회<br/>- RPC 통신(100μs 이내)<br/>- HW IP 독점 접근<br/>- 보안 파일 시스템<br/>- GP API 통신<br/>- Framework 무수정 확장 | - HW IP 사용 인터페이스 미구체화<br/>- pVM-TEE 통신 경로 미명세 |
| 3 | 검증팀 | - Stage-2 Page Table 격리 독립 검증 필요<br/>- 성능 수치 객관 측정 필요 | - 재현 가능한 메모리 격리 테스트 케이스<br/>- pVM 장애 격리 자동화 테스트<br/>- 성능 벤치마크 방법론 개발 초기 합의 | - 검증 환경 구성 지원 범위 불명확<br/>- 측정 방법론 미합의 시 신뢰성 저하<br/>- 인용 수치가 타 환경 기준으로 자사 환경 재현 불확실 |
| 4 | 고객사 PM | - TrustZone 한계 경험<br/>- 기존 TA/CA 자산 재활용 가능 여부가 도입 결정 중요 기준<br/>- 납기·유지보수 비용에 민감 | - 공식 호환성 보증 문서<br/>- API 하위 호환성 유지<br/>- 규제 준수 검증 증거 제공<br/>- API 명세·개발자 가이드·샘플 코드 제공 | - 기술 지원 SLA 불명확 시 운영 리스크<br/>- 납기 지연 시 자사 제품 출시 영향 |
| 5 | 경영진 | - 사이버보안 위협 급증으로 사업 리스크 인식<br/>(보안사고 시 연 매출 6.5% 이상 손실)<br/>- 기존 TrustZone 한계 극복을 위한 전략적 투자 | - ROI 정량 성과 보고<br/>- 마일스톤 주기적 보고<br/>- 다수 보안 시나리오 지원으로 복수 고객사 수요 대응 | - 기술 불확실성으로 일정 지연 시 사업 기회 상실<br/>- Android AVF 대비 차별화 포인트 전달 미흡<br/>- HV파트 3명으로 일정 유지 어려움 |
| 6 | PM | - Security 9명·HV 3명·해외 SRCX 5명 총 17명 과제 관리<br/>- 팀 간 의존성 명확화 없이 병렬 개발 불가<br/>- 해외 연구소 협업 오버헤드 존재 | - 팀 간 인터페이스 개발 초기 문서화<br/>- 목표별 달성 기준·마일스톤 정의<br/>- 기술 리스크 조기 식별 및 대응 계획<br/>- 성능 측정 방법론 검증팀과 초기 합의 | - pVM-TEE 통신 경로가 3개 파트에 걸쳐 합의 없이 시작 시 통합 실패 위험<br/>- HW IP 인터페이스 미확정 시 재작업<br/>- 해외 연구소 긴급 이슈 대응 속도 저하 |
| 7 | 제품기획팀 | - 보안 시나리오(Secure AI·Camera 등) 시장 수요 분석 담당<br/>- TrustZone 이진 구조 한계로 복수 워크로드 동시 출시 미달성<br/>- Linux 기반 시장 공백 인식 | - Framework 무수정 시나리오 확장·배포<br/>- 앱 개발자 친화적 API 추상화<br/>- 보안 시나리오별 격리 경계·HW IP 할당 제품 명세화 | - HW IP 인터페이스 미구체화 시 핵심 시나리오 제품 명세 불가<br/>- 확장성 미보장 시 출시 비용 예측 불가<br/>- 차별화 포인트가 기술 문서 수준에 머물면 고객사 설득력 약화 |
| 8 | Hypervisor 개발자 | - EL2 nVHE 모드에서 pKVM 구현 담당<br/>- Stage-2 Page Table로 pVM 간·pVM↔Host 물리 메모리 접근 완전 차단<br/>- EL2 TCB(Trusted Computing Base) 최소화 원칙 준수 필요 | - Hypercall 인터페이스 사전 명세 확정<br/>- Stage-2·HW IP 접근 제어 아키텍처 문서화<br/>- SMC 패스스루 방식 설계 초기 결정<br/>- pVM-TEE 통신 경로에서의 HV 역할 Secure OS 개발자와 합의 | - EL1↔EL2 컨텍스트 스위치 오버헤드가 성능 목표 미달 가능성<br/>- HW IP 가상화 시 IOMMU 연동 복잡도로 일정 리스크<br/>- TCB 최소화 vs 기능 제공 사이 트레이드오프 정의 필요 |
| 9 | Secure OS 개발자 | - S-EL1 Secure OS Firmware 개발·유지 및 GP API 제공 담당<br/>- pKVM 도입으로 기존 SMC 경로 변경 가능성 우려<br/>- pVM-TEE 신규 통신 경로의 구현 변경 범위 불확실 | - SMC 경로 불변 보장 및 HV 개발자로부터 패스스루 명세 수령<br/>- pVM-TEE 통신 경로에서의 Secure OS 추가 구현 범위 명확화<br/>- GP API 호환성 유지<br/>- pVM-TEE vs Host-TEE 통신 경로 식별·분리 합의 | - pKVM이 SMC 인터셉트 방식 구현 시 기존 TA 동작 부작용 발생 가능<br/>- Secure OS 수정 범위 확대 시 기존 TA/CA 호환성 훼손 위험<br/>- 검증팀 격리 경계 명세와 내부 구현 미연동 시 검증 설계 불가 |
